# IA 검토 예시 — jieun-planner 결과 샘플

> 가상의 "온라인 클래스 예약 서비스" IA를 검토한 결과 예시입니다.
> `jieun-planner`에게 "이 IA에서 빠진 화면 없는지 봐줘"라고 하면 이런 형태로 답합니다.
> (아래 서비스·화면명은 모두 가상의 예시이며 실제 프로젝트와 무관합니다.)

---

## 검토 결론

추천: 예약 취소·환불 흐름부터 보강 후 개발 전달. 왜냐면 현재 IA는 정상 예약 흐름만
있고, 운영 시 가장 문의가 많은 취소·변경 경로가 비어 있어 개발 중 재작업 위험이 큽니다.

## 우선순위별 지적 사항

### 🔴 치명 — 이대로 가면 개발이 되물어봅니다

1. **예약 취소·환불 화면 없음**
   - 빠진 것: 마이페이지에 "예약 내역"은 있는데 취소 진입점·환불 안내 화면이 없음
   - 왜 위험한지: 취소 정책(며칠 전까지·수수료)이 화면에 없으면 CS 분쟁이 그대로 운영 부담이 됨
   - 보완안: 예약 상세 하위에 "예약 취소(정책 안내 포함)" 화면 추가, 환불 상태는 예약 내역 목록에 표기

2. **결제 실패 이후 경로 없음**
   - 빠진 것: 결제 화면 다음이 "예약 완료"뿐 — 실패·이탈 시 어디로 가는지 없음
   - 왜 위험한지: 실패 사용자가 막다른 길에 갇히고, 좌석(정원) 선점 해제 기준도 정할 수 없음
   - 보완안: 결제 실패 화면(재시도/다른 수단) 추가, 선점 해제 시간은 확인요망으로 분리

### 🟡 중요 — 지금 정하면 싸게 끝납니다

3. **클래스 목록의 빈 결과(0건) 상태 미정의**
   - 빠진 것: 필터 결과 0건일 때 화면 정의 없음
   - 왜 위험한지: 빈 화면만 나오면 사용자가 오류로 오해하고 이탈함 — 개발도 임의로 처리하게 됨
   - 보완안: "조건에 맞는 클래스 없음 + 필터 초기화" 상태 1줄 추가

4. **강사(파트너) 권한 분기 없음**
   - 빠진 것: Admin에 운영자 메뉴만 있고, 강사 본인이 자기 클래스만 보는 권한 구분이 없음
   - 왜 위험한지: 강사에게 운영자 권한을 그대로 주면 다른 강사의 클래스·매출까지 보임
   - 보완안: Admin 권한을 운영자/강사 2단으로 분리 `[추정]` — 강사 직접 접속 여부는 확인요망

### 🟢 권장 — 여유 있을 때

5. **예약 완료 후 후속 처리 표기** — 완료 화면에 알림톡 발송·캘린더 추가 여부를 IA 비고에 명시하면 개발 견적이 정확해짐

## 확인요망 — 고객이 확정해야 할 것 (본문과 분리)

| # | 확인할 것 | 왜 필요한지 |
|---|---|---|
| 1 | 취소 가능 기한·수수료 정책 | 취소 화면 문구·버튼 노출 조건이 여기에 걸림 |
| 2 | 결제 실패 시 정원 선점 해제 시간 | 실패 경로 설계의 전제 |
| 3 | 강사가 Admin에 직접 접속하는지 | 권한 2단 분리 여부 결정 |
