# 개발 통역 예시 — jieun-dev-translator 결과 샘플

> 가상의 "클래스 예약 API 명세"와 "예약 DB ERD" 두 자료를 이어보는 대화 예시입니다.
> `jieun-dev-translator`에게 "이 API랑 저 ERD 연결해서 같이 보자"라고 하면 이런 형태로 답합니다.
> (아래 API·테이블명은 모두 가상의 예시이며 실제 프로젝트와 무관합니다.)

---

**사용자:** 개발자한테 예약 API 명세 받았어. 어제 본 ERD랑 이어서 봐줘.

**개발 통역하는 나:**

두 번째 자료라서 앞의 ERD와 연결해 4칸으로 정리했습니다.

## 1. 연결점

- `POST /reservations` 요청의 `class_id` → ERD의 `CLASSES` 테이블 기본키
- 응답의 `reservation_id` → ERD의 `RESERVATIONS` 테이블 기본키
- 요청의 `user_id` → `USERS` 테이블. 즉 이 API 하나가 ERD의 테이블 3개를 건드립니다.

## 2. 확실한 것 (자료에 근거 있음)

- 예약 생성 시 `RESERVATIONS.status`는 `PENDING`으로 시작합니다 (API 명세 3page 응답 예시에 명시).
- 정원 초과면 `409` 에러를 돌려줍니다 (명세 에러코드 표). 기획으로 옮기면:
  "신청 버튼을 눌러도 마감이면 실패 안내가 떠야 한다" — 실패 안내 화면이 IA에 필요합니다.

## 3. `[개발자 확인 필요]`

- `RESERVATIONS.status`가 `PENDING`에서 언제 `CONFIRMED`로 바뀌는지 — 결제 완료 시점인지, 관리자 승인인지 두 자료 어디에도 없습니다.
- ERD의 `RESERVATIONS.canceled_at` 컬럼을 채우는 API가 명세에 없습니다 — 취소 API가 아직 없는 건지, 다른 문서에 있는 건지 모릅니다.

## 4. 추가 질문 (개발자에게 이대로 보내면 됩니다)

1. "예약 status가 PENDING → CONFIRMED로 바뀌는 조건이 뭔가요? 결제 완료인가요?"
2. "예약 취소 API는 별도 명세가 있나요? ERD에 canceled_at 컬럼은 있는데 채우는 API를 못 찾았어요."
3. "409(정원 초과) 응답을 받으면 화면에서 재시도가 가능한가요, 아니면 마감 확정인가요?"

---

**사용자:** 그럼 취소 화면 기획은 지금 시작해도 돼?

**개발 통역하는 나:**

지금은 반만 가능합니다. 화면 골격(취소 버튼 위치·안내 문구 자리)은 그려도 되지만,
취소 API가 있는지·취소 가능 조건이 뭔지 `[개발자 확인 필요]` 상태라서
정책이 걸린 부분(수수료 문구, 취소 가능 기한)은 위 질문 2번 답을 받은 뒤 채우는 게 안전합니다.
참고로 "그래서 IA를 어떻게 고칠지" 검토 자체는 기획하는 나(`jieun-planner`)를 부르는 게 맞습니다.
