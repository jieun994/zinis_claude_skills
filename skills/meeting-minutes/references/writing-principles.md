# 회의록 작성 원칙

회의록의 목적은 "무슨 얘기가 나왔는지 나열"이 아니라, **회의가 끝난 뒤 누가 무엇을 다시 확인·정정·전달해야 하는지 보이게 하는 것**이다. 아래 3원칙과 할루시네이션 방지 철칙을 모든 회의록에 적용한다.

---

## 원칙 1. 이해 차이와 자료 무효는 풀어서 적는다

양측이 다르게 이해하고 있었음이 드러난 사안, 또는 한쪽이 전달한 자료와 실제 결정이 달라진 사안은 한 줄로 뭉뚱그리지 않는다. **어느 쪽이 무엇을 어떻게 알고 있었는지, 어떤 자료의 어떤 부분이 더 이상 유효하지 않은지**가 드러나야 누가 무엇을 정정해야 하는지 보인다.

작성 형식 (회의 내용 표의 "내용" 칸):

```
- [이해 차이 확인] {안건명}
  ・ {A측}: {A측이 전제하고 있던 내용}
  ・ {B측}: {B측이 전제하고 있던 내용}
  ・ 회의 결과: {확인된 사실 / 어느 방향으로 정리되었는지 / 미정이면 "결론 미도출, 후속 협의"}
```

자료 무효화는 비고 칸 또는 내용 칸에 파일명·위치까지 적는다:

```
- [자료 일부 무효] {파일명} {페이지/시트/항목}의 {내용}은 {사유}로 이번 범위에서 제외
```

**나쁜 예:** `검색 기능 시나리오 제공 예정`
**좋은 예:**
```
- [이해 차이 확인] 검색 처리 구조
  ・ 고객사: 자연어 입력을 AI가 통합 판단하여 결과 반환하는 구조로 전제
  ・ 수행사: 관리자 화면에서 유형별로 분리 매핑하는 구조로 전제
  ・ 회의 결과: 고객사 전제 구조로 확인됨. 수행사 설계 문서의 매핑 방식 부분 수정 필요
- [자료 일부 무효] MVP화면_v1.pptx 3p 음성인식 버튼 — 예전 디자인으로 이번 범위에서 제외 (고객사 구두 확인)
```

판별 신호: "저희는 ~인 줄 알았는데요", "어? 그게 아니라…", "그 자료는 예전 거예요", "그 부분은 무시하셔도 돼요", 같은 안건에서 양측 설명이 어긋나는 구간.

**주의:** 이해 차이가 없는 평범한 회의에서 억지로 차이를 만들어내지 않는다. 없으면 이 형식을 쓰지 않으면 된다.

## 원칙 2. "검토하겠다"에는 사유를 붙인다

고객이 "내부 논의 후 전달", "검토해보겠다"고 답한 항목은 **무엇 때문에 검토하는지 그 배경·쟁점**을 함께 적는다. 사유가 없으면 검토하는 쪽도 어떤 관점에서 결정해야 하는지 모르고, 회신도 부정확해진다.

작성 형식:

```
- {안건}: 내부 검토 후 전달 예정
  ・ 검토 사유: {회의에서 나온 배경 — 왜 즉답하지 못했는지, 어떤 쟁점 때문인지}
```

**나쁜 예:** `가드레일 적용 방식 — 내부 논의 후 전달 예정`
**좋은 예:**
```
- 가드레일 적용 방식: 내부 논의 후 전달 예정
  ・ 검토 사유: 통합 처리 구조로 갈 경우 기능별 가드레일의 의미가 축소될 가능성이
    회의 중 제기되어, 적용 단위(전체 공통 vs 기능별)를 내부에서 재판단하기로 함
```

검토 사유를 원문(녹취록·메모)에서 찾을 수 없으면 지어내지 말고 `검토 사유: 회의 중 미언급 — 확인 필요`로 적는다.

## 원칙 3. TO-DO는 항목별로 쪼갠다

후속 정리 사항을 "내부 논의 후 전달" 한 묶음으로 넣지 않는다. **전달물 하나 = TO-DO 한 행.** 각 행에 담당자·완료희망일·요청 형식(무엇을 어떤 형태로)까지 적어야 받는 쪽도 정확히 알고 요청한 쪽도 추적할 수 있다.

분해 기준:
- "정책서 정리해서 드릴게요" → 정책서가 여러 개면 각각 한 행 (예: 1.보안가이드 / 2.AI가이드 / 3.영역구분 플로우정의서)
- 회의 중 우리가 요청한 사항(예: "플로우에 영역 색깔 구분해주세요")도 빠짐없이 TO-DO에 등재
- 내용 칸에는 요청 형식을 명시: `- 형식: 문서(PDF), 항목별 담당 영역 색상 구분 포함`

**나쁜 예:** `정책 관련 사항 — 내부 논의 후 전달 (고객사)`
**좋은 예 (3행으로 분해):**

| 항목 | 영역 | 내용 | 완료희망일 | 담당자 |
|------|------|------|-----------|--------|
| 정책 | 고객사 | 보안가이드 문서 전달 - 형식: 사내 보안규정 PDF | 2026-07-10 | 홍길동 |
| 정책 | 고객사 | AI 운영 가이드 전달 - 형식: 초안 수준 가능 | 2026-07-10 | 홍길동 |
| 플로우 | 고객사 | 처리 플로우 영역 구분본 전달 - 형식: AI/API 영역 색상 구분 표시 | 미정(확인 필요) | 김철수 |

담당자·기한이 회의에서 언급되지 않았으면 `미정(확인 필요)`로 적는다. 임의로 채우지 않는다.

## 할루시네이션 방지 철칙 (Iron Law)

**원문(녹취록·메모·자료)에 없는 내용은 회의록에 쓰지 않는다.** 회의록은 책임 소재가 걸린 문서다. 한 칸이라도 지어낸 것이 발견되면 문서 전체의 신뢰가 무너진다.

| 흔한 합리화 | 실제로 해야 할 일 |
|------------|------------------|
| "맥락상 이 사람이 담당일 것 같다" | 담당자 칸에 `미정(확인 필요)` |
| "보통 일주일 뒤가 기한이니까" | 완료희망일 칸에 `미정(확인 필요)` |
| "빈 칸이 많으면 없어 보이니까 채우자" | 빈 칸은 정보 부족의 정직한 표시다. 그대로 둔다 |
| "화자가 누군지 대충 알겠다" | 화자 구분 없는 녹취록이면 `화자 미상` 표기 |
| "이해 차이가 있었던 것 같은 느낌" | 근거 발언을 먼저 인용할 수 없으면 쓰지 않는다 |

절차 규칙:
1. 원칙 1·2를 적용할 때는 **근거가 되는 발언·자료 위치를 먼저 확인한 뒤** 해석을 쓴다.
2. 확실하지 않은 항목은 모두 `(확인 필요)`를 붙이고, 마지막에 "확인 필요 목록"으로 모아서 사용자에게 보여준다.
3. 녹취 유실·불명확 구간은 앞뒤 맥락으로 재구성하지 않고 해당 안건에 `녹취 불명확 — 원문 확인 필요`로 남긴다.
4. 번복된 안건은 최종 결론만 쓰지 말고 `(최초 A안 → 논의 후 B안으로 변경)` 형태로 변경 경과를 남긴다.

## 표기 통일

| 상황 | 표기 |
|------|------|
| 기한·담당자 미언급 | `미정(확인 필요)` |
| 발언자 불명 | `화자 미상` |
| 검토 사유 미언급 | `검토 사유: 회의 중 미언급 — 확인 필요` |
| 이해 차이 없음(검토는 했음) | 별도 표기 없음 — 억지로 만들지 않는다 |
| 자료 일부 제외 | `[자료 일부 무효] 파일명 + 위치 + 사유` |
