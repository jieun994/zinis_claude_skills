# 회의록 작성 (meeting-minutes)

> 녹취록·메모를 받아 표준 엑셀 양식 회의록을 만든다. 회의록을 처음 쓰는 사용자도 따라올 수 있도록 단계별로 안내하며 진행한다.

사용자는 비개발자일 수 있다. 전문 용어 없이 안내하고, 각 단계에서 다음에 무엇을 할지 미리 알려준다.

## 워크플로우

### Step 1: 준비물 확인
**타입**: prompt

사용자가 파일을 첨부하지 않았으면 한 번에 다 요구하지 말고 순서대로 물어본다:

1. "회의 녹음이나 녹취록(클로바노트 txt/docx 등)이 있나요? 없으면 기억나는 내용을 그냥 대화창에 적어주셔도 돼요."
2. 녹취록을 받았으면 추가로: "품질을 높여줄 선택 재료예요 — 있으면 같이 주세요: ① 본인 메모 ② 회의에서 같이 본 자료(PPT/엑셀 등) ③ 직전 회의록"

입력별 역할을 이해하고 활용한다:

| 입력 | 필수 여부 | 역할 |
|------|----------|------|
| 녹취록 | 필수(없으면 메모로 대체) | 회의 내용·TO-DO의 근거 |
| 본인 메모 | 권장 | 녹취에 안 잡힌 맥락·검토 사유 보완 |
| 회의에서 다룬 자료 | 권장 | "어떤 자료의 어떤 부분이 무효인지" 기록의 근거 |
| 직전 회의록 | 선택 | 지난 결정이 번복된 부분 감지 |

녹취록 없이 메모만 있으면 진행은 하되 미리 알린다: "메모만으로 만들 수 있어요. 다만 발언 단위 기록이 없어서 '누가 어떻게 이해하고 있었는지' 같은 분석은 빠질 수 있어요."

### Step 2: 기본 정보 확인
**타입**: prompt

원천 자료에서 프로젝트명·회의명·일자·작성자·참석자·근거문서(회의에서 참조한 문서들)를 추출해 채운 결과를 보여주고 확인받는다. 사용자가 직접 쓰게 하지 말고, 추출한 값을 "이렇게 채울게요, 틀린 부분만 알려주세요" 형식으로 제시한다.

참석자는 양측(예: 고객사/수행사)으로 나눠 적되, 사내 회의처럼 구도가 다르면 팀·부서 단위로 유연하게 나눈다. 원문에서 찾을 수 없는 항목은 사용자에게 물어본다 — 추측으로 채우지 않는다.

### Step 3: 분석 및 초안 작성
**타입**: prompt + rag

`references/writing-principles.md`를 읽고 3원칙 + 할루시네이션 방지 철칙을 적용해 분석한다:

1. **회의 내용**: 안건을 영역(구분)별로 묶고, 항목 번호 목록 + 항목별 상세를 작성
   - 양측이 다르게 이해했던 사안 → `[이해 차이 확인]` 형식으로 양쪽 입장 병기
   - 회의에서 무효화된 자료 → `[자료 일부 무효]` 형식으로 파일명·위치·사유 기록
   - "검토하겠다/내부 논의" 답변 → 검토 사유를 원문에서 찾아 병기
2. **TO-DO**: 후속 전달물·요청 사항을 전달물 하나 = 한 행으로 분해. 담당자·완료희망일·요청 형식 포함. 원문에 없으면 `미정(확인 필요)`
3. **이슈**: 프로젝트 진행을 막는 문제가 언급됐으면 기록. 없으면 비워둔다

원문에 없는 내용은 절대 채우지 않는다. 이해 차이·검토 사유는 근거 발언을 먼저 확인한 뒤에만 쓴다.

### Step 4: 초안 검토
**타입**: review

엑셀을 만들기 전에 마크다운 초안을 대화창에 보여주고 검토받는다. 검토 요청 시 사용자가 봐야 할 지점을 쉬운 말로 짚어준다:

- "여기 N군데는 회의에서 담당자·날짜 언급이 없어서 '미정(확인 필요)'로 뒀어요. 아시는 게 있으면 알려주세요."
- "이 안건은 양쪽이 다르게 알고 있던 걸로 보여 양쪽 입장을 나눠 적었어요. 사실과 맞는지 봐주세요."

수정 요청을 반영한 뒤 다음 단계로 간다. 초안 확인 없이 바로 엑셀을 만들지 않는다 — 이해 차이 판단과 TO-DO 분해는 잘못 쓰면 되돌리기 어려운 해석 작업이기 때문이다.

### Step 5: 엑셀 생성
**타입**: script + generate

확정된 초안을 JSON으로 변환해 스크립트를 실행한다:

```bash
python "${CLAUDE_PLUGIN_ROOT}/scripts/build_minutes.py" data.json -o "[프로젝트명] 회의록_YYYYMMDD.xlsx"
```

(Windows PowerShell에서는 `$env:CLAUDE_PLUGIN_ROOT`, `python3`이 없으면 `python`.)

JSON 스키마는 build_minutes.py 상단 docstring 참조. 날짜는 "YYYY-MM-DD" 문자열, 미확정 값은 `미정(확인 필요)` 문자열 그대로 넣는다. data.json은 임시 폴더에 만들고 산출물 엑셀은 사용자 작업 폴더에 저장한다.

### Step 6: 마무리 안내
**타입**: review

생성된 파일을 열어 값이 제대로 들어갔는지 확인한 후, 사용자에게 전달한다:

1. 파일 위치 안내
2. **확인 필요 목록** — `(확인 필요)`로 표기된 항목을 모아 "회의 참석자에게 이것들을 확인받으면 회의록이 완성돼요"라고 안내
3. 다음 행동 제안 — "이 회의록을 공유 메일로 정리해드릴까요?" 등

## 실패 처리

- 스크립트 실행이 실패하면 Step 4의 마크다운 초안을 .md 파일로 저장해서라도 전달한다 — 사용자의 분석 결과가 유실되지 않게 한다.
- 텍스트를 읽을 수 없는 입력(이미지 스캔 등)은 시작 전에 알리고 진행 여부를 확인한다.
- 녹취록이 매우 길면 나눠 읽되, 같은 안건이 여러 번 등장할 수 있으므로 최종 결론은 전체를 다 읽은 뒤에 판단한다.

## References

- **`references/writing-principles.md`** — 작성 3원칙(이해차이 풀어쓰기 / 검토사유 병기 / TO-DO 분해) + 할루시네이션 방지 철칙 + 표기 통일. Step 3 시작 전에 반드시 읽는다.

## Scripts

- **`scripts/build_minutes.py`** — 회의록 JSON → 템플릿 기반 엑셀 생성 (서식·병합 셀 보존, 행 자동 추가)

## Assets

- **`assets/회의록_템플릿.xlsx`** — 표준 회의록 양식 (회의 개요 / 회의 내용 / TO-DO TASK / ISSUE REPORT). 직접 수정하지 않는다.

## Settings

| 설정 | 기본값 | 변경 방법 |
|------|--------|-----------|
| 참석자 구분 라벨 | 고객사/수행사 | 회의 성격에 맞게 자동 판단, 사용자가 말하면 그대로 사용 |
| 초안 검토 단계 | 항상 수행 | 사용자가 "초안 확인 없이 바로 만들어줘"라고 하면 생략 가능 (확인 필요 목록은 반드시 전달) |
| 출력 파일명 | [프로젝트명] 회의록_YYYYMMDD.xlsx | 사용자 지정 시 그대로 |
