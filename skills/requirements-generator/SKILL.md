# 요구사항 정의서 생성기 (requirements-generator)

> 회의록·RFP·기존 엑셀에서 요구사항을 뽑아, 실제 SI 산출물 양식 그대로 엑셀 요구사항 정의서를 만든다.

이 스킬의 결과물은 **초안**이다. 우선순위·분류·관련화면은 사람이 검증해야 한다. 지어내지 않고, 근거를 밝히고, 애매하면 되묻는 것이 이 스킬의 핵심 가치다.

## Iron Law: 근거 없는 요구사항을 만들지 않는다

모든 요구사항에는 **출처**가 있어야 한다. 입력 자료에서 나왔으면 그 근거를, AI가 관례상 추정했으면 `AI추정(확인필요)`를 출처에 적는다. 출처를 못 대는 요구사항은 지어낸 것이므로 표에 넣지 않는다.

입력이 너무 빈약하면(한 문장짜리 설명 등) **바로 만들지 말고 되묻는다.** 방대한 요구사항을 상상해서 채우는 것은 실패다.

### 합리화 차단 (Excuse → Reality)

| 이럴 때 드는 생각 | 실제로는 |
|---|---|
| "입력이 부족하지만 일반적인 기능은 다 넣어주자" | 그건 이 프로젝트의 요구사항이 아니라 남의 프로젝트 얘기다. 되물어라. |
| "우선순위는 대충 상/중으로 채워두면 되겠지" | 우선순위는 고객 예산·정치·일정에 달렸다. 근거 없으면 `미정`으로 두고 검토 요청한다. |
| "관련화면을 그럴듯하게 추측해 채우자" | 틀린 화면 링크는 신뢰를 깬다. IA가 없으면 비워둔다. |
| "출처 컬럼은 귀찮으니 비워도 되겠지" | 출처가 비면 원문 요구사항인지 AI 추정인지 구분이 사라진다. 반드시 채운다. |

### Red Flags — 이 신호가 보이면 멈추고 되묻어라

- 입력이 프로젝트명 한 줄 또는 "○○ 만들어줘" 수준일 때
- 첨부 엑셀의 컬럼 구조를 확신할 수 없을 때 (임의 파싱 금지)
- 생성하려는 요구사항의 30% 이상이 입력에 근거가 없을 때

## 워크플로우

### Step 1: 입력 감지·수집
**타입: prompt**

사용자가 무엇을 줬는지 판별한다:
- **파일 첨부(회의록·RFP·기존 IA/기능정의 엑셀)** → 그 내용을 읽어 요구사항의 1차 근거로 삼는다. 엑셀이면 시트·컬럼 구조를 먼저 요약해 "이 구조로 읽어도 될지" 확인받는다. 임의 파싱 금지.
- **자연어 설명** → 프로젝트 개요로 받는다.
- **둘 다** → 파일에서 최대한 뽑고, 부족분만 설명으로 보완한다.

작업 폴더에 `_inputs/`가 있으면 관련 자료가 있는지 먼저 확인한다.

### Step 2: 빈약 입력 검증 (되묻기)
**타입: review**

입력에서 요구사항을 도출할 근거가 부족하면 즉시 생성하지 않는다. AskUserQuestion으로 3~5개만 짧게 되묻는다:
- 대상 사용자 / 핵심 기능 몇 개 정도인지 / 비기능(성능·보안) 요건 유무 / 대분류 체계를 어떻게 나눌지

사용자가 "그냥 알아서"라고 하면, 최소 골격만 만들고 각 항목 출처를 `AI추정(확인필요)`로 명시한다.

### Step 3: 요구사항 도출 (2단계 — 원문 먼저, 추정 나중)
**타입: prompt**

원문 요구사항과 AI 추정을 절대 섞지 않기 위해 순서를 지킨다:
1. **원문 매핑** — 입력 자료에 실제로 있는 항목을 먼저 요구사항으로 옮긴다. 각 행 `출처`에 근거를 적는다(예: `RFP`, `킥오프 회의록`, `인텐트통합_V3.1`).
2. **추정 보완** — 그다음, 관례상 빠졌을 법한 항목만 별도로 추가하고 `출처`를 `AI추정(확인필요)`로 적는다.

`유형`(기능/비기능/운영/인터페이스/보안), `우선순위`(최상/상/중/하/미정), `대분류`·`중분류`는 §template-spec의 값 체계를 따른다. 모르는 셀은 비운다.

### Step 4: 사용자 검토·확정
**타입: review**

엑셀을 만들기 전에 도출한 요구사항을 **마크다운 표로 먼저 보여준다.** 사용자가 분류·우선순위·관련화면·누락을 확인·수정하게 한다. 이 단계는 건너뛰지 않는다 — 초안의 오류를 사람이 잡는 유일한 지점이다.

### Step 5: 엑셀 생성
**타입: script**

확정된 요구사항을 JSON으로 만들어 스크립트에 넘긴다. 요구사항ID는 스크립트가 대분류별로 자동 채번하므로 비워둔다.

```bash
python "${CLAUDE_PLUGIN_ROOT}/scripts/build_requirements_xlsx.py" <input.json> <output.xlsx> "프로젝트명"
```

Windows PowerShell에서는 `${CLAUDE_PLUGIN_ROOT}` → `$env:CLAUDE_PLUGIN_ROOT`, `python`이 없으면 `py -3`으로 호출한다. JSON 스키마·채번 규칙은 `references/template-spec.md` 참조.

### Step 6: 산출물 리뷰
**타입: generate**

스크립트가 낸 결과 JSON을 사용자에게 한국어로 요약한다: 파일 경로, 총 건수, **AI추정 건수·비율**, ID 중복 경고 등. AI추정 비율이 높으면(30%+) 입력 보강을 권한다.

## 엣지 케이스 대응

- **대량 입력(요구사항 수백 개)** — 화면·섹션 단위로 나눠 도출하되, 요구사항ID는 스크립트의 대분류별 카운터로 전체에서 이어 채번되어 중복·결번이 없다.
- **지저분한 첨부 엑셀(병합셀·컬럼 뒤섞임)** — 파싱 오류를 조용히 삼키지 않는다. "구조 확인 필요"를 알리고 사용자에게 컬럼 매핑을 확인받는다.
- **중간 실패** — 이미 확정된 요구사항 JSON은 보존하고, 어디까지 됐는지 한국어로 알린 뒤 그 지점부터 재개한다. 처음부터 다시 만들게 하지 않는다.
