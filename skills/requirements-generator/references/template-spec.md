# 요구사항 정의서 양식 명세 (template-spec)

이 스킬이 만들어내는 엑셀은 실제 SI 프로젝트 산출물인 **`04_요구사항 정의서`** 시트를 그대로 재현한다.

## 1. 시트 구조

- 시트명: `04_요구사항 정의서`
- 1행: 제목 `요구사항 정의서` (병합, 굵게 size 13)
- 2행: 공백
- 3~4행: 헤더 (검정 배경 + 흰색 굵은 글씨). 세로 병합(3:4), 단 `요구사항명`·`요청자`는 하위 컬럼이 있어 가로 병합.
- 5행~: 데이터. 틀고정 `A5`.

## 3. 요구사항ID 채번 규칙

**대분류별로 접두어가 다르고, 대분류마다 독립적으로 001부터 채번한다.**

| 대분류 | 접두어 | 예 |
|--------|--------|-----|
| 챗봇 | `REQ-CHT` | REQ-CHT-001 |
| 공통 | `REQ-COM` | REQ-COM-001 |
| 어드민 | `REQ-BOS` | REQ-BOS-001 |
| 시스템 | `REQ-SYS` | REQ-SYS-001 |
| 관리 | `REQ-MNG` | REQ-MNG-001 |
| (그 외) | `REQ-ETC` | REQ-ETC-001 |

스크립트가 자동 처리하므로 JSON에서는 `요구사항ID`를 비워두면 된다. 기존 ID를 수정·이어치기 하는 경우엔 값을 넣으면 스크립트가 번호를 이어서 매긴다.

## 4. 출처(근거) 컬럼 — 할루시네이션 방지의 핵심

`출처` 컬럼은 **각 요구사항이 어디서 나왔는지**를 반드시 밝히는 자리다.

- 입력 자료에 실제로 있던 요구사항 → 출처에 근거를 적는다.
  - 예: `RFP`, `인텐트통합_V3.1`, `킥오프 회의록 2026-07-06`, `BOS_의견설정_v3.1`
- 입력에 없는데 AI가 관례상 추정해 채운 요구사항 → 출처를 반드시 `AI추정(확인필요)` 로 적는다.
  - 스크립트가 이 행의 출처 셀을 연노랑으로 강조하고, AI추정 비율이 30% 이상이면 경고를 낸다.

**절대 규칙: 출처가 비어 있는 요구사항을 만들지 않는다.** 근거를 못 대면 그건 지어낸 것이다.

## 5. 입력 JSON 스키마

```json
{
  "project": "〇〇 프로젝트",
  "requirements": [
    {
      "대분류": "챗봇",
      "중분류": "대화/인텐트",
      "요구사항명_1depth": "배송문의",
      "요구사항명_2depth": "배송기사연결",
      "상세설명": "텍스트 안내형. 기사연결 절차를 RAG로 안내(연결 실행 아님)",
      "유형": "기능",
      "출처": "인텐트통합_V3.1",
      "rfp_id": "SFR-002",
      "q_n": "",
      "요청자_소속": "", "요청자_파트": "", "요청자_이름": "", "요청일시": "",
      "우선순위": "상",
      "난이도": "중",
      "수용여부": "",
      "담당자": "",
      "처리방안": "",
      "관련자료": ""
    }
  ]
}
```

- `대분류`, `요구사항명_1depth`, `상세설명`, `유형`, `출처` 는 사실상 필수.
- 나머지는 모르면 공백으로 둔다. **모르는 값을 지어내 채우지 않는다.**
- `requirements`만 담은 순수 배열 `[ {...}, {...} ]` 로 줘도 동작한다.

## 6. 스크립트 실행

```bash
python "${CLAUDE_PLUGIN_ROOT}/scripts/build_requirements_xlsx.py" <input.json> <output.xlsx> "프로젝트명"
```

Windows PowerShell: `${CLAUDE_PLUGIN_ROOT}` → `$env:CLAUDE_PLUGIN_ROOT`, `python`이 없으면 `py -3`.

출력(stdout)은 결과 요약 JSON: `count`(생성 건수), `ai_estimated`(AI추정 건수), `ai_ratio_percent`, `warnings`(ID 중복·AI추정 과다 경고).
