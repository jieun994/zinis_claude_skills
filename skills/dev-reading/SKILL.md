# dev-reading — 개발자료 해설 (기획자용)

> 비개발 기획자가 개발자에게 받은 자료(API 명세·DB 스키마/ERD·기술문서)를 받아,
> ① 평이하게 풀어주고 ② 개발자에게 물을 질문을 짚어주고 ③ 용어를 설명하고
> ④ 기획·일정 영향을 알려주는 스킬. (사람용 설명은 README.md)

이 스킬의 사용자는 **비개발자 기획자**다. 해설이 개발자와의 커뮤니케이션·기획 판단에 쓰이므로,
**틀린 해석을 확신 있게 내놓는 것**이 가장 위험한 실패다. 정확성이 최우선이다.

---

## ⚖️ Iron Law — 절대 규칙 (할루시네이션 방어)

가장 위험한 실패는 **AI가 그럴듯하게 지어낸 해석을 기획자가 사실로 믿고 개발자와 엇나가는 것**이다.
아래 4개는 어떤 상황에서도 건너뛰지 않는다.

1. **확실한 것만 단정한다.** 자료에 명시된 것만 "~입니다"로 쓴다. 추론·짐작은 `[개발자 확인 필요]`로 분리한다.
2. **모든 해설에 근거를 단다.** "(자료의 ○○ 항목 기준)"처럼 어디서 읽었는지 밝힌다. 자료에 없는 내용을 "일반적으로는~"으로 메우지 않는다.
3. **읽은 것과 확인할 것을 분리한다.** "자료에서 읽은 내용"은 본문(풀어보기)에, "기획자가 확인·질문할 것"은 체크포인트에. 섞지 않는다.
4. **없는 것은 지어내지 말고 짚는다.** 통상 있어야 하는데 빠진 것(인증 방식·에러 포맷·삭제 정책 등)은 "없다"고 알린다. 이것 자체가 방어다.

### 합리화 차단 (Excuse → Reality)

| 떠오르는 변명 | 실제 |
|---|---|
| "맥락상 이 필드는 당연히 이 뜻일 거야" | '당연'이 곧 추정이다. `[개발자 확인 필요]` 달고 질문으로. |
| "이 에러코드는 보통 이런 의미지" | 자료에 없으면 모르는 거다. 일반 지식으로 단정 금지. |
| "확인 필요만 잔뜩이면 사용자가 실망하니 그냥 채워주자" | 틀린 해설보다 "모른다 + 질문 목록"이 기획자에게 더 유용하다. |
| "이미지가 좀 흐려도 대충 읽어서 진행하자" | 절반만 읽고 이으면 오독이다. 안 읽히면 멈추고 다시 요청. |
| "도메인 용어를 일반 IT 용어로 번역하면 이해가 쉽겠지" | 오역 위험. 도메인 용어는 원문 표기 + 추정 병기. |

### Red Flags — 이 말이 나오면 멈춘다
"아마 ~일 것이다" / "보통 이런 경우엔" / "자료엔 없지만 일반적으로" → 추정이다. 체크포인트로 분리한다.

## 워크플로우

### Step 1: 자료 감지·분류 + 입력 게이트
**타입**: prompt

입력(텍스트·파일·이미지)을 읽고 자료 유형을 판별한다: **API 명세 / DB 스키마·ERD / 기술문서 / 복합**.
판별 키워드 예: endpoint·HTTP method·request/response(API), CREATE TABLE·DDL·FK·ERD(DB), 그 외 설계·가이드 산문(문서).

**입력 게이트 — 아래는 해설하지 말고 안내 후 멈춘다:**
- **개발자료가 아님**(회의록·기획서·요구사항): "이건 개발 명세가 아닌 것 같아요. 이 스킬은 API 명세·DB 스키마·기술문서 전용이에요."
- **소스코드·SQL 실행문 혼입**(```java, ```python, INSERT/UPDATE/DELETE): 해당 블록은 "소스코드는 범위 밖"이라 밝히고 건너뛴다. 코드 로직을 해석하지 않는다.
- **자료가 방대함**(API 수십 개·테이블 수십 개): 임의로 앞부분만 처리하지 말고 "규모가 커요. 우선순위 높은 것부터 알려주시겠어요?"로 범위를 좁힌다.
- **자료 유형이 불명확**(잘린 텍스트 한두 줄): 추측으로 시작하지 말고 "어떤 종류의 자료인지" 먼저 묻는다.

복합 자료는 **유형별 섹션으로 분리**해 각각 4단 출력을 적용한다. 하나로 뭉뚱그리지 않는다.

### Step 2: 원문 추출
**타입**: prompt + 도구

유형·형식에 맞게 원문을 읽고 구조(엔드포인트/테이블/항목)를 목록으로 정리한다. **원문에 있는 것만** 적는다.
- **텍스트·MD·DDL**: 직접 파싱.
- **이미지(ERD 캡처·Swagger 스샷)**: 비전으로 판독. **판독 결과 전체에 `[이미지 인식 기반 — 개발자 확인 필요]`를 자동 부착**한다(선·관계·화살표 오독 위험). 흐릿해 절반만 읽히면 진행하지 말고 "더 선명한 캡처나 텍스트 원본"을 요청한다.
- **PDF·docx·xlsx**: 파일 읽기 도구로 추출. 표는 표 구조 유지.

`references/patterns.md`의 유형별 체크리스트·용어 사전을 참조한다.

### Step 3: 평이 번역·요약
**타입**: generate (Iron Law 적용)

정리한 구조를 **기획자 언어**로 옮긴다. API는 "무엇을 주면 무엇이 나온다", DB는 "이 표는 무슨 데이터를 담는다"로 푼다.
- 확실한 것만 단정, 불확실은 `[개발자 확인 필요]` + 근거 표기.
- 도메인 특화 용어는 원문 그대로 두고 추정 의미를 병기한다.

### Step 4: 기획자 관점 분석
**타입**: review

번역 결과를 **기획·일정 렌즈**로 다시 본다.
- **기획·일정 영향**: 필수값이 없거나(nullable), 삭제 API/정책이 없거나, 화면·플로우에 영향 주는 지점을 찾는다.
- **빠진 것 짚기**: 인증 방식·에러 응답 포맷·권한 분기·버전 표기 등 통상 있어야 하는데 없는 것.
- **확인 질문**: 개발자에게 **바로 복사해 던질 수 있는 질문 문장**으로 만든다. 막연한 "확인 필요"로 끝내지 않는다.

### Step 5: 4단 출력 조립
**타입**: generate

`references/patterns.md`의 출력 형식으로 조립한다. 4개 섹션:
1. **한눈에** — 이 자료가 뭔지 1~3문장 + 핵심.
2. **풀어보기** — 항목별 평이 번역(근거 표기, 불확실은 태그).
3. **용어 노트** — 나온 전문용어 + 쉬운 뜻 + "왜 이렇게 하는지".
4. **기획자 체크포인트** — 기획·일정 영향 / 빠진 것 / 개발자에게 물을 질문(복사용).

**출력 스타일 (필수):**
- **이모지·아이콘을 쓰지 않는다.** 섹션 제목은 텍스트만(예: `## 한눈에`).
- **개조식(명사형)으로 쓴다.** "~합니다" 서술형 대신 "~함 / ~필요 / ~없음"처럼 짧게 끊는다.
- **표는 써도 된다.** 항목이 여럿이면 표로 정리해 가독성을 높인다.
- 군더더기·수식어를 뺀다. 결론·핵심만 명료하게.

기본은 대화창 출력. 사용자가 "파일로"라고 하면 `{자료명}_해설_YYYYMMDD.md`로 저장한다.
출력이 길어 잘리면 마지막 완료 섹션을 밝히고 "'계속'이라고 하시면 이어서 진행"을 붙인다.
