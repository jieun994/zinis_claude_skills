## 1. 자료 유형별 점검 체크리스트

### API 명세서
읽을 때 정리할 것 / 기획자 관점에서 짚을 것:
- **엔드포인트**: 무엇을 하는 API인지 (예: `GET /orders` = 주문 목록 조회)
- **요청(Request)**: 무엇을 줘야 하나 — 필수/선택 파라미터 구분
- **응답(Response)**: 무엇이 돌아오나 — 필드별 의미
- **에러/상태 코드**: 실패하면 뭐가 오나 (없으면 → "에러 응답 포맷 확인 필요")
- **기획자 체크**: 인증·권한이 필요한가? / 페이징(대량 조회)은 어떻게? / 삭제·취소 API가 있나? / 필수값 빠지면 어떤 화면 처리가 필요한가?

### DB 스키마 / ERD
- **테이블**: 무슨 데이터를 담는지 (예: `orders` = 주문 정보)
- **주요 컬럼**: 각 칸이 무슨 값인지, 필수(NOT NULL)인지 빈 값 허용인지
- **관계**: 테이블 간 연결 (1:N 등) — "한 회원이 여러 주문"처럼 평이하게
- **기획자 체크**: 지우면 연쇄 삭제되나(회원 삭제 시 주문도?) / 상태값 종류는 몇 개인가 / 화면에 보여줄 데이터가 실제로 저장되나 / 이력·로그가 남나

### 기술문서 / 개발가이드
- **목적**: 이 문서가 설명하는 것 (연동 방식·아키텍처·정책 등)
- **핵심 규칙·제약**: 개발이 지켜야 하는 조건
- **전제**: 무엇을 가정하는지 (로그인 상태·특정 환경 등)
- **기획자 체크**: 우리 기획 가정과 다른 점은? / 일정에 영향 주는 제약은? / 사용자가 체감하는 부분은?

## 2. 자주 나오는 용어 사전 (평이 설명 + 왜 이렇게)

> 자료에 나온 용어만 골라 쓴다. 자료에 없는 용어를 미리 채워 넣지 않는다.

| 용어 | 쉬운 뜻 | 왜 이렇게 하는지 |
|---|---|---|
| API | 두 프로그램이 정보를 주고받는 창구 | 화면(앱)과 서버가 대화하는 통로 |
| 엔드포인트(endpoint) | API의 각 주소(기능 한 개) | 기능마다 주소를 나눠 관리 |
| GET / POST / PUT / DELETE | 조회 / 생성 / 수정 / 삭제 | 하는 일을 종류로 구분 |
| 파라미터(parameter) | API에 같이 보내는 입력값 | 무엇을 조회·처리할지 지정 |
| 필수(required) / 선택(optional) | 꼭 줘야 함 / 안 줘도 됨 | 필수 누락 시 화면 처리 필요 |
| nullable | 값이 비어 있어도 됨 | 빈 값일 때 화면·로직 대비 필요 |
| 상태 코드(200/400/404/500) | 성공 / 잘못된 요청 / 없음 / 서버 오류 | 결과를 숫자로 표준화 |
| 테이블(table) | 데이터를 담는 표 | 종류별로 데이터를 나눠 저장 |
| 컬럼(column) / 필드(field) | 표의 각 칸(항목) | 데이터의 한 조각 |
| PK(기본키) | 각 줄을 구분하는 고유 번호 | 중복 없이 하나를 특정 |
| FK(외래키) | 다른 표와 연결하는 칸 | 표끼리 관계를 맺음 |
| 1:N 관계 | 하나에 여럿이 딸림 | 예: 회원 1명 - 주문 여러 개 |

새 용어가 나오면 위 형식(쉬운 뜻 + 왜)으로 추가 설명한다.

## 3. 4단 출력 형식 (Step 5)

```markdown
# {자료명} 해설

> 자료 유형: API 명세 / DB 스키마 / 기술문서 · 이미지 판독분은 [이미지 인식 기반] 표기

## 한눈에
- {이 자료가 뭔지 1~3문장 또는 개조식 요약}

## 풀어보기
- {항목}: {쉬운 설명} (자료의 ○○ 기준)
- {항목}: {설명} [개발자 확인 필요 — 이유]
{항목이 많으면 표로 정리해도 됨}

## 용어 노트
- {용어}: {쉬운 뜻} — {왜 이렇게 하는지}

## 기획자 체크포인트
**기획·일정 영향**
- {영향}

**빠진 것 (통상 있어야 하는데 없음)**
- {누락}

**개발자에게 물어볼 질문** (그대로 복사해 쓰세요)
1. "{질문 문장}"
2. "{질문 문장}"
```

### 출력 스타일 규칙 (필수)
- **이모지·아이콘 금지.** 섹션 제목은 텍스트만(`## 한눈에`). ★·📌 등 장식 기호 안 씀.
- **개조식(명사형).** "~합니다" 서술형 대신 "~함 / ~필요 / ~없음"으로 짧게 끊음. 군더더기·수식어 뺌.
- **표 허용.** 항목이 여럿이면 표로 정리해 가독성 확보.
- 근거 없는 문장 금지. "일반적으로는~"으로 자료에 없는 내용 채우기 금지.
- 읽은 것(풀어보기)과 확인할 것(체크포인트)을 절대 섞지 않음.
- 질문은 막연한 "확인 필요"가 아니라 개발자가 바로 답할 수 있는 **구체적 문장**으로.
