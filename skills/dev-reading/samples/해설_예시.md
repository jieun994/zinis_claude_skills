# 결과 예시 — 개발자료 해설

> `dev-reading` 스킬의 결과 예시입니다.
> 아래 **가상의 API 명세**(실제 고객 데이터 아님)를 붙여넣고 "이 API 명세 풀어줘"라고 했을 때 나오는 결과물입니다.
> 산출물은 아이콘 없이 개조식으로 작성되며, 표는 사용합니다.

---

## (입력한 원본 — 개발자가 준 가상 API 명세)

```
GET /api/v1/deliveries/{orderId}
설명: 주문의 배송 상태 조회

Path
- orderId (string, required): 주문 번호

Response 200
- status (string): READY | SHIPPING | DONE
- courier (string, nullable): 택배사명
- invoiceNo (string, nullable): 송장번호
- eta (string): 도착 예정일 (YYYY-MM-DD)

Response 404: 주문 없음
```

# 배송 상태 조회 API 해설

> 자료 유형: API 명세

## 한눈에
- 주문 번호 하나로 해당 주문의 **배송 상태를 조회**하는 API
- 확인 가능 항목: 배송 단계 · 택배사 · 송장번호 · 도착 예정일 (Response 200 기준)

## 풀어보기

**무엇을 주면**
- 주문 번호(`orderId`) 1개 — 필수 (Path, required 기준)

**무엇이 나오나**

| 항목          | 뜻                                       | 비고                                        |
| ----------- | --------------------------------------- | ----------------------------------------- |
| `status`    | 배송 단계: READY(준비)/SHIPPING(배송중)/DONE(완료) | 3단계뿐                                      |
| `courier`   | 택배사명                                    | 빈 값 가능(nullable) · [개발자 확인 필요 — 언제 채워지는지] |
| `invoiceNo` | 송장번호                                    | 빈 값 가능(nullable)                          |
| `eta`       | 도착 예정일(YYYY-MM-DD)                      | —                                         |

**실패 시**
- 주문 없으면 `404`(없음) 반환 (Response 404 기준)

## 용어 노트
- 엔드포인트: API의 기능 하나(여기선 "배송 상태 조회") — 기능마다 주소를 나눔
- nullable: 값이 비어 있어도 되는 항목 — 화면에서 빈 값 처리 필요
- 상태 코드 404: "요청 대상 없음" — 결과를 숫자로 표준화

## 기획자 체크포인트

**기획·일정 영향**
- `courier`·`invoiceNo` 빈 값 가능 → 배송 준비중 화면에 "택배사/송장 미정" 표시 처리 필요
- 배송 단계 3개뿐(취소·반품 없음) → 취소/반품 상태가 화면에 필요하면 이 API로는 부족

**빠진 것 (통상 있어야 하는데 없음)**
- 인증·권한 정보 없음 → 로그인 본인만 조회 가능한지 불명확
- 404 외 실패 응답(서버 오류 등) 포맷 없음

**개발자에게 물어볼 질문** (그대로 복사해 쓰세요)

2. "취소·반품 상태도 이 API로 확인되나요, 아니면 별도 API가 있나요?"
3. "이 조회는 로그인한 본인만 가능한가요? 권한 체크가 있나요?"
