# 개발자료 해설 스킬 (dev-reading, 기획자용)

## 1. 무엇을 하나요

- 개발자에게 받은 자료(**API 명세서 · DB 스키마/ERD · 기술문서**)를 비개발 기획자가 이해하도록 평이한 말로 해설
- 단순 번역에 그치지 않음
  - **개발자에게 뭘 물어봐야 하는지** 제시
  - **우리 기획·일정에 미치는 영향**까지 정리
- 가장 중요한 특징: **모르는 건 지어내지 않음** — 자료에 없는 내용은 `[개발자 확인 필요]`로 표시해 잘못된 이해로 엇나가는 일 방지

## 2. 어떻게 쓰나요

- 개발자에게 받은 자료를 붙여넣거나 파일로 첨부하고 아래처럼 요청

- "이 API 명세 풀어줘"
- "이 ERD(DB 구조) 해석해줘"
- "개발자한테 받은 문서인데 이해가 안 가"
- "기획자 입장에서 이 자료 어떻게 봐야 해?"

- 이미지 캡처·표 붙여넣기·PDF/워드 파일도 가능
- 대상 아님: 회의록·기획서 등 개발자료가 아닌 것, 소스코드

## 3. 결과 구성 (4단)

| 섹션 | 내용 |
|---|---|
| 한눈에 | 이 자료가 뭔지 한두 문장 요약 |
| 풀어보기 | 항목별로 쉽게 번역 (근거와 함께) |
| 용어 노트 | 나온 전문용어를 쉬운 말로 + 왜 이렇게 하는지 |
| 기획자 체크포인트 | 기획·일정 영향 / 빠진 것 / **개발자에게 바로 물을 질문** |

- 산출물은 **아이콘 없이, 개조식(짧은 명사형)** 으로 작성. 표는 사용

## 4. 예시

- **입력**: 배송 상태 조회 API 명세 붙여넣기 + "이 API 명세 풀어줘"
- **결과 일부**:

> **한눈에**
> - 주문 번호로 배송 상태를 조회하는 API
>
> **기획자 체크포인트**
> - 택배사·송장번호 빈 값 가능 → "미정" 표시 처리 필요
> - 개발자에게 물어볼 질문: "취소·반품 상태도 이 API로 확인되나요?"

자세한 예시: [samples/해설_예시.md](samples/해설_예시.md)
