# IA(사이트 구조 및 기능정의) 생성 스킬

서비스 설명을 받아 메뉴 트리를 설계하고, `scripts/build_ia.py`로 엑셀 IA를 만든다.
출력은 신지은님의 실제 IA 양식(NO 자동번호, Depth1~5, Front/Admin 시트 분리, 권한 O/X)에
기능정의 컬럼(페이지 정의·세부 사항·page·개발구분)을 더한 형태다. 스타일·색상은
wbs-generator 샘플을 따른다(컬럼 헤더=검정+흰글씨, Depth1=진회색·Depth2=연노랑,
Malgun Gothic). **표지·사용안내·개정이력 시트가 자동 생성**된 뒤 IA 시트들이 붙는다.
(사람용 설명은 README.md 참조)

## 핵심 원칙

- **입력이 빈약하면 바로 만들지 않는다.** "쇼핑몰 만들어줘"처럼 한 줄만 오면,
  서비스 성격·대상 사용자·핵심 기능을 먼저 1~2개 묻고 생성한다. 임의 생성 금지.
- **Depth는 억지로 맞추지 않는다.** 랜딩처럼 얕으면 1~2단계, 복잡하면 4~5단계까지.
  서비스에 맞는 자연스러운 깊이로 짠다(최대 5).
- **컬럼은 사용자가 고른다(가변 요소).** Code(첫 열)·Depth·비고는 항상 넣고, 나머지
  (Tab/서브플로우·페이지 정의·세부 사항·Page·개발구분·권한)는 시작할 때 물어서
  필요한 것만 넣는다.
- **트리 먼저 보여주고 확정받는다.** 엑셀을 만들기 전에 들여쓰기 트리로 미리 보여주고,
  사용자가 수정·확정한 뒤 파일을 만든다.

## 작동 순서

### 0. 시작 인터뷰 (순서대로 묻고, 답 받은 뒤 생성)

한꺼번에 묻지 말고 대화로 진행한다. 모르면 제안·기본값을 제시한다.

1. **프로젝트명** — 제목에 들어간다.
2. **서비스 개요** — 무엇을 하는 서비스인지 1~3문장. 메뉴 분해의 기준이라 가장 중요.
   기획서/PRD 파일을 첨부하면 먼저 읽어 반영한다.
3. **대상 사용자 / 화면 구분** — 일반 사용자만인지, 관리자(Admin) 화면도 있는지.
   관리자가 있으면 **Front / Admin 시트를 나눈다.** 관리자 화면이 역할별로 권한이
   다르면(예: 조사원/지방실사/관리자) 역할 목록을 받아 권한(O/X) 칸을 만든다.
4. **넣을 컬럼 선택** — 아래 "컬럼 선택"의 AskUserQuestion을 호출한다(가변 요소).
5. **작성자 · 버전** (선택) — 우측 상단에 들어간다. (작성일 = 오늘 자동)

### 컬럼 선택 (★ AskUserQuestion, multiSelect)

Code(첫 열)·Depth·비고는 항상 포함된다. 그 외 컬럼을 사용자가 고르게 한다.
관리자(Admin) 화면이 있으면 "권한(역할별 CRUD)" 옵션도 함께 제시한다.

```json
{
  "questions": [
    {
      "question": "IA에 어떤 컬럼을 넣을까요? 필요한 걸 모두 골라주세요. (Code·Depth·비고는 기본 포함)",
      "header": "컬럼 선택",
      "multiSelect": true,
      "options": [
        {"label": "페이지 정의/요구사항 (추천)", "description": "각 화면이 어떤 유형인지(디자인 페이지·게시판·폼 등)와 요구사항을 적어요."},
        {"label": "개발구분(디자인/퍼블/개발) (추천)", "description": "각 화면을 누가 작업하는지 ●로 표시해요. 페이지 유형 보고 자동 제안해요."},
        {"label": "Page(페이지 수)+총계", "description": "화면별 페이지 수를 적고 맨 아래 총계를 자동 합산해요."},
        {"label": "세부 사항", "description": "화면 안 세부 항목·분기(예: 입력 필드, 상태 구분)를 적어요."},
        {"label": "Tab/서브 플로우", "description": "한 화면 안의 탭·서브 흐름을 적어요."},
        {"label": "권한(역할별 CRUD)", "description": "관리자 화면용. 역할마다 C/R/U/D 가능 여부를 표시해요. Admin 시트에 적용."}
      ]
    }
  ]
}
```

선택 결과를 입력 JSON의 `columns` 배열로 매핑한다:
페이지 정의→`page_def`, 개발구분→`dev_split`, Page→`page`, 세부 사항→`detail`,
Tab/서브플로우→`tab`. 권한은 해당 시트에 `roles`(역할 목록)를 지정하면 자동 생성된다.

### 1. 참고 자료 확인

기획서·요건 문서·경쟁사 메뉴 메모를 첨부/붙여넣어 주면 먼저 읽어 초안에 반영한다.
(없으면 개요만으로 분해한다.) 경쟁사 URL은 실시간 접근이 어려우니 "쿠팡 같은
쇼핑몰" 식 레퍼런스로 받아 관례 구조를 적용한다.

### 2. 메뉴 트리 설계 — 단계적으로 ("제안 → 확인")

개요와 서비스 유형 관례를 바탕으로 **1depth(대분류) 초안을 먼저 제안**한다.
서비스 유형별 관례 메뉴와 페이지 유형은 `references/page-types.md`를 참조한다.

- **(A) 1depth 제안** — 대분류 목록을 표로 제시 → 사용자 수정·확정.
- **(B) 하위 depth 채우기** — 각 대분류 아래 2~5depth를 서비스에 맞게 제안.
  억지로 깊이를 맞추지 않는다. Head/Footer 같은 공통 영역도 포함한다.
- **(C) 컬럼 값 채우기** — 선택된 컬럼에 맞게 페이지 유형·개발구분(●)·page 수 등을
  자동 제안한다. 개발구분 ● 기본값은 `references/page-types.md`의 추천 기준을 따른다.
  **칸을 단어 한두 개로 비워두지 말고, 아래 "콘텐츠 품질 기준"대로 기획자가 바로 쓸
  수준으로 채운다.**

#### 콘텐츠 품질 기준 (★ raw 금지 — 정교하게 채운다)

기획자가 이 IA만 보고 화면을 그릴 수 있을 만큼 구체적으로 쓴다. 단순 명사 나열은 금지.

- **페이지 정의 / 요구사항** — `유형 — 핵심 목적` 형태로 쓴다. 그 화면이 무엇을 하는
  화면인지가 한눈에 보이게.
  - raw: `"게시판"` → good: `"리스트 게시판 — 공지 목록·상세, 상단 고정"`
  - raw: `"로그인"` → good: `"로그인 — 계정 인증, 소셜로그인 포함"`
- **세부 사항** — 그 화면의 **구체 요소**를 콤마로 나열한다. 다음을 떠올려 채운다:
  입력 필드와 검증 규칙 / 목록 컬럼·정렬·검색·페이지네이션 / 상태 구분 /
  외부 연동(지도·결제·인증·메일) / 예외·분기 처리.
  - raw: `"회원가입 폼"` → good: `"아이디(중복검사), 비밀번호(8자+영문/숫자/특수),
    이메일(형식검증), 주소(도로명 검색), 휴대폰 본인인증"`
  - raw: `"문의"` → good: `"필드(이름·연락처·이메일·유형·내용·첨부), reCAPTCHA,
    접수 시 관리자 메일·신청자 자동응답"`
- **개발구분(●)** — 페이지 유형 기준 자동 제안(`references/page-types.md`). 정적 화면은
  디자인 위주, 게시판·폼·로그인은 퍼블리싱+개발, 연동·관리자 기능은 개발.
- **권한(CRUD)** — 역할 성격에 맞는 기본값을 제안한다. 조회 전용 역할은 대부분 `R`,
  운영 역할은 콘텐츠 `CRUD`·회원/설정 `RU`, 최고관리자는 대부분 `CRUD`.
- **page 수** — 단순 화면은 1, 탭·다단계 폼·목록+상세는 2 이상으로 현실적으로 잡는다.

상세 예시와 페이지 유형별 작성법은 `references/page-types.md`를 참조한다.

### 3. 트리 미리보기 + 확정 (review)

엑셀을 만들기 전에 **대화창에 들여쓰기 트리**로 보여준다. 예:

```
■ Front
  로그인
  일계부
    월별 현황
    일계부 등록
      수입 / 지출 / ...
  내 정보
    개인정보 변경 / 공지사항
■ Admin (권한: 조사원/지방실사/관리자)
  대시보드  [O/O/O]
  ...
```

"이렇게 수정해줘" 예시를 함께 안내해 진입 장벽을 낮춘다.
(예: "Depth 하나 더 추가해줘", "관리자 화면 빼줘", "공지사항은 2차로 빼줘")
사용자가 확정하면 다음 단계로 간다.

### 4. JSON 작성 후 스크립트 실행

확정된 트리를 입력 JSON으로 만들고 스크립트를 실행한다:

```
cd skills/ia-builder
PYTHONIOENCODING=utf-8 PYTHONUTF8=1 python scripts/build_ia.py <입력.json> <출력.xlsx>
```

Windows PowerShell에서는 `$env:PYTHONUTF8=1; python scripts/build_ia.py <입력.json> <출력.xlsx>`.
`python`이 없으면 `py -3`을 쓴다. 사전 요구: `pip install openpyxl`.
출력 파일명은 `프로젝트명_IA_YYYYMMDD.xlsx` 식으로.

### 5. 결과 안내

생성 경로를 알려주고, "엑셀에 표지·사용안내·개정이력 + 정보구조도 시트가 있다"고
안내한다. (표지·사용안내·개정이력은 자동 생성되며, 작성자/작성일/버전은 표지와
개정이력에 들어간다.)
IA는 보통 PRD·WBS의 앞단 산출물이므로, 필요하면 "이 IA로 PRD를 만들까요?"처럼
다음 단계를 제안할 수 있다. (PRD=기능 명세, WBS=일정 / IA=구조 — 역할이 다르다.)

## 입력 JSON 구조

```json
{
  "project": {
    "name": "프로젝트명",
    "author": "신지은", "author_date": "2026-06-26", "version": "v0.1"
  },
  "doc_type": "사이트 구조 및 기능정의",
  "columns": ["page_def", "dev_split", "page", "detail", "code", "tab"],
  "max_depth": 5,
  "sheets": [
    {
      "sheet_name": "Front 정보구조도",
      "title": "사이트 구조 및 기능정의",
      "rows": [
        {"depth": 1, "name": "로그인", "code": "1", "page_def": "디자인 페이지", "dev": ["디자인"]},
        {"depth": 1, "name": "일계부", "code": "2"},
        {"depth": 2, "name": "월별 현황", "code": "2.1", "page_def": "리스트 게시판", "page": 1, "dev": ["퍼블리싱", "개발"]},
        {"depth": 2, "name": "공지사항", "code": "2.2", "note": "2차", "phase2": true}
      ]
    },
    {
      "sheet_name": "Admin 정보구조도",
      "title": "Admin 정보구조 및 권한",
      "roles": ["에디터", "운영자", "최고관리자"],
      "columns": ["page_def", "page"],
      "rows": [
        {"depth": 1, "name": "대시보드", "code": "A.1", "page_def": "대시보드", "auth": {"에디터": "R", "운영자": "R", "최고관리자": "R"}},
        {"depth": 1, "name": "회원 관리", "code": "A.2", "page_def": "리스트 + 상세", "auth": {"에디터": "R", "운영자": "RU", "최고관리자": "CRUD"}}
      ]
    }
  ]
}
```

### 필드 참고

- **project**: `name`(필수), `author`·`author_date`·`version`(선택, 표지·개정이력에 들어감).
- **doc_type**(선택): 문서 종류 라벨. 표지 부제·사용안내 제목에 쓰인다. 없으면 첫 시트 title 사용.
- **표지·사용안내·개정이력 시트는 자동 생성**되므로 JSON에 적지 않는다.
- **columns**: 넣을 선택 컬럼 키 배열. Code(첫 열)·Depth·비고는 항상 포함되므로 적지 않는다.
  키: `tab` `page_def` `detail` `page` `dev_split`. (권한은 `roles`로 자동 생성)
- **max_depth**: 최대 Depth 칸 수(기본 5). 트리 최대 깊이에 맞춰 정한다.
- **sheets**: 시트 배열. 시트마다 `columns`를 따로 줄 수 있다(없으면 상위 `columns` 사용).
  - `sheet_name`: 시트 탭 이름. `title`: 제목줄에 들어갈 문구.
  - `roles`: (Admin 시트) 권한 역할 목록 → 역할마다 C/R/U/D 4칸이 생성된다.
  - `rows`: 메뉴 행. 각 행은 `depth`(1~max_depth)와 `name` 필수.
    - `code`: 화면 코드(첫 열). 예: `"1.1"`, `"A.3.2"`, `"H.1"`.
    - 선택: `tab` `page_def` `detail` `note` 텍스트, `page` 숫자.
    - `dev`: 개발구분 배열. `"디자인"` `"퍼블리싱"` `"개발"` 중 해당하는 것 → ● 표시.
    - `auth`: 역할별 권한. `{"역할명": "CRUD부분집합"}` 형태. 예: `"R"`(조회만),
      `"RU"`(조회·수정), `"CRUD"`(전체), `""`(권한 없음). 허용된 동작 칸에 O, 나머지는 연회색.
    - `phase2`: true면 행 전체가 회색(2차 표시).
- Page 총계는 `SUM` 수식으로 자동 계산된다. 정보구조도 시트는 눈금선이 꺼져 있다.

## 엣지 케이스 대응

- **입력이 한 줄뿐**: 생성 전에 서비스 성격·대상·핵심 기능을 1~2개 묻는다.
- **긴 PRD 첨부**: 먼저 핵심 기능 영역 5~7개로 압축한 뒤 메뉴로 재구성한다.
- **역할이 섞인 서비스**(소비자/관리자 등): Front/Admin 시트를 나눌지 먼저 확인한다.
- **깊이가 얕거나 깊음**: 3단계를 강제하지 않는다. 얕으면 얕게, 깊으면 최대 5단계.
- **메뉴명 중복**: 같은 이름이 여러 곳에 반복되면 비고에 구분 메모를 제안한다.
- **엑셀 생성 실패**: openpyxl 미설치면 `pip install openpyxl` 안내. 그래도 안 되면
  들여쓰기 트리/마크다운 표로 대체 제공하고 "복사해 엑셀에 붙여넣기 가능"이라 안내한다.
