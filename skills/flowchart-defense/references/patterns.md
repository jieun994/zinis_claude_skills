# 방어설계 패턴 카탈로그 (패턴 1~8)

> 고객이 준 흐름도는 대부분 **정상 경로(happy path)** 위주다. 이 카탈로그는 흐름도의
> 모든 노드를 순회하며 "여기 빠진 게 없나"를 체계적으로 점검하기 위한 8개 패턴이다.
> 각 패턴은 **[정의 / 탐지 조건 / 보강 방법 / 적용 예시]** 4블록으로 통일돼 있다.
>
> **사용법**: 흐름도의 각 노드·엣지에 대해 패턴 1→8을 순서대로 적용한다. 해당되면
> 보강안을 만들고, 해당 안 되면 "해당없음"으로 기록한다. 모든 보강에는 반드시
> "어느 패턴 때문에 추가했는지"를 근거로 남긴다.

---

## 패턴 1 — 분류·라벨 명확화

**정의**: 노드의 성격(사용자 행동/UI/API/판단/시스템)을 구분하고, 판단 노드의 게이트
조건과 분기 라벨을 명시한다. 같은 종류의 게이트가 중복되거나 라벨이 비면 흐름이 모호해진다.

**탐지 조건**:
- 판단(분기) 노드에 "예/아니오" 또는 조건 라벨이 비어 있다
- 게이트가 무엇을 묻는지 불명확하다 (존재 여부? 상태? 조건?)
- 비슷한 게이트가 두 번 나오는데 구분이 안 된다 (예: "상품 있어?"가 두 곳)

**보강 방법**:
- 게이트를 **존재(있/없) · 상태(예정/중/완료) · 조건(발화 일치)** 세 종류로 나눠 라벨을 명확히 한다
- 모든 분기 화살표에 조건 라벨을 붙인다
- 노드를 5분류 색상(사용자/UI/API/판단/시스템)으로 태깅한다 (output-templates.md 참조)

**적용 예시 (배송일정변경)**:
- 세 게이트를 존재(주문 존재?)·상태(배송 예정·중?)·조건(발화 조건에 맞는 상품?)으로 구분
- "배송상품 존재?"(상태)와 "발화 조건에 맞는 상품?"(조건)의 중복을 분리해 라벨 명시

## 패턴 2 — 진입·전제 검증

**정의**: 본 흐름을 시작하기 전에 필요한 전제(데이터 존재·세션·권한·로그인)를 먼저
확인하고, 전제가 없을 때 갈 대체 시나리오를 연결한다.

**탐지 조건**:
- 흐름이 "데이터가 당연히 있다"고 가정하고 바로 시작한다
- 로그인/권한/세션이 필요한데 확인 단계가 없다
- 전제 실패 시 어디로 가는지 정의되지 않았다

**보강 방법**:
- 흐름 진입부에 전제 게이트를 추가한다 (예: "주문내역 존재?", "로그인 상태?")
- 전제가 없을 때의 **대체 시나리오 전환**을 연결한다 (다른 인텐트/안내+버튼)

**적용 예시 (배송일정변경)**:
- "주문내역 존재?"(없음 → 타쇼핑몰 주문등록 인텐트 전환)
- "배송 예정·배송 중 상품 존재?"(없음 → 타쇼핑몰 주문등록·제품추천 전환)

---

## 패턴 3 — 빈 결과(0건) 분기

**정의**: 조회/검색/필터 결과가 0건일 때 빈 화면을 보여주지 않도록, 0건 분기와
대안(전체 보기·다른 상품·종료)을 연결한다.

**탐지 조건**:
- 리스트·캘린더·검색 결과를 노출하는데 "결과 0건"일 때가 그려져 있지 않다
- API 조회 결과를 그대로 화면에 뿌리는데, 빈 응답 처리가 없다

**보강 방법**:
- 결과를 노출하는 모든 지점 앞에 "결과 존재?" 게이트를 둔다
- 0건일 때 **안내 + 대안 버튼**(전체 보기/다른 조건/종료)을 연결한다

**적용 예시 (배송일정변경)**:
- 발화 조건 불일치(0건) → "말씀하신 상품을 못 찾았어요" + 전체 보기 버튼
- 변경 가능 날짜 0건 → "변경 가능한 날짜가 없어요" + 다른 상품/종료

---

## 패턴 4 — 발화 추출·결손 입력

**정의**: 사용자 발화에서 이미 말한 값은 추출·저장해 재입력을 막고, 빠진 값(결손분)만
UI(폼·캘린더)에서 입력받는다. 발화로 다 말했는데 또 묻거나, 안 말했는데 안 묻는 누락을 방지한다.

**탐지 조건**:
- 사용자가 발화로 제공할 수 있는 값을 무조건 화면에서 다시 묻는다
- 또는 필요한 값인데 발화에도 화면에도 받는 곳이 없다
- "발화 언급 여부"에 따른 분기가 없다

**보강 방법**:
- 각 입력값마다 "발화에서 언급?" 게이트를 두고, 언급분은 **저장 후 UI에 반영**, 결손분만 입력받는다
- 발화 추출 조합(둘 다 언급/하나만/둘 다 미언급)별 처리를 표로 정리한다

**적용 예시 (배송일정변경)**:
- 대상일/희망일 각각 "발화 언급?" → 언급분 저장 후 캘린더 반영, 결손분만 캘린더 입력
- 조합 4가지(둘 다/대상일만/희망일만/둘 다 미언급)를 표로 명시

---

## 패턴 5 — 재시도·반복 제한

**정의**: 조회 실패·일시 오류 시 재시도 분기를 두되, 무한 반복을 막는 종료 경로와
반복 횟수 가드를 함께 둔다.

**탐지 조건**:
- 일시 오류/실패가 나는 API인데 재시도 경로가 없다
- 재시도는 있는데 빠져나갈 종료가 없다 (무한 루프 위험)
- 같은 노드로 되돌아가는 회귀에 횟수 제한이 없다

**보강 방법**:
- 실패 안내 뒤에 "재시도?" 게이트를 두고, **재시도 → 원래 API, 종료 → 종착 노드**로 연결한다
- 회귀 흐름에는 반복 한계(예: 3회)를 명시하거나, 매 회귀마다 종료 선택지를 제공한다

**적용 예시 (배송일정변경)**:
- 주문내역 조회 실패 → "일시 오류로 조회가 안 됐어요" → 재시도?(재시도 → 조회 API / 종료 → 종착)

---

## 패턴 6 — 취소·수정 회귀

**정의**: 최종 확인(쓰기 직전) 단계에 확인/수정/취소 분기를 두고, 수정은 적절한 이전
지점으로 회귀, 취소는 안내 후 종료시킨다.

**탐지 조건**:
- 확인 화면에서 "확인"만 있고 수정·취소 경로가 없다
- 수정을 누르면 어디로 돌아가는지 정의가 없다
- 취소 후 종착 처리가 없다

**보강 방법**:
- 확인 단계에 **확인 / 수정 / 취소** 3분기를 둔다
- 수정은 무엇을 고치느냐(상품/날짜 등)에 따라 해당 입력 지점으로 회귀시킨다
- 취소는 "취소했어요" 안내 후 종착으로 마감한다

**적용 예시 (배송일정변경)**:
- 최종 확인 → 확인(변경 API) / 수정(상품 → 리스트, 날짜 → 캘린더) / 취소(안내 후 종료)

---

## 패턴 7 — 노드 분리·선행 데이터

**정의**: 한 노드에 API 호출과 판단이 뭉쳐 있으면 **API 노드 + 판단 노드**로 분리하고,
화면 노출 전에 필요한 데이터를 확보할 **선행 조회 API**가 있는지 점검한다.

**탐지 조건**:
- 한 박스가 "조회하고 분기까지" 한 번에 한다 (API+판단 혼재)
- 캘린더·리스트를 노출하는데, 그 데이터를 어디서 가져오는지 API가 없다
- 결과 판단(완료/실패)이 호출 노드에 묻혀 있다

**보강 방법**:
- API 호출 노드와 그 결과를 판단하는 게이트 노드를 분리한다 (예: 조회 API → "결과?" 게이트)
- 화면 노출 전 데이터원이 비면 **선행 조회 API 노드**를 추가한다 (※ 실제 API 존재 여부는 확인요망으로)

**적용 예시 (배송일정변경)**:
- 주문내역 조회·배송일정 변경을 각각 API 노드 + 판단 노드(B0/G0)로 분리
- 캘린더 노출 전 "변경 가능 날짜 조회 API" 선행 노드 추가 (API 미확정 → 확인요망)

---

## 패턴 8 — 막다른 길 제거·실패 후속

**정의**: 모든 말단(leaf)을 명시적 종착 노드로 마감하고, 쓰기성 API 실패에는 후속
처리(콜백·접수)를, 조회성 API 실패에는 재시도/안내를 연결한다.

**탐지 조건**:
- 화살표가 어디로도 안 가고 끊긴 노드가 있다 (막다른 길)
- 쓰기 API(등록/변경/결제) 뒤에 "실패하면?"이 없다
- 종료처럼 보이지만 종착 노드로 명시되지 않은 곳이 있다

**보강 방법**:
- 모든 leaf를 **종착 노드**(종료·전환·대기)로 명시한다
- 쓰기 API 결과를 완료/실패로 나누고, 실패는 **콜백 접수·재처리 안내** 등 후속을 연결한다
- 조회 API 실패는 재시도/안내로 연결한다 (패턴 5와 연계)

**적용 예시 (배송일정변경)**:
- 배송일정변경 실패 → "처리가 지연돼 접수했어요 · 콜백 안내" → 콜백 접수 → 대기 종착
- 모든 leaf(타쇼핑몰 전환·제품추천 전환·각종 종료)를 종착 노드로 마감

---

## 적용 우선순위 & 과잉 보강 경계

- **항상 점검**: 패턴 1(라벨)·3(빈결과)·6(취소수정)·8(막다른길)은 거의 모든 흐름에 해당한다.
- **맥락 의존**: 패턴 2(전제)·4(발화추출)·5(재시도)·7(노드분리)은 흐름 성격에 따라
  해당 없을 수 있다 — 억지로 끼워넣지 않는다.
- **과잉 보강 금지**: 챗봇 맥락에 안 맞는 분기(예: 결제 없는 단순 안내봇에 결제 실패 흐름)는
  추가하지 않는다. 패턴은 **제안하되 근거를 달고**, 애매하면 보강 대신 "확인요망"으로 올린다.
- **원본 변형 금지**: 패턴 적용은 **추가**만 한다. 원본 노드의 의미를 바꾸지 않는다.
