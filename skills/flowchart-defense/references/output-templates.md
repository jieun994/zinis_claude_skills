# 산출물 템플릿 (검토리포트.md)

검토리포트는 아래 8개 섹션 순서로 작성한다. 모든 표는 마크다운 표로 쓴다.

> **최상단 고지문(자동 삽입)**:
> `> ⚠️ 이 문서는 원본 흐름도 + AI 방어설계 보강을 포함합니다. [보강]·[추정] 표시 항목은 검토가 필요합니다.`

---

## 섹션 1. 보강 플로우차트 (mermaid)

원본 노드와 보강 노드를 **시각적으로 구분**한다. 할루시네이션 방지의 핵심이다.

| 구분 | 의미 | mermaid 표현 |
|------|------|--------------|
| **원본** | 고객 흐름도에 있던 노드 | 실선 박스 (기본) |
| **보강** | 패턴으로 추가한 노드 | 라벨 앞에 `[추가]`, classDef `add`(점선) |
| **추정** | 불확실해서 가정한 노드 | 라벨 앞에 `[추정]`, classDef `guess`(회색) |

**색상 분류(노드 성격)** — 위 구분과 별개로 노드 종류를 색으로 태깅:

```mermaid
classDef user   fill:#e3f2fd,stroke:#1565c0,stroke-width:2px,color:#0d47a1
classDef ui     fill:#e8f5e9,stroke:#2e7d32,stroke-width:1px,color:#1b5e20
classDef api    fill:#fff3e0,stroke:#e65100,stroke-width:2px,color:#bf360c
classDef logic  fill:#f3e5f5,stroke:#6a1b9a,stroke-width:1px,color:#4a148c
classDef system fill:#eceff1,stroke:#546e7a,stroke-width:1px,color:#263238
classDef add    stroke-dasharray:5 5,stroke-width:2px
classDef guess  fill:#fafafa,stroke:#9e9e9e,stroke-dasharray:2 2,color:#616161
```

- `flowchart TB`로 시작, 노드는 영문 ID + 한글 라벨.
- 게이트는 `{...}`, 사용자/UI/시스템 박스는 `[...]`, 종착은 `([...])`.

## 섹션 2. 분류 기준 표

| 구분 | 색상 | 해당 노드 |
|------|------|-----------|
| 사용자 행동 | 파랑 | (노드 나열) |
| 화면 노출(UI) | 초록 | (노드 나열) |
| API 통신 | 주황 | (노드 나열) |
| 판단/분기 로직 | 보라 | (노드 나열) |
| 시스템 처리 | 회색 | (노드 나열) |

## 섹션 3. 원본 / 보강 / 추정 구분표 ★ (보강 항목)

가장 중요한 표. 모든 노드가 원본인지 AI 보강인지 추정인지 한눈에 보이게 한다.

| 노드 | 구분 | 근거(패턴) | 원본 위치 |
|------|------|-----------|-----------|
| 주문내역 조회 | 원본 | - | 슬라이드1 |
| 재시도 분기 | 보강 | 패턴5 | - |
| 콜백 방식 | 추정 | 패턴8·확인요망 | - |

- `구분` = 원본 / 보강 / 추정 중 하나 (필수)
- 보강·추정에는 반드시 근거 패턴 번호를 단다. 근거 없는 추가는 금지.

---

## 섹션 4. 패턴 적용 체크리스트 ★ (보강 항목)

패턴 1~8을 전수 점검했음을 보여준다.

| 패턴 | 적용 | 적용 위치 / 사유 |
|------|------|-----------------|
| 1 분류·라벨 명확화 | ✅ | 게이트 B0/C0/D0 구분 |
| 2 진입·전제 검증 | ✅ | 주문 존재·배송상품 존재 |
| 3 빈 결과(0건) 분기 | ✅ | D0 0건, CLc 0건 |
| 4 발화 추출·결손 입력 | ✅ | 대상일/희망일 |
| 5 재시도·반복 제한 | ✅ | 조회 실패 재시도 |
| 6 취소·수정 회귀 | ✅ | 최종 확인 분기 |
| 7 노드 분리·선행 데이터 | ✅ | API/판단 분리 |
| 8 막다른 길 제거·실패 후속 | ✅ | 종착·콜백 |

- `적용` = ✅ 적용 / ➖ 해당없음 / ⚠️ 미적용(사유). 해당없음·미적용도 사유를 적는다.

## 섹션 5. 리스크 우선순위 표 ★ (보강 항목)

발견한 구멍을 심각도로 정렬해 "어디부터 고칠지" 의사결정을 돕는다.

| 항목 | 심각도 | 발생 시 사용자 영향 | 권고 |
|------|--------|--------------------|------|
| 쓰기 API 실패 후속 없음 | 🔴 치명 | 변경 유실·재문의 폭증 | 콜백 접수 추가 |
| 빈 캘린더 노출 | 🟡 중요 | 막힌 화면에서 이탈 | 0건 분기 추가 |
| 분기 라벨 누락 | 🟢 권장 | 흐름 해석 혼선 | 라벨 명시 |

- 심각도 = 🔴 치명 / 🟡 중요 / 🟢 권장. 치명부터 위로 정렬.

---

## 섹션 6. Before / After 노드 비교 ★ (보강 항목)

변경 전후를 대조해 보강의 타당성을 보여준다.

| 노드 | Before (원본) | After (보강) | 근거 |
|------|---------------|--------------|------|
| 변경 결과 | 완료만 그려짐 | 완료 / 실패(콜백) 분기 | 패턴8 |
| leaf 노드들 | 화살표 끊김 | 종착 노드로 마감 | 패턴8 |

---

## 섹션 7. 원본 대비 변경 이력 표

| 구분 | 내용 | 근거 |
|------|------|------|
| 노드 분리 | (무엇을 어떻게) | 패턴N — (이유) |
| 0건 분기 | (추가한 분기) | 패턴3 — (이유) |

- `구분` = 노드 분리 / 게이트 명확화 / 0건 분기 / 취소·수정 회귀 / 실패·종착 등
- `근거` = 패턴 번호 + 한 줄 사유

## 섹션 8. 고객 확인요망 (별도 분리) ★

고객에게 되물어야 할 항목. **내부 분석과 섞지 않고 이 섹션(또는 별도 파일)에 모은다.**
배치 모드에서는 `고객확인요망.md`로 별도 파일 생성도 가능.

- **(항목명)**: (무엇이 불확실한지) — 미확정, (확인 대상) 확인 요망
- 예) **변경 가능 날짜 조회 API**: 캘린더에 노출할 날짜의 출처 API가 원본에 없음 — 미확정, 풀무원/BXAI 확인 요망

## 고객 요약본 (4섹션) — 워드 전달본 기본 구조 ★

전체 8섹션은 **기획자(내부)용 MD 마스터**다. 고객에게 보내는 워드(.docx)는 가독성을 위해
**고객이 보고 결정할 것만** 담은 4섹션 요약본으로 만든다. 파일명은 `{흐름명}_고객요약본_YYYYMMDD.md` → docx.

| # | 섹션 | 내용 | 마스터의 어디서 |
|---|------|------|----------------|
| 1 | 보강 흐름도 | mermaid + 작은 색 범례 (원본/보강/추정 구분) | 섹션1 (범례 압축) |
| 2 | 주요 변경 요약 (Top 5) | 변경 내용 + 이유, 한 줄씩 5개 | 섹션7 압축 |
| 3 | 꼭 고쳐야 할 리스크 | 🔴치명·🟡중요만 (🟢권장은 "상세본 참조") | 섹션5 필터 |
| 4 | 고객 확인요망 | 미확정 항목 전부 (고객이 답할 것) | 섹션8 그대로 |

**빼는 것(고객용)**: 분류 기준 상세표·패턴 체크리스트·원본/보강/추정 전체 인벤토리·Before/After 상세.
(전부 MD 마스터에는 남아 있으니 손실 없음.)

> 신뢰가 특히 중요한 고객이면 "원본/보강/추정 구분표"(섹션3)를 5번째로 추가할 수 있다(5섹션 모드).

---

## 작성 규칙

- 추정으로 빈칸을 메우지 않는다. 모르면 `[추정]` 표시 + 확인요망에 적는다.
- 원본 노드의 의미·라벨을 바꾸지 않는다. 보강은 추가만.
- mermaid는 문법 오류로 차트가 깨지지 않게 노드 ID 중복·따옴표 이스케이프를 점검한다.
