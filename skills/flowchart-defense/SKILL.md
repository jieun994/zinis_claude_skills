# flowchart-defense — 흐름도 검토·방어설계

> 고객이 준 플로우차트(PPT·이미지·PDF·mermaid)를 읽어, happy-path에서 빠진
> 예외·실패·빈결과·취소·막다른 길을 **패턴 1~8**로 전수 점검해 보강하고,
> 고객에게 전달할 검토 리포트를 만든다. (사람용 설명은 README.md)

이 스킬의 사용자는 **비개발자 기획자**다. 고객사에서 받은 흐름도를 검토·보강해
다시 고객에게 회신하는 게 목적이다. 산출물이 고객에게 나가므로 **정확성**이 최우선이다.

---

## ⚖️ Iron Law — 절대 규칙 (할루시네이션 방어)

이 스킬에서 가장 위험한 실패는 **AI가 그럴듯하게 지어낸 흐름을 고객이 원본으로 오해하는 것**이다.
아래 4개는 어떤 상황에서도 건너뛰지 않는다.

1. **원본 인벤토리 먼저 확정** — 보강을 시작하기 전에, 원본에서 읽은 노드·연결 목록을
   사용자에게 보여주고 승인받는다(Step 2). 승인 전엔 어떤 보강도 하지 않는다.
2. **원본·보강·추정 끝까지 분리** — 산출물 전체에서 원본/보강/추정을 색·태그·`구분` 컬럼으로
   구분한다. 셋을 섞지 않는다.
3. **모르면 지어내지 말고 멈춰 묻는다** — 라벨·연결·값이 불확실하면 추정으로 메우지 말고
   `[추정]` 표시 + 고객 확인요망에 올린다.
4. **모든 보강에 패턴 근거** — 추가한 모든 노드·분기에 "패턴 몇 번 때문인지" 근거를 단다.
   근거 없는 추가는 금지.

### 합리화 차단 (Excuse → Reality)

| 떠오르는 변명 | 실제 |
|---|---|
| "흐름이 명확하니 원본 확정 생략하고 바로 보강하자" | 원본 오독은 *명확해 보일 때* 가장 많이 샌다. Step 2는 생략 불가. |
| "이 라벨은 당연히 이런 뜻일 거야" | '당연'이 곧 추정이다. `[추정]` 달고 확인요망으로. |
| "배치라 일일이 확인 못 하니 그냥 다 통과시키자" | 배치에선 저신뢰 항목을 *플래그*해서 확인받는다. 전수 통과가 아니다. |
| "고객이 빨리 달라니 확인요망 항목도 채워서 보내자" | 추정을 사실처럼 보내면 책임이 우리에게 온다. 미확정은 미확정으로 전달. |

### Red Flags — 이 말이 나오면 멈춘다
"아마 ~일 것이다" / "보통 이런 흐름은" / "원본에 없지만 넣어두자" → 추정이다. 확인요망으로 분리.

## 모드 판별

| 상황 | 모드 |
|------|------|
| 흐름도 파일 1개 | **단건 모드** (원본 확정 게이트 풀 적용) |
| 흐름도 파일 여러 개 / 폴더 | **배치 모드** (1개당 리포트 1개, 5~8개씩 자동 분할) |

여러 개를 받아도 **절대 하나로 합치지 않는다.** 1개당 독립 리포트를 만든다.

## 워크플로우 (단건 모드)

### Step 1: 입력 판독
**타입**: prompt + script

파일 형식에 따라 흐름을 읽는다. **흐름 구조(노드·화살표 방향)는 Claude 비전 판독이 가장 정확**하다.

- **이미지(PNG/JPG)**: Claude 비전으로 직접 판독. 도형 텍스트 + 화살표 방향을 함께 해석.
- **PDF**: Read 도구의 PDF 페이지 읽기(비전)로 판독. 텍스트 PDF여도 흐름은 레이아웃이 의미라 비전 우선.
- **PPTX**: `scripts/extract_pptx.py`로 도형 텍스트를 추출(누락 방지)하고, 동시에 슬라이드를
  비전으로 읽어 화살표 연결을 복원한다. **PPTX의 화살표 연결관계 복원이 가장 어려우니** 비전을 주 경로로 쓴다.
  ```
  cd skills/flowchart-defense
  PYTHONUTF8=1 python scripts/extract_pptx.py <파일.pptx>
  ```
  Windows PowerShell에서는 `$env:PYTHONUTF8=1; python scripts/extract_pptx.py <파일.pptx>`. `python`이 없으면 `py -3`.
  python-pptx가 있으면 도형 좌표까지, 없으면 XML에서 텍스트만 뽑는다(둘 다 작동).
- **mermaid/MD**: 이미 mermaid면 그대로 파싱해 재검토한다.

판독 결과를 중간 그래프(노드 목록 + 엣지 목록)로 정리한다. 이때 **원본에 있는 것만** 적는다.

### Step 2: 원본 인벤토리 확정 ★ (Iron Law)
**타입**: review

판독한 원본을 사용자에게 보여주고 확인받는다. **이 게이트 없이 Step 3으로 가지 않는다.**

```
원본을 이렇게 읽었어요. 맞는지 봐주세요:

[노드] (N개)
- A: 사용자 발화
- B: 주문내역 조회 API
- B0: 주문내역 존재? (예/아니오)
...

[연결]
- A → B → B0
- B0 --예--> C0
...

⚠️ 못 읽었거나 흐릿한 부분: (있으면 구체적으로)
```

**EXECUTE:** AskUserQuestion으로 확인:

```json
{
  "questions": [{
    "question": "원본 흐름을 이렇게 읽었어요. 맞나요?",
    "header": "원본 확정",
    "options": [
      {"label": "맞아요, 보강 진행 (추천)", "description": "이 원본 위에 방어설계 보강을 시작할게요."},
      {"label": "고칠 게 있어요", "description": "잘못 읽은 노드·연결을 알려주시면 바로잡고 다시 보여드려요."},
      {"label": "원본을 다시 줄게요", "description": "더 선명한 이미지나 다른 형식으로 주시면 다시 읽을게요."}
    ],
    "multiSelect": false
  }]
}
```

추출 신뢰도가 낮으면(텍스트 깨짐·화살표 불명확) 임의로 잇지 말고 "흐릿한 부분"에 명시한다.

### Step 3: 방어설계 (패턴 적용)
**타입**: rag + prompt

`references/patterns.md`의 패턴 1~8을 **각 노드·엣지마다 순회 적용**한다.

- 패턴별 탐지 조건에 걸리면 보강안을 만들고, 근거(패턴 번호)를 기록한다.
- 해당 안 되면 "해당없음"으로 체크리스트에 남긴다.
- 불확실하면 보강 대신 **확인요망**으로 올린다.
- **과잉 보강 금지**: 챗봇 맥락에 안 맞는 분기는 넣지 않는다(patterns.md 하단 경계 참조).

### Step 4: 산출물 생성
**타입**: generate

`references/output-templates.md`의 8개 섹션으로 `검토리포트.md`를 만든다:
1. 보강 mermaid 흐름도 (원본=실선 / 보강=점선+[추가] / 추정=회색)
2. 분류 색상표
3. **원본/보강/추정 구분표** ★
4. **패턴 적용 체크리스트** ★
5. **리스크 우선순위(🔴치명/🟡중요/🟢권장)** ★
6. **Before/After 노드 비교** ★
7. 원본 대비 변경 이력
8. 고객 확인요망 (별도 분리)

최상단에 고지문을 자동 삽입한다. 파일명은 `{흐름명}_검토리포트_YYYYMMDD.md`.

이 전체 MD는 **기획자(내부)용 마스터**다.

### Step 5: (옵션) 고객 요약본 + 워드 전달본 변환
**타입**: generate + script

사용자가 "고객 전달본/워드로 만들어줘"라고 할 때 실행한다. **워드는 8섹션 전체가 아니라
가독성을 위해 고객 요약본(4섹션)으로 만드는 것이 기본**이다. (output-templates.md "고객 요약본" 참조)

1. 마스터 MD에서 **고객 요약본 MD**(4섹션: 보강 흐름도 / 주요 변경 Top5 / 리스크(치명·중요) / 확인요망)를
   추려 `{흐름명}_고객요약본_YYYYMMDD.md`로 만든다. 신뢰 중시 고객이면 "원본/보강/추정 구분"을 더해 5섹션.
2. 그 요약본 MD를 워드로 변환한다.

```
cd skills/flowchart-defense
PYTHONUTF8=1 python scripts/render_docx.py <고객요약본.md> <출력.docx>
```
Windows PowerShell에서는 `$env:PYTHONUTF8=1; python scripts/render_docx.py ...`. 사전: `pip install python-docx`.
(고객이 전체 상세본을 원하면 마스터 MD를 그대로 변환해도 된다.)

- 스크립트가 표를 워드 표로 변환하고 **모든 글자에 한글 폰트(맑은 고딕)를 지정**한다(한글 깨짐 방지).
- mermaid를 PNG로 렌더(`mermaid-cli` 필요)해 워드에 삽입한다.
- `mermaid-cli`(mmdc)가 없으면 설치 안내만 하고 **MD는 그대로 유지**한다(작업 손실 0).
  렌더 실패 시 워드에는 흐름도를 mermaid 코드 블록 + "흐름도는 MD에서 확인" 안내로 대체한다.

## 워크플로우 (배치 모드)

여러 파일/폴더를 받으면:

1. **목록 확인** — 발견한 흐름도 파일 N개를 사용자에게 보여주고, 5~8개씩 묶음으로 나눈다.
   N이 8을 넘으면 "8개씩 나눠 처리할게요"라고 안내하고 첫 묶음부터 진행한다.
   > 한 대화에 30~40개를 한꺼번에 넣으면 컨텍스트 한계로 뒤쪽 품질이 떨어지고 원본 확정이
   > 무력화된다. 그래서 **묶음(≤8)으로 나눠 순차 처리**한다.
2. **묶음별 처리** — 각 파일을 Step 1·3·4로 처리해 **1개당 리포트 1개**를 만든다.
   배치에서는 Step 2(원본 확정)를 **저신뢰 추출만 플래그**하는 방식으로 완화한다:
   추출 신뢰도가 낮은 흐름도만 모아 "이건 사람 확인이 필요해요"로 표시한다(나머지는 자동 진행).
3. **묶음 요약** — 묶음이 끝나면 "8개 중 6개 완료, 2개 확인 필요"처럼 요약하고 다음 묶음으로.
4. **확인요망 취합** — 배치에서는 각 리포트의 확인요망을 `고객확인요망_취합.md`로 모아도 좋다.

배치에서도 Iron Law(원본/보강/추정 분리, 근거 부착, 모르면 추정 금지)는 그대로 적용한다.

## 엣지 케이스 대응

- **PPT 텍스트가 이미지로 박힘 / 화살표가 도형에 안 붙음**: 위치로 임의 연결하지 말고
  비전 판독 + "흐릿한 부분"에 명시. 신뢰도 미달이면 원본 재요청.
- **멀티 플로우가 한 파일에**: 자동 분리하지 말고 "N개 흐름으로 보여요. 나눌까요?" 먼저 확인.
- **입력이 흐름도가 아님**(와이어프레임·캡처·표): 강제 변환 금지. "흐름도로 안 보여요" 재질의.
- **한글 깨짐·OCR 손상**: 깨진 비율이 높으면 OCR 신뢰 안 하고 "원본 PPT나 텍스트로 다시" 요청.
- **흐름이 너무 단순(노드 2~3개)**: "보강 여지 적음" 명시. 억지 보강 금지.
- **흐름이 과복잡(노드 다수)**: 서브플로우 구간으로 쪼개 처리.
- **순환/종료 없는 루프**: 패턴 8로 종착 처리하되 "원본에 종료가 없었다"를 변경이력에 명시.
- **렌더 실패(mmdc 없음)**: 설치 안내 후 MD 유지. 워드는 코드블록 대체.

## 실패 시 복구

단계별로 중간 결과를 남긴다(판독 → 인벤토리 → 보강 → 리포트). 보강 단계에서 실패해도
"읽은 원본 인벤토리"는 사용자가 확보한다. 빈 부분을 지어내 메우지 않는다. 신뢰도 미달이면
**잘못된 산출물을 만드느니 멈추고**, "왜 멈췄는지 + 다음에 뭘 주면 되는지"를 한국어로 안내한다.
