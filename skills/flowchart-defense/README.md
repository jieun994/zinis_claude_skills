# 흐름도 검토·방어설계 스킬 (flowchart-defense)

## 1. 무엇을 하나요

- 고객이 보낸 **챗봇/서비스 흐름도(PPT·이미지·PDF·mermaid)** 를 읽어 빠진 부분을 자동 보강
- 고객 흐름도는 보통 "정상 흐름"만 그려져 있음
- 빠지기 쉬운 부분을 **패턴 1~8**로 전수 점검
  - 오류가 났을 때 / 결과가 0건일 때 / 사용자가 취소·수정할 때 / 화살표가 끊겨 막다른 길이 될 때
- 고객에게 그대로 전달 가능한 **검토 리포트** 생성

가장 신경 쓴 점: **원본에 없는 걸 지어내지 않음.**
- 원본/보강/추정을 색과 표로 구분
- 보강 전에 "원본을 이렇게 읽었어요" 확인 절차 필수

---

## 2. 어떻게 쓰나요

- 흐름도 파일을 첨부하고 아래처럼 요청

- "이 챗봇 흐름도 검토해줘"
- "플로우차트 방어설계 해줘"
- "이 흐름에서 빠진 예외 찾아줘"
- "막다른 길이랑 빈 결과 분기 체크해줘"

진행 순서:

1. 흐름도를 읽고 **"원본을 이렇게 읽었어요, 맞나요?"** 확인 (잘못 읽었으면 여기서 교정)
2. 패턴 1~8로 빠진 부분 보강
3. **검토 리포트(.md)** 생성 — 보강 흐름도 + 표 8종 (상세 마스터)
4. (선택) **"고객 전달본 워드로 만들어줘"** → 고객용 **요약본 4섹션**을 `.docx`로 변환
   - 4섹션 = ① 보강 흐름도 ② 주요 변경 요약(Top 5) ③ 꼭 고쳐야 할 리스크 ④ 고객 확인요망
   - 전체 8섹션 상세본은 MD에 그대로 유지 (워드는 읽기 좋게 슬림)

### 여러 개를 한 번에 (배치)
- 흐름도가 많으면(예: 30개, 108개) 폴더째 주거나 여러 개 첨부
- **한 번에 5~8개씩 자동 분할** → 1개당 리포트 1개 생성
- 30~40개를 한 대화에 한꺼번에 넣으면 품질 저하 → 스킬이 묶음으로 나눠 처리하니 묶음마다 결과 확인하며 진행

## 3. 예시

- **입력**: 고객이 준 `배송일정변경_흐름도.pptx` (정상 흐름만 있음)
- **Claude에게**: "이 챗봇 흐름도 검토하고 방어설계 해줘"
- **결과**: `배송일정변경_검토리포트.md`

| 리포트 구성 | 내용 |
|---|---|
| ① 보강 흐름도 | 원본(실선) + 보강(점선 `[추가]`) + 추정(회색) 구분 mermaid |
| ② 분류 색상표 | 사용자/UI/API/판단/시스템 5색 |
| ③ 원본/보강/추정 구분표 | 원본과 AI 보강을 한눈에 구분 |
| ④ 패턴 적용 체크리스트 | 패턴 1~8 전수 점검 증거 |
| ⑤ 리스크 우선순위 | 치명 / 중요 / 권장 |
| ⑥ Before/After 비교 | 변경 전후 노드 대조 |
| ⑦ 변경 이력 | 무엇을 왜 바꿨는지 (근거=패턴번호) |
| ⑧ 고객 확인요망 | 고객에게 되물어야 할 미확정 항목 (별도 분리) |

실제 결과물: [`samples/배송일정변경_검토리포트_예시.md`](samples/배송일정변경_검토리포트_예시.md)

## 4. 참고

- **워드(.docx) 변환은 선택** — 안 해도 MD 리포트는 항상 생성
- 워드에 흐름도를 이미지로 넣으려면 `mermaid-cli` 설치 필요 (없으면 흐름도를 코드로 넣고 안내)
- 이 스킬은 흐름도를 새로 그리는 게 아니라, **받은 흐름도를 검토·보강**하는 용도
