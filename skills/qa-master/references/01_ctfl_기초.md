# ISTQB CTFL 기반 테스트케이스 작성·QA 실무 지식

- 출처: ISTQB CTFL(파운데이션 레벨) 실러버스 v4.0.1 내용을 실무 관점으로 재정리한 문서
- 용도: 비개발 기획자가 요구사항·기획서를 바탕으로 테스트케이스/시나리오를 만들거나 QA할 때 적용할 규칙 모음
- 시험 대비 요소(학습목표 번호, 인지수준 등)는 제외하고 실무 적용 지식만 수록

## 1. 테스팅의 기본 개념

### 1.1 테스팅이란
- 결함을 찾아내고 산출물의 품질을 평가하는 활동 전체를 의미. 프로그램 실행(테스트 실행)만이 아니라 계획·분석·설계·구현·완료까지 포함
- 두 가지 확인 축이 모두 필요
  - 베리피케이션: "요구사항대로 만들었는가" (명세 대비 일치 확인)
  - 밸리데이션: "사용자에게 실제로 필요한 것을 만들었는가" (기대·요구 충족 확인)
- 실행 없이 문서·코드를 검토하는 정적 테스팅과, 실행해서 확인하는 동적 테스팅으로 구분
- 테스팅과 디버깅은 별개: 테스팅은 문제를 드러내는 활동, 디버깅은 원인을 찾아 고치는 활동. 수정 후에는 확인 테스팅(고쳐졌는가)과 리그레션 테스팅(다른 곳이 망가지지 않았는가)이 뒤따름

### 1.2 오류 → 결함 → 장애의 구분 (결함 리포트 소통에 필수)
| 용어 | 의미 | 예시 |
|---|---|---|
| 오류(error) | 사람이 저지른 실수 | 기획자가 할인율 조건을 잘못 씀 |
| 결함(defect) | 실수의 결과로 산출물에 남은 결점 | 기획서·코드에 잘못된 할인 로직이 들어감 |
| 장애(failure) | 결함이 실행되어 겉으로 드러난 오동작 | 결제 화면에서 할인가가 틀리게 표시됨 |
- 결함이 있어도 특정 조건에서만 장애로 드러나는 경우가 많음 → "재현 조건"의 기록이 중요한 이유
- 근본 원인(root cause): 문제를 만들어낸 근본적 이유. 장애 하나를 고치는 데서 끝내지 말고 근본 원인을 다루면 유사 결함의 재발을 예방 가능
- 결함은 코드에만 있지 않음. 요구사항 문서·기획서·테스트케이스 자체에도 결함이 존재 → 기획서 QA가 성립하는 근거

### 1.3 테스팅과 QA의 차이
- 테스팅(품질 제어): 제품 지향. 만들어진 것의 품질을 확인하고 교정하는 활동
- 품질 보증(QA): 프로세스 지향. 좋은 프로세스를 지키면 좋은 제품이 나온다는 전제의 예방 활동. 프로젝트 참여자 전원의 책임
- 실무 함의: "QA한다"고 할 때 테스트 실행만 떠올리지 말 것. 작업 방식 자체의 개선(리뷰 절차, 산출물 기준)도 QA의 영역

## 2. 테스팅 7원칙 — 실무에서의 의미

| 원칙 | 실무 적용 1줄 |
|---|---|
| 1. 테스팅은 결함의 존재를 보여줄 뿐, 없음을 증명하지 못함 | "테스트 다 통과했으니 버그 없음"이라고 보고하지 말 것. 잔존 결함 가능성을 항상 전제 |
| 2. 완벽한 테스팅은 불가능 | 모든 조합을 다 돌리려 하지 말고, 테스트 기법·우선순위·리스크 기반으로 어디에 힘을 쏟을지 선택 |
| 3. 조기 테스팅이 시간과 비용을 절약 | 코드가 나오기 전, 기획서·요구사항 단계에서 리뷰로 결함을 잡는 것이 가장 저렴함 |
| 4. 결함은 집중된다 | 결함이 몰리는 소수 영역(복잡한 기능, 잦은 변경 영역)에 테스트를 더 배치. 과거 결함 이력이 좋은 힌트 |
| 5. 같은 테스트의 반복은 효과가 감소 | 기존 케이스만 돌리면 새 결함을 못 찾음. 케이스와 데이터를 주기적으로 갱신·추가 (단, 자동 리그레션의 반복은 예외적으로 유익) |
| 6. 테스팅은 정황(컨텍스트)에 의존 | 프로젝트마다 접근을 다르게. 결제 시스템과 이벤트 페이지에 같은 강도의 테스트를 적용하지 않음 |
| 7. 결함 부재는 궤변 | 명세대로 다 통과해도 사용자가 원하는 제품이 아닐 수 있음. 명세 검증과 별개로 실제 사용 관점 확인(밸리데이션) 병행 |

## 3. 테스트 프로세스 — 활동별 실무 체크

테스트는 아래 활동으로 구성. 순서대로 보이지만 실제로는 반복·병행됨.

### 3.1 테스트 계획
- 테스트 목적을 정의하고, 제약(기간·인력·환경) 안에서 달성 방법을 결정
- 계획서에 담을 것: 범위·목적·테스트 베이시스(근거 문서), 가정과 제약, 역할과 책임, 리스크 목록, 접근법(레벨·유형·기법·시작/완료 조건), 일정과 예산
- 시작 조건(entry criteria): 활동을 시작하기 위한 전제. 예: 테스트 가능한 요구사항 존재, 환경·데이터 준비 완료, 스모크 테스트 통과
- 완료 조건(exit criteria): 활동 종료를 선언하는 기준. 예: 목표 커버리지 달성, 미해결 중대 결함 0건, 계획한 케이스 전부 실행
- 시간·예산 소진도 유효한 완료 조건이 될 수 있으나, 이때는 잔여 리스크를 이해관계자가 검토·수용해야 함

### 3.2 테스트 분석 — "무엇을 테스트할 것인가"
- 테스트 베이시스(요구사항 정의서, 기획서, 화면설계서 등)를 분석해 테스트할 대상(테스트 컨디션)을 식별
- 이때 리스크 수준을 고려해 컨디션에 우선순위를 부여
- 베이시스 자체의 결함(모호함, 누락, 모순)도 이 단계에서 발견 → 발견 즉시 기획서 수정 요청 또는 결함으로 기록

### 3.3 테스트 설계 — "어떻게 테스트할 것인가"
- 테스트 컨디션을 구체적인 테스트케이스로 상세화. 테스트 기법(4장) 활용
- 케이스 입력값을 정하는 커버리지 항목 식별, 테스트 데이터 요구사항 정의, 테스트 환경·도구 식별 포함

### 3.4 테스트 구현
- 실행에 필요한 것 준비: 테스트 데이터 생성, 케이스를 절차(procedure)로 묶기, 테스트 스위트 조립, 우선순위 반영한 실행 일정 수립, 환경 구축·점검

### 3.5 테스트 실행
- 일정에 따라 수행하고, 실제 결과를 기대 결과와 비교. 결과는 반드시 기록
- 기대와 다른 이상 사항은 원인을 분석한 뒤 결함 보고서로 보고 (이상 사항이 전부 결함은 아님 — 테스트 오류, 환경 문제, 변경 요청일 수도 있음)

### 3.6 테스트 완료
- 릴리스·반복 주기 종료 등 마일스톤에서 수행
- 미해결 결함은 변경 요청 또는 백로그로 전환, 재사용할 테스트웨어 보관·인계, 교훈 정리, 완료 보고서 작성·공유

### 3.7 추적성(traceability) 유지
- 요구사항 ↔ 테스트 컨디션 ↔ 테스트케이스 ↔ 결과 ↔ 결함이 서로 연결되게 ID로 관리
- 효과: 요구사항별 테스트 커버 여부 확인, 변경 발생 시 영향받는 케이스 즉시 식별, 진행 보고의 근거 확보
- 실무 규칙: 테스트케이스에 반드시 근거 요구사항 ID(또는 기획서 화면/기능 ID)를 기입

## 4. 테스트 레벨과 테스트 유형 — 언제 무엇을 쓰나

### 4.1 테스트 레벨 (개발 단계별 구분)
| 레벨 | 대상 | 언제/누가 | 기획자 관점 |
|---|---|---|---|
| 컴포넌트(단위) 테스팅 | 개별 모듈·함수 | 개발자가 개발 환경에서 | 직접 관여 적음. 개발 일정에 포함되어 있는지 확인 |
| 컴포넌트 통합 테스팅 | 모듈 간 인터페이스·상호작용 | 개발 단계, 통합 전략에 따라 | 모듈 간 데이터 전달 규칙이 기획서에 명확한지 확인 |
| 시스템 테스팅 | 전체 시스템의 엔드투엔드 동작 | 독립 테스트팀, 실제 유사 환경 | 기획서 기반 기능·비기능 케이스의 주 무대 |
| 시스템 통합 테스팅 | 타 시스템·외부 서비스와의 연동 | 운영 유사 환경 | 외부 API 연동(PG, 배송사, 알림 등) 확인 구간 |
| 인수 테스팅 | 비즈니스 요구 충족·배포 준비 | 실제 사용자·현업이 수행 이상적 | UAT 시나리오 작성과 진행이 기획자의 핵심 역할 |
- 인수 테스팅의 하위 유형: 사용자 인수(UAT), 운영 인수, 계약/규제 인수, 알파/베타 테스트
- 레벨마다 테스트 대상·목적·베이시스·담당을 다르게 정의해 중복을 방지

### 4.2 테스트 유형 (관점별 구분)
| 유형 | 확인하는 것 | 사용 시점 |
|---|---|---|
| 기능 테스팅 | 시스템이 "무엇을" 하는가 — 기능의 완전성·정확성·적합성 | 모든 레벨. 기획서의 기능 명세가 베이시스 |
| 비기능 테스팅 | "얼마나 잘" 동작하는가 — 성능, 호환성, 사용성, 신뢰성, 보안, 유지보수성, 이식성, 안전성(ISO 25010) | 늦게 발견하면 치명적이므로 가능한 조기 시작. 기능 테스트에 제약 조건(응답시간 등)을 얹어 도출 가능 |
| 블랙박스 테스팅 | 명세 대비 동작 (내부 구조 무관) | 기획자가 작성하는 케이스의 기본형. 구현이 바뀌어도 케이스 유효 |
| 화이트박스 테스팅 | 내부 구조(코드·아키텍처) 커버 | 개발자 영역. 명세가 부실해도 결함을 찾을 수 있으나, "만들지 않은 기능"의 누락은 못 찾음 |

### 4.3 확인 테스팅과 리그레션 테스팅 (변경이 있을 때마다)
- 확인 테스팅: 수정된 결함이 실제로 고쳐졌는지 확인. 원래 실패했던 케이스를 재실행하거나, 수정 사항 확인용 새 테스트를 추가. 가능하면 처음 발견한 사람이 재확인
- 리그레션 테스팅: 변경이 다른 곳(같은 컴포넌트, 다른 컴포넌트, 연동 시스템, 환경)을 망가뜨리지 않았는지 확인
  - 범위 판단을 위해 먼저 영향도 분석 수행
  - 반복 실행되고 케이스가 계속 늘어나므로 자동화에 가장 적합한 대상
- 규칙: 어떤 레벨에서든 수정·변경이 있으면 확인 테스팅 + 리그레션 테스팅 세트로 수행

### 4.4 유지보수 테스팅
- 운영 중 시스템의 변경(핫픽스, 정기 개선, 환경 업그레이드·마이그레이션, 서비스 단종) 시 수행
- 범위 결정 요인: 변경의 리스크 수준, 기존 시스템 크기, 변경 크기
- 마이그레이션 시 데이터 변환 테스트, 단종 시 데이터 보관·복원 테스트까지 고려

## 4-1. 개발 방식과 테스팅의 연결 (시프트 레프트·회고·자동화)

### 시프트 레프트 (조기 테스팅의 실천법)
- 테스트 활동을 개발 후반이 아닌 앞 단계로 당기는 접근. 대표 프랙티스:
  - 테스터(QA) 관점에서 명세·기획서를 리뷰 — 모호성·불완전성·불일치가 이 단계에서 가장 많이 발견됨
  - 코드 작성 전에 테스트케이스를 먼저 작성 (테스트 우선 접근: TDD/ATDD/BDD)
  - 비기능 테스트도 가능한 한 앞 단계부터 시작 (후반에 몰면 발견 시 치명적)
- 초기에 교육·공수가 더 들지만 후반 수정 비용 절감으로 상쇄. 이해관계자의 이해와 동의가 전제 조건

### 테스트 우선 접근법 3가지 (요구사항을 테스트 형태로 먼저 쓰는 방법)
- TDD: 테스트 코드를 먼저 작성하고 통과하도록 구현 (개발자 영역)
- ATDD: 인수 조건에서 테스트를 도출한 뒤 구현 시작 — 기획자가 참여하는 명세 워크숍이 출발점 (6.9 참조)
- BDD: 기대 동작을 Given/When/Then 자연어로 작성해 이해관계자 모두가 읽을 수 있는 테스트로 표현, 이후 자동 실행 가능한 테스트로 전환

### 모든 개발 방식에 통하는 좋은 테스팅 프랙티스
- 모든 개발 활동에 대응하는 테스트 활동을 둔다 (기획→기획 리뷰, 개발→단위 테스트, 통합→통합 테스트)
- 테스트 레벨마다 목적을 따로 정의해 중복을 피하고 빠짐없이 커버
- 각 레벨의 테스트 분석·설계는 대응하는 개발 단계에서 바로 시작
- 문서 초안이 나오는 즉시 테스터가 리뷰에 참여

### 회고 (프로세스 개선)
- 반복 주기·릴리스 종료 시 진행: 잘된 것/부족한 것/개선 방법을 논의하고 기록
- 효과: 테스트 효율 향상, 테스트웨어·테스트 베이시스 품질 개선, 팀 협업 개선
- 핵심: 결과를 기록하고 개선 항목에 후속 조치가 실제로 이루어져야 의미가 있음

### 테스트 자동화의 효과와 리스크 (도입 판단 기준)
- 효과: 반복 수작업(리그레션, 데이터 입력, 결과 비교) 절감, 사람 실수 방지와 재현성 확보, 객관적 측정(커버리지), 빠른 피드백, 테스터가 심층 테스트를 설계할 시간 확보
- 리스크: 도구 효과에 대한 비현실적 기대, 도입·유지보수 비용 과소 추정, 수동이 더 적합한 곳에 자동화 적용, 도구 과신으로 비판적 사고 소홀, 공급업체 종속
- 판단 기준: 반복성이 높고 판단이 단순한 테스트(리그레션, 스모크)부터 자동화. 사용자 관점 평가(사용성, 탐색적)는 수동 유지

## 5. 정적 테스팅 — 기획서 QA의 핵심

### 5.1 왜 중요한가
- 실행 없이 문서를 검토하는 활동이므로 코드가 없어도 가능 → 기획 단계에서 결함을 잡는 유일한 수단
- 검토 대상: 요구사항 정의서, 기획서, 화면설계서, 계약서, 테스트케이스 자체, 백로그 항목 등 읽을 수 있는 거의 모든 산출물
- 동적 테스팅으로는 못 찾는 결함을 발견: 요구사항의 모호성·모순·누락, 설계 결함, 인수 조건 하나에 대한 테스트 누락 등
- 후반에 결함을 고치는 비용보다 리뷰 비용이 훨씬 저렴 (조기 테스팅 원칙의 실천)

### 5.2 정적 테스팅으로 잘 잡히는 기획서 결함 유형 (리뷰 체크 관점)
- 요구사항 결함: 문서 내 불일치, 모호한 표현, 상호 모순, 항목 누락, 부정확, 중복
- 인터페이스 명세 오류: 주고받는 값의 개수·유형·순서 불일치
- 커버리지 공백: 인수 조건은 있는데 대응 테스트가 없는 경우

### 5.3 리뷰 프로세스 5단계
1. 계획: 리뷰 목적, 대상 산출물, 집중 영역, 완료 조건, 일정 정의
2. 리뷰 착수: 참여자 전원이 산출물 접근 가능한지, 역할을 이해했는지 확인
3. 개별 리뷰: 각자 체크리스트 기반·시나리오 기반 등 기법으로 검토하고 이상 사항·의문 사항 기록
4. 논의 및 분석: 리뷰 회의에서 이상 사항별 상태·담당·조치를 결정 (이상 사항이 전부 결함은 아니므로 논의 필요)
5. 수정 및 보고: 결함 보고서 작성으로 후속 조치를 추적 가능하게 하고, 완료 조건 충족 시 산출물 승인

### 5.4 리뷰 역할
- 관리자(리뷰 대상·자원 결정), 저자(작성·수정), 중재자(회의 진행, 안전한 발언 환경), 서기(이상 사항 기록), 리뷰어(검토 수행), 리뷰 리더(참여자·일정 결정)
- 한 사람이 여러 역할 겸임 가능. 단, 가장 공식적인 리뷰(인스펙션)에서는 저자가 리더·서기를 맡을 수 없음

### 5.5 리뷰 유형 — 상황별 선택
| 유형 | 공식성 | 주 목적 | 실무 사용 예 |
|---|---|---|---|
| 비공식 리뷰 | 낮음 | 이상 사항 빠른 발견 | 동료에게 기획서 초안 훑어봐 달라고 요청 |
| 워크쓰루 | 중간, 저자가 진행 | 이해 공유·합의·아이디어 도출 | 기획자가 화면설계서를 팀에 설명하며 의견 수렴 |
| 기술 리뷰 | 중간~높음, 중재자 진행, 전문가 참여 | 기술 쟁점 합의·결정 | 연동 방식·데이터 구조에 대한 개발 리뷰 |
| 인스펙션 | 가장 높음, 정해진 프로세스 준수 | 이상 사항 최대한 많이 발견, 메트릭 수집 | 규제·안전 관련 산출물의 공식 검수 |
- 같은 산출물을 처음엔 비공식으로, 나중에 공식적으로 여러 번 리뷰 가능

### 5.6 리뷰 성공 요소
- 명확한 목표와 측정 가능한 완료 조건 정의 — 참가자 개인 평가가 목적이 되면 절대 안 됨
- 작은 단위로 나눠 리뷰 (집중력 유지)
- 준비 시간 충분히 제공, 결과 피드백은 저자와 이해관계자에게 전달
- 관리층의 지원과 조직 문화로의 정착, 참여자 교육, 회의 퍼실리테이션

## 6. 테스트 설계 기법 — 케이스 도출 절차와 예시

기법은 3개 계열: 블랙박스(명세 기반), 화이트박스(구조 기반), 경험 기반. 기획자는 블랙박스 + 경험 기반이 주력이며, 서로 보완 관계이므로 한 기능에 여러 기법을 겹쳐 사용.

### 6.1 동등 분할 (Equivalence Partitioning)
- (a) 적용 조건
  - 입력·출력·설정값이 "같은 방식으로 처리되는 값의 묶음"으로 나뉠 때
  - 근거: 같은 분할 안의 값 하나로 결함을 찾을 수 있다면 그 분할의 다른 값도 같은 결함을 찾음 → 분할당 대표값 1개면 충분
- (b) 케이스 도출 절차
  1. 대상 데이터 항목 식별 (입력 필드, 출력, 상태값 등)
  2. 유효 분할(정상 처리되는 값 묶음)과 비유효 분할(거부·무시되어야 하는 값 묶음)을 정의 — 분할끼리 겹치지 않고, 빈 분할이 없어야 함
  3. 각 분할에서 대표값 1개씩 선택해 케이스 작성 (비유효 분할 포함 모든 분할을 최소 1회 커버해야 100%)
  4. 입력 항목이 여러 개면, 각 항목의 각 분할이 최소 1회씩 등장하도록 케이스를 조합 (이치 초이스 커버리지 — 조합 전부를 다 만들 필요는 없음)
- (c) 예시 — 회원가입 비밀번호 입력 (규칙: 8~20자 허용)
  - 유효 분할: 8~20자 문자열 → 대표값 "abcd1234" 1건
  - 비유효 분할 1: 8자 미만 → "abc1" 1건 (오류 메시지 확인)
  - 비유효 분할 2: 20자 초과 → 21자 문자열 1건
  - 비유효 분할 3: 빈값 → "" 1건
  - 총 4건으로 이 규칙의 분할 커버리지 100%
- 보조 예시 — 입력 항목이 2개인 경우 (배송조회: 주문번호 형식 유효/무효 × 조회 권한 있음/없음)
  - 조합 전부(4건)가 아니라, 각 항목의 각 분할이 1회씩 등장하는 2건으로도 이치 초이스 커버리지 충족 가능 (유효+권한있음, 무효+권한없음)
  - 단, 조합 자체가 리스크인 기능(권한×상태)이라면 결정 테이블로 전환해 조합을 명시적으로 다룸

### 6.2 경계값 분석 (Boundary Value Analysis)
- (a) 적용 조건
  - 동등 분할이 순서 있는 범위(숫자, 날짜, 길이, 금액 등)일 때 — 경계에서 개발 실수(초과/미만/이상/이하 혼동, 경계 누락)가 가장 잦으므로 분할의 양 끝값을 집중 공략
- (b) 케이스 도출 절차
  1. 동등 분할을 먼저 정의하고 각 분할의 최솟값·최댓값(경계값) 식별
  2. 2-value 방식: 경계값 + 인접 분할의 가장 가까운 값을 테스트 (기본형)
  3. 3-value 방식: 경계값과 그 양옆 값을 모두 테스트 (더 엄격 — "이상(≥)"을 "같음(=)"으로 잘못 구현한 결함은 3-value여야 잡히는 경우가 있음. 결제·정산 등 중요 로직에 적용)
- (c) 예시 — 주문 시 "3만 원 이상 무료배송"
  - 분할: 배송비 부과(0~29,999원) / 무료(30,000원~)
  - 2-value 케이스: 29,999원(배송비 있어야 함), 30,000원(무료여야 함)
  - 3-value로 강화 시: 29,999 / 30,000 / 30,001원 모두 확인
  - 함께 확인: 0원, 최대 주문 금액 등 다른 경계

### 6.3 결정 테이블 테스팅 (Decision Table Testing)
- (a) 적용 조건
  - 여러 조건의 조합에 따라 결과가 달라지는 비즈니스 규칙이 있을 때 (할인 정책, 권한, 노출 조건 등)
  - 강점: 사람이 놓치기 쉬운 조합까지 체계적으로 나열되어 기획서의 규칙 누락·모순이 드러남 → 케이스 도출과 기획서 검증을 동시에 수행
- (b) 케이스 도출 절차
  1. 조건들과 결과 동작들을 나열 (표의 행)
  2. 조건 조합별로 열을 만들고 각 열에 결과를 기입 (T=충족, F=불충족, –=무관, N/A=불가능)
  3. 실현 불가능한 조합 삭제, 결과가 같고 일부 조건이 무관한 열은 병합해 단순화
  4. 남은 각 열(실행 가능한 규칙)마다 테스트케이스 1건 작성 — 전 열 커버 시 100%
  5. 조건이 많아 조합이 폭증하면 테이블 최소화 또는 리스크 기반으로 중요 규칙만 선별
  6. 채우다 결과가 정의 안 된 열을 발견하면 즉시 기획 확인 (누락된 규칙)
- (c) 예시 — 쿠폰 적용 (조건: 회원 여부, 최소 주문금액 충족, 쿠폰 유효기간 내)
  | 조건 | R1 | R2 | R3 | R4 |
  |---|---|---|---|---|
  | 회원인가 | T | T | T | F |
  | 최소 주문금액 충족 | T | T | F | – |
  | 유효기간 내 | T | F | – | – |
  | 결과: 할인 적용 | X | | | |
  | 결과: 오류 메시지 | | 기간만료 | 금액미달 | 로그인유도 |
  - 4개 규칙 → 4개 케이스. 비회원+금액미달 같은 조합의 메시지 우선순위가 기획서에 없다면 그 자체가 발견된 결함

### 6.4 상태 전이 테스팅 (State Transition Testing)
- (a) 적용 조건
  - 시스템·데이터가 상태를 가지며 이벤트에 따라 상태가 바뀔 때 (주문 상태, 회원 등급, 게시물 승인 흐름, 화면 전환)
  - 같은 이벤트라도 현재 상태에 따라 결과가 달라지는 기능에 필수
- (b) 케이스 도출 절차
  1. 상태 목록과 이벤트 목록 정의
  2. 상태 다이어그램 또는 상태 테이블(행=상태, 열=이벤트, 셀=전이 결과) 작성 — 테이블 방식은 비유효 전이(빈 셀)가 명확히 드러나 검증에 유리
  3. 커버리지 수준 선택 후 이벤트 순서(경로)로 케이스 작성
     - 모든 상태 커버리지: 모든 상태를 1회 이상 방문 (가장 약함)
     - 유효 전이 커버리지: 모든 유효 전이를 1회 이상 실행 (가장 널리 쓰는 기준, 달성 시 상태 커버리지도 자동 충족)
     - 모든 전이 커버리지: 유효 전이 실행 + 비유효 전이 시도까지 포함 (미션 크리티컬 최소 기준. 비유효 전이는 케이스당 1개만 시도해 결함이 서로 가려지는 것을 방지)
- (c) 예시 — 배송 조회 상태 (결제완료 → 상품준비중 → 배송중 → 배송완료, 결제완료·상품준비중에서만 취소 가능)
  - 유효 전이 케이스: 정상 흐름 4단계 진행 1건, 결제완료→취소 1건, 상품준비중→취소 1건
  - 비유효 전이 케이스: 배송중 상태에서 취소 시도 → 거부 및 안내 확인, 배송완료에서 취소 시도 → 거부 확인
  - 기획서에 "배송중 취소 시도 시 처리"가 없다면 상태 테이블의 빈 셀로 즉시 드러남

### 6.5 화이트박스 기법 (구문/분기 커버리지) — 개념만
- 구문 커버리지: 코드의 실행 가능한 문장을 모두 1회 이상 실행. 분기 커버리지: 모든 조건 분기(참/거짓)를 실행. 분기 100%면 구문 100%가 자동 충족(역은 성립 안 함)
- 기획자 활용: 개발팀의 단위 테스트 수준을 물을 때 "커버리지 몇 %인가"의 의미를 이해하는 용도. 커버리지 100%여도 결함 없음을 뜻하지 않음(데이터 의존 결함 등)

### 6.6 오류 추정 (Error Guessing)
- (a) 적용 조건
  - 과거 결함 이력, 유사 서비스 경험, "개발자가 어디서 실수하는지"에 대한 지식이 있을 때. 공식 기법의 보완용
- (b) 케이스 도출 절차
  1. 발생 가능한 오류·결함·장애 목록을 작성 (경험, 과거 결함 데이터, 일반 지식 기반)
  2. 목록의 각 항목을 노출시킬 테스트를 설계 (이 방식을 결함 공격이라 부름)
  3. 점검 관점: 입력(잘못된 값 미인식, 파라미터 누락), 출력(형식 오류, 잘못된 결과), 논리(케이스 누락), 계산 오류, 인터페이스 불일치, 데이터(초기화·타입 오류)
- (c) 예시 — 로그인 기능에 대한 결함 공격 목록
  - 이메일 앞뒤 공백 입력, 대소문자 섞인 이메일, 비밀번호 연속 5회 오류 후 처리, 로그인 상태에서 뒤로가기, 세션 만료 후 버튼 클릭, 동시에 두 기기 로그인 → 각각을 확인하는 케이스 작성

### 6.7 탐색적 테스팅 (Exploratory Testing)
- (a) 적용 조건
  - 명세가 부족·부적합하거나 시간 압박이 심할 때, 공식 기법 케이스를 보완하고 싶을 때. 테스터의 도메인 지식·분석력이 좋을수록 효과적
- (b) 진행 절차 (세션 기반)
  1. 테스트 차터 작성: 이번 세션의 목적·탐색 범위를 한두 줄로 정의 (예: "주문 취소와 환불 흐름에서 금액 표시 이상 탐색")
  2. 시간을 정해 놓고(타임박스) 대상을 배워가며 설계·실행·평가를 동시에 수행
  3. 세션 시트에 수행 절차와 발견 사항 기록
  4. 세션 종료 후 이해관계자와 결과 공유(디브리핑)
  - 세션 중 동등 분할 등 다른 기법을 즉석에서 활용 가능
- (c) 예시 — 신규 오픈한 이벤트 페이지를 "응모 조건 경계, 중복 응모, 새로고침·뒤로가기, 모바일 회전" 관점 차터로 60분 탐색하고 발견 이슈를 기록·보고

### 6.8 체크리스트 기반 테스팅 (Checklist-Based Testing)
- (a) 적용 조건
  - 상세 케이스를 만들 시간이 없거나, 반복적으로 확인할 공통 품질 항목(UI 공통 규칙, 사용성, 보안 기본)이 있을 때. 일관성은 확보되나 재현성은 낮음
- (b) 진행 절차
  1. 경험·사용자 중요도·과거 실패 원인을 바탕으로 확인 항목 목록 작성 — 각 항목은 개별적이고 직접 확인 가능해야 하며, 질문형이 좋음
  2. 제외할 것: 자동으로 점검 가능한 항목, 시작/완료 조건에 더 적합한 항목, 너무 일반적인 항목
  3. 각 항목을 확인하는 테스트를 설계·실행
  4. 결함 분석 결과로 정기 업데이트 — 효과 떨어진 항목 제거, 새 중대 결함 기반 항목 추가, 너무 길어지지 않게 유지
- (c) 예시 — 웹서비스 화면 공통 체크리스트
  - 필수 입력 누락 시 오류 메시지가 필드 옆에 표시되는가 / 저장 중 버튼 중복 클릭이 방지되는가 / 목록 0건일 때 빈 상태 안내가 있는가 / 뒤로가기 시 입력값이 보존되는가 / 권한 없는 URL 직접 접근이 차단되는가

### 6.9 협업 기반 접근 — 인수 조건과 ATDD
- 사용자 스토리 3요소(3C): 카드(스토리 기록), 대화(사용 방법 논의), 확인(인수 조건)
- 좋은 스토리 기준 INVEST: 독립적, 협상 가능, 가치 있음, 추정 가능, 작음, 테스트 가능 — "테스트 방법을 모르겠다"는 스토리는 스토리 자체가 불명확하다는 신호
- 인수 조건 작성 2가지 형식
  - 시나리오 기반: Given(전제)/When(행동)/Then(기대 결과) — 예: "Given 장바구니에 3만 원 이상 담긴 상태에서 When 결제하면 Then 배송비 0원이 표시된다"
  - 규칙 기반: 입력-출력 매핑 표 또는 확인 목록
- ATDD 진행: 명세 워크숍(스토리·인수 조건을 함께 분석해 모호성 해소) → 인수 조건 기반으로 테스트케이스 작성 → 구현
  - 케이스 순서: 정상(긍정) 케이스 먼저 → 부정/예외 케이스 → 비기능(성능·사용성) 순
  - 케이스는 스토리의 모든 특성을 다루되 스토리 범위를 벗어나지 않게, 케이스 간 중복 없이 작성

## 7. 테스트케이스 우선순위와 리스크 기반 접근

### 7.1 우선순위 지정 3가지 전략
| 전략 | 방법 | 적합 상황 |
|---|---|---|
| 리스크 기반 | 리스크 분석 결과 가장 중요한 리스크를 다루는 케이스부터 실행 | 기본 권장. 일정 압박 시 특히 유효 |
| 커버리지 기반 | 커버리지를 가장 많이 달성하는 케이스부터 실행 | 커버리지 목표가 명확할 때 |
| 요구사항 기반 | 이해관계자가 정한 요구사항 우선순위 순으로 실행 | 요구사항 중요도가 정리되어 있을 때 |
- 실행 순서 제약: 케이스 간 종속성이 있으면 우선순위가 낮아도 선행 케이스를 먼저 실행. 환경·도구·인력의 가용 시점도 반영

### 7.2 리스크의 정의와 평가
- 리스크 수준 = 발생 가능성 × 영향도. 수준이 높을수록 대응이 중요
- 프로젝트 리스크(일정·예산·인력·공급업체 문제 — 프로젝트 목표에 영향)와 제품 리스크(기능 오류, 성능 저하, 보안 취약점 등 품질 문제 — 사용자 불만, 매출·평판 손실, 법적 책임으로 이어짐)를 구분해 관리

### 7.3 제품 리스크 분석 → 테스트에 반영
- 절차: 리스크 식별(브레인스토밍, 워크숍, 인터뷰 등으로 목록화) → 리스크 평가(분류, 가능성·영향·수준 결정, 우선순위화, 대응 제안)
- 평가 방식: 정량(가능성×영향 수치 계산) 또는 정성(리스크 매트릭스로 상/중/하 판단)
- 분석 결과의 활용
  - 테스트 범위와 철저함(어디를 얼마나 깊게) 결정
  - 수행할 테스트 레벨·유형 결정
  - 사용할 기법과 목표 커버리지 결정 (예: 고위험 결제 로직은 3-value 경계값 + 결정 테이블 전체 규칙, 저위험 화면은 체크리스트만)
  - 노력 추정과 테스트 우선순위 지정
- 리스크 완화 조치: 해당 리스크에 맞는 경험자 배치, 테스팅 독립성 적용, 리뷰·정적 분석 수행, 적절한 기법·커버리지·테스트 유형 적용, 리그레션 포함 동적 테스팅

### 7.4 테스트 피라미드와 테스팅 사분면 (배분 감각)
- 테스트 피라미드: 아래층(단위 테스트)일수록 작고 빠르고 많이, 위층(E2E·UI 테스트)일수록 크고 느리고 적게. UI 자동화만 잔뜩 만드는 구조는 비효율의 신호
- 테스팅 사분면: 비즈니스/기술 × 팀 지원/제품 평가의 2축
  - Q1 기술·팀 지원: 단위·통합 테스트 (자동화, CI 포함)
  - Q2 비즈니스·팀 지원: 기능 테스트, 스토리 테스트, 프로토타입 (인수 조건 확인)
  - Q3 비즈니스·제품 평가: 탐색적 테스팅, 사용성 테스트, UAT (수동 중심 — 기획자 주 활동 영역)
  - Q4 기술·제품 평가: 성능 등 비기능 테스트, 스모크 테스트 (자동화 중심)

### 7.5 추정 기법 (테스트 공수 산정)
- 비율 기반: 과거 프로젝트의 개발:테스트 비율을 새 프로젝트에 적용 (예: 3:2 비율, 개발 600MD면 테스트 400MD)
- 외삽법: 현재 프로젝트에서 초기 실측치를 쌓아 남은 작업을 추정 (반복형 개발에 적합)
- 와이드밴드 델파이: 전문가들이 독립 추정 → 편차 논의 → 재추정을 합의까지 반복 (플래닝 포커가 변형)
- 3점 추정: 낙관치 a, 최빈치 m, 비관치 b로 E=(a+4m+b)/6, 오차 SD=(b-a)/6 — 추정치를 "10±2일"처럼 범위로 제시 가능
- 공통 원칙: 큰 작업은 작게 쪼개서 추정하고, 추정에는 항상 오차가 있음을 이해관계자에게 명확히 전달

## 8. 결함 리포트 — 필수 항목

### 8.1 결함 보고서의 목적
- 처리 담당자가 문제를 해결하는 데 충분한 정보 제공
- 산출물 품질 추적 수단 제공
- 개발·테스트 프로세스 개선 아이디어 제공

### 8.2 결함 보고서 필수 항목
| 항목 | 내용 | 작성 요령 |
|---|---|---|
| 고유 식별자 | 결함 ID | 도구 사용 시 자동 부여 |
| 제목 | 이상 사항 한 줄 요약 | 증상+위치가 드러나게 (예: "주문서 쿠폰 적용 시 할인가 미반영") |
| 발견 일자·보고자 | 날짜, 조직, 작성자와 역할 | |
| 테스트 대상·환경 | 대상 버전, 기기·OS·브라우저 등 | 재현에 필수 |
| 결함의 정황 | 실행 중이던 테스트케이스, 활동, 사용 기법·데이터 | 관련 케이스 ID 연결 |
| 재현 절차 | 발견 절차 단계 + 로그·스크린샷·녹화 등 증빙 | 제3자가 그대로 따라 하면 재현되게 |
| 기대 결과 vs 실제 결과 | 두 가지를 반드시 분리 기술 | 기대 결과의 근거(기획서 조항)를 함께 명시하면 논쟁 방지 |
| 심각도(severity) | 이해관계자 이익·요구사항에 끼치는 영향 크기 | 영향 기준 (기능 불능/주요 기능 저해/경미) |
| 우선순위(priority) | 수정의 시급성 | 심각도와 별개 — 심각도 낮아도 릴리스 직전 노출 영역이면 우선순위 높을 수 있음 |
| 상태 | 신규/수정 대기/확인 테스트 대기/재오픈/완료/거부/중복/연기 등 | 워크플로우에 따라 갱신 |
| 참조 | 관련 테스트케이스, 요구사항 ID | 추적성 유지 |

### 8.3 결함 관리 운영 규칙
- 보고된 이상 사항이 전부 결함은 아님(오탐, 변경 요청일 수 있음) → 기록 → 분석·분류 → 대응 결정 → 종료의 워크플로우로 처리하고, 모든 이해관계자가 같은 프로세스를 따름
- 정적 테스팅(기획서 리뷰)에서 찾은 결함도 같은 방식으로 기록·추적하는 것이 좋음
- 나쁜 소식 전달이라는 특성상, 결함 정보는 제품·작성자 비난이 아닌 사실 중심의 건설적 방식으로 전달

## 9. 테스트 모니터링·보고 (진행 상황 공유)

### 9.1 주요 메트릭
- 테스트 진행: 케이스 구현률, 실행/미실행 수, 합격/불합격 수
- 결함: 발견·수정 결함 수와 우선순위, 결함 밀도, 발견 비율
- 커버리지: 요구사항 커버리지, 코드 커버리지
- 리스크: 잔여 리스크 수준 / 품질: 가용성, 응답 시간 등

### 9.2 테스트 진행 상황 보고서 (정기 발행)
- 포함 내용: 테스트 기간, 진행 상황(계획 대비 빠름/늦음과 편차), 방해 요소와 대응, 메트릭, 신규·변경 리스크, 다음 주기 예정 테스팅

### 9.3 테스트 완료 보고서 (마일스톤 종료 시 1회)
- 포함 내용: 테스트 요약, 계획 대비 테스팅·품질 평가, 계획과의 편차, 방해 요소와 대응, 메트릭, 완화되지 않은 리스크와 미수정 결함, 교훈
- 보고 대상에 따라 상세 수준·형식·빈도를 조정 (팀 내부는 비공식·자주, 경영층은 요약 중심)

### 9.4 형상 관리 기본
- 테스트케이스·결과·보고서 등 산출물에 고유 ID와 버전을 부여하고 변경을 추적 — "어느 버전 기획서로 어느 버전 앱을 테스트했는지" 답할 수 있어야 함
- 승인된 산출물은 베이스라인으로 고정하고 공식 변경 절차로만 수정

## 10. 실무 요약 — 기획자가 테스트케이스를 만들 때의 작업 순서

1. 테스트 베이시스(기획서·요구사항) 리뷰부터: 모호·누락·모순을 정적 테스팅으로 먼저 제거 (7원칙의 조기 테스팅)
2. 리스크 분석: 기능별 발생 가능성×영향도로 우선순위를 정하고, 고위험 영역에 기법과 커버리지를 집중 (완벽한 테스팅은 불가능하므로)
3. 테스트 컨디션 도출: 기능·화면·규칙 단위로 "무엇을 확인할지" 목록화, 요구사항 ID와 연결(추적성)
4. 기법 적용해 케이스 설계
   - 입력값 범위 → 동등 분할 + 경계값 분석
   - 조건 조합 규칙 → 결정 테이블
   - 상태 흐름 → 상태 전이 (비유효 전이 포함)
   - 공통 품질 → 체크리스트
   - 보완 → 오류 추정·탐색적 테스팅
5. 케이스마다 전제 조건, 입력(테스트 데이터), 절차, 기대 결과를 명시 — 기대 결과 없는 케이스는 케이스가 아님
6. 정상 케이스 → 예외·부정 케이스 → 비기능 순으로 작성하고, 우선순위(리스크/커버리지/요구사항 기반)를 부여
7. 실행 후 이상 사항은 재현 절차·기대vs실제·증빙을 갖춘 결함 보고서로 기록
8. 수정본에는 확인 테스팅 + 영향도 분석 기반 리그레션 테스팅 수행
9. 종료 시 미해결 결함·잔여 리스크를 명시한 완료 보고로 마무리 — "결함이 없다"가 아니라 "이 범위를 테스트했고 이런 리스크가 남았다"로 보고
