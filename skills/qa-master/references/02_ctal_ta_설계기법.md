# 테스트케이스 작성 심화 기법 지식 문서

- 출처 기반: ISTQB CTAL-TA(Advanced Level Test Analyst) 실러버스 v4.0 (한글 v1.1, KSTQB)
- 목적: 요구사항·기획서에서 테스트 조건과 테스트케이스를 체계적으로 도출하는 심화 기법 정리
- 관점: 웹서비스/앱 기획 산출물(기능정의서, IA, 플로우차트, 유저스토리)을 테스트 베이시스로 사용하는 실무자
- 주의: 시험 대비용 내용은 제외하고 실무 적용 지식만 재정리한 2차 저작물

## 1. 테스트 분석가의 테스트 프로세스

### 1.1 전체 흐름

| 단계 | 핵심 산출물 | 한 줄 요약 |
|---|---|---|
| 테스트 분석 | 테스트 컨디션 목록 | 기획서에서 "무엇을 테스트할지" 도출 |
| 테스트 설계 | 테스트케이스(상위→하위) | "어떻게 테스트할지" 구체화 |
| 테스트 구현 | 테스트 절차·스위트·데이터 | 실행 가능한 상태로 조립 |
| 테스트 실행 | 결과 기록·결함 보고 | 실행하고 이상현상 분석 |

### 1.2 테스트 분석 — 테스트 컨디션 도출

- 시작 전 확인할 조건 3가지
  - 테스트 범위·목적·접근법이 담긴 계획 존재
  - 테스트 베이시스(요구사항, 유저스토리, 기획서) 정의 완료
  - 사전 식별된 제품 리스크의 평가 완료
- 수행 내용
  - 테스트 베이시스의 완전성 확인 및 결함 식별(기획서 자체의 결함 발견이 첫 번째 임무)
  - 문서뿐 아니라 구두 정보(스토리 협의 대화 등)도 수집 대상
  - 범위 내 각 테스트 항목에 대해 테스트 컨디션 정의 후 우선순위 지정
  - 테스트 컨디션은 테스트 베이시스 요소로 추적 가능해야 함
- 단계적 상세화 방식
  - 상위 컨디션부터 시작: "화면 X의 기능"
  - 이후 상세 컨디션으로 분해: "화면 X에서 계좌번호가 한 자리로 짧으면 거부"
  - 이 방식이 커버리지 확보와 테스트 설계 조기 착수를 동시에 지원
- 애자일 환경에서는 테스트 컨디션을 유저스토리의 인수 기준(Acceptance Criteria) 형태로 표현 가능
- 이해관계자를 테스트 컨디션 리뷰에 참여시켜 기획 이해가 일치하는지 확인

### 1.3 테스트 설계

- 테스트 컨디션마다 명확한 성공/실패 판정 기준 식별
- 상위 수준으로 둘 영역과 하위 수준까지 구체화할 영역을 구분해 결정
- 테스트 베이시스 ↔ 테스트 컨디션 ↔ 테스트케이스 간 추적성 확보
- 테스트케이스는 의사소통 수단: 작성자가 아닌 사람(다른 테스터, 개발자, 감사자)도 수행 방법과 목적을 이해할 수 있어야 함
- 설계 종료 판단: 계획의 완료 기준 + 잔존 리스크 수준 + 예산·시간 제약

### 1.4 상위 수준 vs 하위 수준 테스트케이스

| 구분 | 상위 수준(논리적) | 하위 수준(구체적) |
|---|---|---|
| 내용 | 테스트 컨디션만 명시, 값은 추상적 | 전제조건·입력값·기대결과·사후조건을 구체값으로 명시 |
| 예 | "두 권 이상 주문하면 할인 적용됨을 확인" | "도서 B10($10)+B2($20) 주문, 총 $30 → 10% 할인, 최종 $27" |
| 용도 | 컨디션 커버리지 확인, 탐색적 테스트 차터의 목표 | 실행·재현·자동화 |

- 일반적 순서: 상위 수준을 먼저 설계 → 이를 기초로 하위 수준 전개(1:N 가능)
- 상위→하위 전환은 값 채우기가 아니라 개념 단계→기술 단계의 이동
- 실무 대부분의 테스트케이스는 일부만 구체적인 하이브리드 형식(유지보수성 vs 이해용이성의 절충)

### 1.5 테스트케이스 품질 기준 (셀프 체크리스트)

- 정확성: 기반 테스트 컨디션을 정확히 확인하는가
- 실행 가능성: 실제로 수행할 수 있는가
- 필요성: 명확한 테스트 목적이 있고 중복이 없는가
- 이해 용이성: 작성자가 아닌 사람이 검토·수정·실행 가능한가
- 추적성: 컨디션·요구사항·리스크로 거슬러 올라갈 수 있는가
- 일관성: 언어·형식·구조가 통일되어 있는가(용어집 활용)
- 정밀도: 해석이 하나뿐인가 — "적절한", "필요에 따라", "여러 개의" 같은 모호어 금지
- 완전성: 필요한 테스트 데이터와 명확한 기대 결과가 있는가
- 간결성: 큰 케이스 하나보다 작은 케이스 여러 개가 유리(장애 원인 추적 용이, 조합 유연, 실패 시 후속 차단 없음)

### 1.6 테스트 오라클 (기대 결과의 근거)

- 기대 결과를 판정할 근거가 테스트 오라클 — 이상적으로는 기획서/명세가 그 역할
- 테스트 오라클 문제: 명세 불완전·모호, 데이터 복잡성, 비결정성(AI 시스템), 확률적 동작 때문에 기대 결과를 정할 수 없는 상황
- 해결책
  - 수도 오라클: 동일 명세를 만족하는 독립 시스템(레거시, 단순화 버전)과 비교
  - 모델 기반: 테스트 모델에 기대 동작을 형식화해 자동 도출
  - 속성 기반: 입력과 출력 간 성립해야 할 속성으로 검증
  - 변성 테스팅: 입력 변화와 출력 변화의 관계로 검증(3.9절)
  - 인간 오라클: 사람의 판단(탐색적 테스팅에서 선호)
- 기획 실무 시사점: 기획서에 기대 결과가 안 적혀 있으면 테스트 오라클 문제가 생김 → 기대 결과를 명세에 반드시 포함

### 1.7 테스트 데이터 요구사항

- 운용 데이터 유사성: 실데이터는 변동성이 부족할 수 있음 → 합성 데이터로 패턴·분포·특이값을 통제. 퍼소나를 데이터 생성 가이드로 활용
- 기밀성: 개인정보는 가명화(식별자 대체) 또는 익명화(식별정보 제거), 규정(GDPR 등) 준수
- 커버리지 일치: 선택한 기법의 커버리지 기준에 맞는 데이터 + 네거티브 테스트용 무효 데이터
- 유지보수성: 하위 수준 케이스에 데이터 하드코딩 금지, 로직과 데이터 분리
- 시간 민감성: 오래된 데이터·날짜 의존 데이터가 예상 밖 동작을 유발할 수 있음
- 가용성: 외부 연동 시스템이 없으면 서비스 가상화로 대체

### 1.8 테스트 환경 요구사항

- 테스트 환경에서의 통과/실패가 운용 환경에서도 재현되어야 신뢰 가능
- 각 환경 항목 명세에 포함할 것: 고유 식별자, 설명, 책임자, 필요 기간, 충실도(운용 환경 대표 정도)
- 환경 준비 완료 확인은 스모크 테스트 설계·실행으로 수행

### 1.9 키워드 주도 테스팅 (테스트 스크립트의 구조화)

- 개념: 테스트 단계를 재사용 가능한 "키워드"로 정의하고, 키워드를 조합해 테스트 스크립트를 작성하는 방식. 자동화 접근법이지만 수동 테스트에 적용해도 이후 자동화 전환을 쉽게 만듦
- 키워드 2종
  - 액션 키워드: 테스트 대상·환경과 상호작용 (예: 데이터 제출, 화면 이동, 설정 변경)
  - 베리피케이션 키워드: 실제 결과와 기대 결과의 일치 여부 확인
- 추상화 계층 2단
  - 도메인 계층: 비즈니스 용어를 반영한 업무 단위 (예: "회원 인증", "주문 생성")
  - 테스트 인터페이스 계층: 기술적 조작 단위 (예: "버튼 클릭", "필드 입력")
- 키워드 작성 필수 요건: 동사(+명사)의 명령형, 의미는 단 하나, 도메인 용어 반영, 재사용 가능, 문서화
- 도출 예시 — 유저스토리 "회원으로서 본인 인증을 하고 싶다" + 인수 기준 "유효한 회원 카드로 인증 가능"
  - 액션 키워드 "회원 인증(파라미터: 회원 카드)" + 베리피케이션 키워드 "접근 확인"
- 실무 시사점: 기획서의 기능을 동사형 업무 단위로 쪼개는 훈련 자체가 테스트 스텝 표준화와 중복 방지에 기여

### 1.10 테스트웨어 관리

- 도구 유형: 테스트 관리(저장소·추적성·일정·보고), 결함 관리, 테스트 데이터 관리, 형상 관리, 요구사항 관리
- 테스트 분석가의 관리 포인트
  - 대상 버전과 일치하는 테스트웨어 하위 집합 선택
  - 기능별·모듈별 등 케이스 구성 구조 정의
  - 케이스에 메타데이터 부여(예상 공수, 필요 환경)
  - 요구사항↔컨디션↔테스트↔결함 추적성 유지
  - 오래된(더 이상 유효하지 않은) 테스트케이스 식별·정리

## 2. 리스크 기반 테스팅

### 2.1 개요

- 테스트 항목의 리스크 수준에 따라 테스트 노력의 우선순위를 정하는 접근법
- 구성: 리스크 분석(식별+평가) → 리스크 제어(완화+모니터링)
- 리스크는 고정이 아님 → 정기 모니터링(반복 주기마다 리스크 목록 갱신)

### 2.2 리스크 식별

- 활용 수단: 회고, 리스크 워크숍, 브레인스토밍, 체크리스트, 이해관계자 인터뷰
- 테스트 분석가는 "무엇이 잘못될 수 있고 그 영향이 무엇인지"에 대한 경험·직관을 제공하는 핵심 이해관계자

### 2.3 리스크 평가 — 리스크 수준 결정 요소

- 영향받는 기능의 사용 빈도와 중요도
- 영향받는 비즈니스 목표의 중요도
- 재정·환경·평판 피해 규모
- 테스트 베이시스(기획서)의 품질
- 법적·안전 요구사항 여부
- 리스크가 균일 분포가 아니므로 테스트 대상을 항목(컴포넌트, 인터페이스, 기능)으로 세분화해 개별 평가

### 2.4 리스크 완화 — 우선순위를 테스트로 연결

- 리스크 수준에 따라 결정할 것: 테스트 레벨, 테스트 유형, 테스트 기법, 독립성 수준, 테스트 철저함
- 리스크가 높을수록 → 더 엄격한 커버리지 기준(예: 페어와이즈 대신 전체 조합), 더 공식적인 기법
- 리스크가 낮거나 일정이 촉박하면 → 경험 기반 기법으로 대체 가능
- 시프트 레프트: 같은 리스크를 완화할 수 있는 가장 이른 시점의 활동(리뷰 등 정적 테스팅 포함)을 선택

### 2.5 리그레션 테스트 범위 결정 (변경 영향 분석)

- 목적: 변경 후 기존 품질에 대한 확신 확보. 제약(시간·예산) 때문에 전부 실행 불가 → 선택 기법 필요
- 선택 기법 (여러 개 조합이 일반적)
  - 리스크 기반 선택: 리스크 목록↔리그레션 스위트 추적성 유지, 변경 시 최고 리스크 커버로 조정
  - 이력 기반: 과거에 결함이 나왔거나 유사 변경에 민감했던 테스트 재실행 + 오래 안 돌린 테스트 일부 포함
  - 커버리지 기반: 적은 수로 높은 커버리지를 내는 테스트 선택
  - 요구사항 추적성 매트릭스: 요구사항 변경이 미치는 직·간접 영향 기능의 테스트 선택 (애자일에서는 영향받는 인수 기준 기준으로 동일 수행)
  - 운영 프로파일 기반: 실제 사용 패턴(로그인→검색→장바구니→주문) 순서의 핵심 흐름 우선
  - 영향도 분석: 변경된 구성 항목과 상호작용하는 테스트케이스 선택(도구 지원 시 가장 신뢰도 높음)
- 매 주기 후 결과를 분석해 효과 있던 기법은 유지, 없던 기법은 교체 → 선택 프로세스 자체를 지속 개선

## 3. 블랙박스 테스트 기법 심화

기법 분류 (모델링하는 대상 기준)
- 데이터 기반: 동등분할, 경계값, 도메인 테스팅, 조합(분류 트리·페어와이즈), 랜덤
- 동작(행위) 기반: CRUD, 상태 전이, 시나리오/유스케이스
- 규칙 기반: 결정 테이블, 변성 테스팅

### 3.1 동등분할 (Equivalence Partitioning)

- (a) 언제: 입력·출력 값의 집합을 "시스템이 같게 취급하는 그룹"으로 나눌 수 있을 때. 거의 모든 입력 검증의 출발점
- (b) 절차
  1. 입력(또는 출력) 파라미터 식별
  2. 같은 동작을 유발하는 값의 그룹(분할)으로 나눔 — 유효 분할과 무효 분할 모두
  3. 각 분할에서 대표값 1개씩 선택해 테스트케이스화
  4. 무효 분할은 한 번에 하나씩만 적용(결함 가려짐 방지)
- (c) 커버리지: 실행된 분할 수 ÷ 전체 분할 수
- (d) 예시 — 로그인 비밀번호 정책(8~20자)
  - 유효 분할: 8~20자 / 무효 분할: 7자 이하, 21자 이상, 빈 값, 허용 외 문자 포함
  - 각 분할에서 1개씩 → 최소 5개 케이스

### 3.2 경계값 분석 (Boundary Value Analysis)

- (a) 언제: 분할의 경계가 숫자·순서로 정의될 때. 결함은 경계에 몰리는 경향("이상/초과" 구현 오류)
- (b) 절차
  1. 동등분할 먼저 수행
  2. 각 분할의 최소·최대 경계 식별
  3. 2-value: 경계값 + 경계 바로 바깥 값 (경계당 2개)
  4. 3-value: 경계 바로 안 값 + 경계값 + 경계 바로 바깥 값 (경계당 3개, 더 엄격 — 경계 구현이 복잡하거나 리스크가 높을 때)
- (c) 커버리지: 테스트된 경계값 수 ÷ 식별된 전체 경계값 수
- (d) 예시 — 주문 수량 제한(1~99개)
  - 2-value: 0, 1, 99, 100
  - 3-value: 0, 1, 2, 98, 99, 100

### 3.3 도메인 테스팅 (EP+BVA의 다변수 확장)

- (a) 언제: 여러 파라미터가 부울 조건으로 얽혀 하나의 판정을 만드는 복잡한 입력 도메인일 때 (예: "높이 > 1.29 AND 무게/높이² ≥ 30")
- (b) 절차
  1. 조건식을 분해해 각 관계식이 만드는 경계 식별
  2. 경계를 닫힌 경계(≤, ≥, =)와 열린 경계(<, >, ≠)로 분류
  3. 각 경계에 대해 ON/OFF/IN/OUT 값 선정
     - ON: 닫힌 경계면 경계 위의 값, 열린 경계면 경계에 가장 가까운 분할 내부 값
     - OFF: 닫힌 경계면 분할 바깥 가장 가까운 값, 열린 경계면 경계 위의 값
     - IN: 분할 내부 값(ON 아님) / OUT: 분할 외부 값(OFF 아님)
- (c) 커버리지 기준
  - 단순 도메인 커버리지: 부등호 경계마다 ON 1개+OFF 1개, `=`는 ON 1개+양쪽 OFF 2개, `≠`는 OFF 1개+양쪽 ON 2개
  - 신뢰 도메인 커버리지: 부등호 경계마다 ON+OFF+IN+OUT 각 1개 (항목 수는 약간 늘지만 결함 검출력이 훨씬 높음)
  - 커버리지 항목 수는 경계 수에 선형 비례, 인접 경계 간 값 공유로 최적화 가능
- (d) 예시 — 무료배송 판정 "주문금액 ≥ 30,000원 AND 배송지가 도서산간 아님"
  - 금액 경계(닫힌): ON=30,000, OFF=29,999, IN=50,000, OUT=10,000
  - 지역 조건과 조합해 잘못된 연산자(>로 잘못 구현)·상수(3만→3만1천 오타) 결함 검출

### 3.4 분류 트리 + 조합 테스팅 (페어와이즈 포함)

- (a) 언제: 여러 파라미터의 "값 조합"이 동작에 영향을 주고, 전체 조합이 폭발적으로 많을 때(환경 조합, 옵션 조합). 장애의 97%가 1~2개 파라미터 상호작용에서 발생한다는 연구가 페어와이즈의 근거
- (b) 절차
  1. 파라미터와 각 파라미터의 값 식별
  2. 값이 많은 파라미터는 동등분할을 먼저 적용해 값 수 축소
  3. 분류 트리(또는 기능 모델)로 파라미터-값 구조를 시각화
  4. 실행 불가능한 조합(제약)을 제거하고, 문제로 알려진 조합은 수동 추가
  5. 커버리지 기준을 정해 조합 생성(도구 활용 권장)
- (c) 커버리지 기준
  - 베이스 초이스: 파라미터마다 기본값 1개를 정해 기본 조합 1개 생성 → 파라미터를 하나씩만 비기본값으로 바꿔 추가 조합 생성 (중요 조합에 집중)
  - 페어와이즈: 임의의 두 파라미터 간 모든 값 쌍이 최소 1회 등장하도록 조합 생성
  - 전체 조합: 리스크가 매우 높을 때만(개수 = 각 파라미터 값 수의 곱)
- (d) 예시 — 앱 결제 화면 호환성
  - 파라미터: OS(iOS/Android) × 결제수단(카드/간편결제/휴대폰) × 회원등급(일반/VIP) × 쿠폰(유/무)
  - 전체 조합 24개 → 페어와이즈로 약 8개 내외로 축소하면서 모든 2개 파라미터 쌍 커버

### 3.5 랜덤 테스팅

- (a) 언제: 도메인 지식이 부족하거나 대량 데이터가 필요할 때, 사람의 편향(익숙한 기능만 테스트)을 피하고 싶을 때. 퍼즈 테스팅·카오스 엔지니어링의 기반
- (b) 절차: 확률 분포 지정 → 입력 도메인에서 무작위 추출 → 자동화된 오라클로 판정. 밸리데이션 목적이면 운영 프로파일 분포, 베리피케이션 목적이면 사용 사례와 무관한 분포
- (c) 커버리지: 공인 기준 없음 → 종료 조건은 실행 테스트 수·시간 등으로만 설정
- (d) 한계: 데이터의 의미를 무시해 의미 관련 결함을 놓침, 자동 오라클 의존, 중복 생성 가능
- 예시 — 챗봇 입력창에 무작위 문자열·이모지·초장문을 대량 투입해 크래시/오류응답 탐지

### 3.6 결정 테이블 테스팅

- (a) 언제: 상태와 무관하게 적용되는 비즈니스 규칙(할인 정책, 권한 판정, 승인 조건)이 여러 조건의 조합으로 정의될 때
- (b) 절차
  1. 조건과 액션을 나열해 전체 결정 테이블 작성 (규칙 수 = 각 조건 값 수의 곱)
  2. 실행 불가능한 규칙 제거
  3. 축소(최소화): 액션이 같고 조건 하나만 다르며 그 조건의 모든 값을 포함하는 규칙들을 병합, 무관 조건은 '-' 표기
  4. 체크섬 검증: 축소 테이블 각 규칙의 점수('-' 없는 규칙=1점, '-' 하나당 해당 조건 값 수만큼 곱) 합계가 원본 규칙 수와 같은지 확인 — 작으면 누락, 크면 중복
  5. 이해관계자와 함께 검토: 일관성(같은 조합에 다른 액션 없음), 실행 가능성, 완전성(조합 누락 없음), 정확성(의도 반영)
  6. 규칙(열)마다 테스트케이스 도출
- (c) 커버리지: 실행된 열 수 ÷ 실행 가능한 전체 열 수. 리스크가 높으면 축소하지 말고 전체 테이블 기준으로 커버
- (d) 예시 — 배송비 정책
  - 조건: 회원등급(일반/VIP), 주문금액(3만 미만/이상), 도서산간(예/아니오)
  - 전체 8규칙 → "VIP는 금액 무관 무료"라면 VIP 행 병합으로 축소 → 축소 테이블 검토 중 "VIP+도서산간" 조합의 액션 미정의 발견 = 기획 결함 조기 검출

### 3.7 상태 전이 테스팅

- (a) 언제: 이벤트에 대한 반응이 현재 상태에 따라 달라지는 시스템(주문 상태, 회원 상태, 대화형 챗봇, 구독 수명주기)
- (b) 절차
  1. 상태(노드)·이벤트·전이(엣지)·가드 조건·액션으로 상태 모델 작성 (기획서에 모델이 있으면 테스트 관점으로 검증·보완)
  2. 모델 자체를 먼저 리뷰(누락 상태, 도달 불가 상태, 비결정적 동작 등 — 6.2절)
  3. 커버리지 기준을 정하고 전이 시퀀스를 테스트케이스로 도출
- (c) 커버리지 기준
  - 0-스위치(=유효 전이 커버리지): 모든 개별 전이를 최소 1회
  - 1-스위치: 한 상태로 들어오고 나가는 연속 전이 쌍(전이 2개 시퀀스)을 모두 커버 — 이벤트 순서 의존 결함 검출
  - N-스위치: 연속 전이 N+1개 시퀀스. 2-스위치 이상 100%는 개수가 기하급수적이므로 예기치 않은 이벤트 순서 리스크가 높을 때만
  - 라운드 트립: 시작=종료 상태이고 중간 상태가 반복되지 않는 루프 경로를 커버 — 결함 검출력이 높다고 입증된 기준
- (d) 예시 — 주문 상태 흐름 (결제대기→결제완료→배송준비→배송중→배송완료, +취소)
  - 0-스위치: 각 전이 1회씩
  - 1-스위치: "결제완료→배송준비→배송중"과 "결제완료→취소" 같은 연속 쌍 전부 → "배송준비 중 취소" 같은 순서 의존 시나리오가 기획서에 정의됐는지 드러남
  - 라운드 트립: "결제대기→결제완료→취소→(재주문)→결제대기" 루프

### 3.8 CRUD 테스팅

- (a) 언제: 데이터 엔티티(회원, 주문, 게시글, 쿠폰)의 수명주기 전체가 올바르게 다뤄지는지 확인할 때. 주로 시스템 레벨
- (b) 절차
  1. CRUD 매트릭스 작성: 열=엔티티, 행=기능, 셀에 C/R/U/D 표기
  2. 완전성 테스팅(정적): 모든 엔티티에 C·R·U·D가 모두 존재하는지 — 누락 작업은 조사 필요한 이상현상 (예: 생성만 있고 삭제가 없는 엔티티)
  3. 일관성 테스팅(동적): 기능 간 조합으로 엔티티의 전체 수명주기를 실행, 네거티브 포함(생성 전 조회, 삭제 후 수정 등)
- (c) 커버리지: 실행된 작업 수 ÷ 매트릭스의 전체 작업 수. 엄격 버전은 작업 조합(예: 각 U 후 모든 R)까지 커버
- (d) 예시 — 배송지 관리
  - 배송지 등록(C), 목록/상세(R), 수정(U), 삭제(D) + 주문 화면의 배송지 읽기(R, 암시적 Read에 특히 주의)
  - 일관성 테스트: 배송지 수정 후 진행 중 주문의 배송지가 바뀌는지/유지되는지 → 기획서에 미정의라면 기획 결함
- 검출 결함 유형: 데이터 무결성 위반, 접근 제어 결함, 데이터 불일치

### 3.9 시나리오 기반(유스케이스 기반) 테스팅

- (a) 언제: 사용자 관점의 엔드투엔드 흐름을 검증할 때. 주로 시스템·인수 테스트. 시나리오 발굴에는 유저스토리, 퍼소나, 사용자 여정 맵 활용
- (b) 절차
  1. 워크플로우를 시나리오 모델로 작성 — 활동 다이어그램(플로우차트 확장, 병행 흐름 표현 가능) 또는 유스케이스
  2. 유스케이스라면 시나리오를 3종으로 구분
     - 기본 시나리오(해피 패스): 목표를 달성하는 표준 흐름 (유스케이스당 1개)
     - 확장(대체) 시나리오: 다른 경로지만 결국 목표 달성 (예: 쿠폰 적용 후 주문)
     - 예외 시나리오: 비정상 입력·상황으로 목표 미달성 (예: 결제 실패, 재고 소진)
  3. 시나리오마다 테스트케이스 설계, 리스크 기반으로 우선순위
  4. 루프가 있으면 단순 루프 커버리지 적용: 0회(건너뜀)·정확히 1회·2회 이상·최대 횟수
  5. 시나리오 내 변수에 EP/BVA가 필요하면 한 시나리오를 여러 케이스로 테스트
- (c) 커버리지: 실행된 시나리오 수 ÷ 식별된 전체 시나리오 수
- (d) 예시 — 배송조회
  - 기본: 주문내역 → 배송조회 → 실시간 위치 확인
  - 확장: 비회원이 주문번호+연락처로 조회
  - 예외: 존재하지 않는 주문번호, 배송 시작 전 조회, 택배사 API 응답 없음
  - 기획서 QA 효과: 예외 시나리오를 나열하는 과정에서 기획서에 없는 화면·메시지가 드러남

### 3.10 변성 테스팅 (Metamorphic Testing)

- (a) 언제: 정답(기대 결과)을 미리 알기 어려운 테스트 오라클 문제 상황 — 검색 랭킹, 추천, AI 챗봇 응답, 통계·집계 기능
- (b) 절차
  1. 원본 테스트케이스 준비
  2. 변성 관계(MR) 정의: "입력을 이렇게 바꾸면 출력은 이렇게 변해야 한다"는 속성
  3. MR로 원본을 변형한 후속 테스트케이스 생성(MR 결합 가능, 랜덤 테스팅과 결합해 다수 생성 가능)
  4. 원본+후속을 하나의 절차로 실행, 관계 충족 여부로 합불 판정
- (c) 커버리지: 공인 기준 없음. MR당 1회 커버로는 불충분 — 같은 MR에 여러 입력 조합 적용 권장
- (d) 예시 — 상품 검색
  - MR1: "필터를 추가하면 결과 개수는 늘지 않아야 한다" → "노트북" 검색 결과 대비 "노트북+무료배송" 결과가 더 많으면 실패
  - MR2: "가격 오름차순과 내림차순의 결과 집합은 동일해야 한다"
  - 정확한 결과 목록을 몰라도 결함 검출 가능

## 4. 경험 기반 테스트 기법

### 4.1 탐색적 테스팅 — 테스트 차터 작성법

- 테스트 차터 = 테스트 세션의 미션 문서. 집중을 유지시키면서 자유 탐색의 유연성을 보장하는 로드맵 (실행할 케이스 목록을 명시하지는 않음)
- 미션 표준 문구: "[정보]를 찾기 위해 [자원]을 가지고 [대상]을 탐색한다"
  - 대상: 탐색할 영역·기능·리스크·요구사항
  - 자원: 사용할 테스트 데이터, 구성, 도구, 휴리스틱, 제약
  - 정보: 찾으려는 것 — 품질 특성 평가, 예상 결함 유형, 표준 위반
- 차터 설계 시 고려 요인: 고객·요구사항 요인(비즈니스 유스케이스, 사용자 여정 맵), 제품 요인(기능 흐름, 인터페이스), 프로젝트 관리 요인(시간 제약, 비즈니스 가치)
- 차터에 담을 수 있는 추가 정보
  - 조직 정보(세션 기간, 시작 시각, 테스터)
  - 테스트 목표와 범위(관심 영역, 사용할 기법, 테스트 아이디어, 종료 조건, 테스트하지 않을 것)
  - 시작 조건(전제 조건), 테스트 환경 설명
  - 제한 사항(제품이 절대 해서는 안 되는 것 — 거짓 양성 보고를 줄여줌)
  - 이력 정보(과거 결함, 미결 질문, 장애 패턴), 규제·표준 제약
- 정보량과 유연성은 반비례: 목표만 주면 탐색 여지가 크고, 기법까지 지정하면 활동이 제한됨
- 세션 중 테스트 로그 기록: 결과 + 향후 테스트용 질문·관찰·아이디어 → 세션 시트에 문서화
- 차터 미션 예시 — "결제 실패 후 재시도 흐름에서 중복 결제 결함을 찾기 위해, 테스트 카드와 네트워크 지연 시뮬레이터를 가지고 결제 모듈을 탐색한다"

### 4.2 체크리스트 기반 테스팅

- 용도: 알려진 필수 확인 항목의 누락 방지, 테스터 간 일관성 확보, 탐색 아이디어 고갈 시 영감 제공. 요구사항이 자주 바뀌는 환경에서 케이스 문서화 부담 절감
- 체크리스트 유형 2가지
  - 읽기-수행(read-do): 확인할 구체 항목 명시 (예: 입력 필드별 무효 값 목록)
  - 수행-확인(do-confirm): 사고를 안내하는 보조 질문 (예: "검색 결과가 관련성이 있는가?")
- 작성 절차
  1. 범위·목표·형식 결정
  2. 정보 수집: 전문가 통찰, 결함 라이브러리·결함 분류법, 과거 리스크·시나리오 분석
  3. 항목 작성: 명확·구체·비모호·실행 가능·측정 가능, "예/아니오/해당없음"으로 답할 수 있는 질문형, 중요도·리스크로 우선순위 부여
  4. 구조화: 기능 영역·사용자 역할·테스트 레벨별로 그룹화
  5. 지속 개선: 새 발견·피드백·이전 주기의 교훈 반영, 다른 테스터와 공유
- 주의: 체크리스트는 상세 매뉴얼(how-to)이 아니라 숙련자용 빠른 도구. 목록 밖 항목도 함께 봐야 검출 극대화

### 4.3 결함 기반 접근 (결함 분류법 활용)

- 과거 결함 데이터·결함 분류법(taxonomy)에서 "잘 나오는 결함 유형"을 뽑아 그 유형을 노리는 테스트 설계
- 반복 발생 결함 유형이 알려져 있으면 그 유형에 특화된 기법(주로 체크리스트 기반) 선택
- 이전 반복주기·프로젝트와 유사한 결함이 예상되면 그때 성공했던 기법 재사용이 합리적

### 4.4 크라우드 테스팅

- 다양한 지역·기기·배경의 내외부 테스터 그룹에 테스트를 분산하는 접근법
- 장점: 다양한 실사용 환경(기기·브라우저·네트워크), 확장 유연성, 비용 효율, 신속한 피드백, 실사용자 관점(UX·사용성·인수 테스트에 가치)
- 한계: 테스터 숙련도 편차, 시간대·언어 등 소통 어려움, 보안·기밀 리스크, 중복·거짓 양성 보고 관리 부담
- 위치: 기법을 쓰는 테스트 분석가를 대체하는 것이 아니라 환경 커버리지를 보완하는 수단

## 5. 테스트 기법 선택 가이드

### 5.1 선택에 영향을 주는 요인

- 테스트 목적: 수치 계산 검증엔 도메인 테스팅, 심사·정책 로직엔 결정 테이블 등 평가 대상 성격이 기법을 유도
- 제품 리스크와 결함 유형의 매칭
  - 데이터 기반 기법 → 데이터 처리, 도메인 구현, UI 입력, 계산, 파라미터 조합 결함
  - 동작 기반 기법 → 기능 누락, 통신·처리 결함 등 사용자 요구사항 결함
  - 규칙 기반 기법 → 로직·제어 흐름 결함
- 테스트 베이시스 형태: 명세에 모델이 있으면 그 모델 기반 기법, 오라클 도출이 어려우면 변성·경험 기반
- 테스터의 지식·경험: 익숙하지 않은 기법을 중요 업무에 쓰지 말 것, 도메인 지식이 없으면 탐색적 테스팅 비효율
- SDLC: 순차 모델은 공식적 기법, 반복 모델은 경량(경험 기반) 기법·설계 자동화에 적합
- 고객·계약·규제 요구사항, 시간·예산 제약

### 5.2 기법 조합 패턴

- 상태 전이 테스팅의 가드 조건에 경계값 분석 적용
- 시나리오 기반 테스팅 내 변수 값 결정에 도메인 테스팅 적용
- 시나리오 기반 + 결정 커버리지(프로세스 사이클 테스팅) — 비즈니스 프로세스의 분기 엄격 커버
- 시나리오 기반 + 라운드 트립 커버리지 — 주기적 비즈니스 프로세스 리스크 대응
- 변성 테스팅 + 랜덤 테스팅 — 하나의 MR로 다수 케이스 자동 생성

### 5.3 상황별 빠른 매핑 (실무 요약)

| 기획서에서 보이는 것 | 우선 검토할 기법 |
|---|---|
| 입력 필드와 유효 범위 | 동등분할 → 경계값(2/3-value) |
| 여러 조건이 얽힌 판정식 | 도메인 테스팅 |
| 조건 조합별 정책 표(할인, 권한 등) | 결정 테이블 (+체크섬 검증) |
| 상태·단계가 있는 흐름(주문, 회원) | 상태 전이 (0/1-스위치, 라운드 트립) |
| 데이터 엔티티의 등록·수정·삭제 | CRUD 매트릭스 |
| 사용자 여정·플로우차트 | 시나리오 기반 (기본/확장/예외) |
| 환경·옵션 조합 | 분류 트리 + 페어와이즈 |
| 정답을 정의하기 어려운 기능(검색, AI) | 변성 테스팅 |
| 명세가 없거나 시간이 없음 | 탐색적(차터) + 체크리스트 |

### 5.4 테스트 설계 자동화 (모델 기반 테스팅)

- 방식: 테스트 모델(상태 모델 등)을 만들면 도구가 커버리지 기준에 맞는 테스트웨어를 생성
- 이점: 모델링 과정 자체가 명세 결함을 드러냄(결함 예방), 복잡한 커버리지(N-스위치, 조합) 적용 가능, 반복 작업 감소, 모델이 단일 기준 정보라 유지보수 절감, 추적성 강화
- 리스크: 모델에 없는 테스트 컨디션 간과, 모델 유지보수 노력 과소평가, 이해관계자의 모델 이해 어려움

## 6. 기획서·명세 리뷰 — 테스트 분석가의 체크포인트

### 6.1 결함 예방 관점

- 목표: 결함이 유입된 단계(기획 단계)에서 바로 식별·제거(단계 격리) — 코드로 넘어간 뒤 고치는 것보다 비용이 훨씬 낮음
- 테스트 분석가의 기여: 리스크 분석 참여, 요구사항·모델 리뷰, 회고 참여, 결함 데이터 수집·분류, 근본 원인 분석 참여

### 6.2 모델링으로 기획서 결함 찾기

- 비형식 텍스트 명세를 모델로 옮기는 행위 자체가 강력한 리뷰: 정형화·시각화 과정에서 불완전성·불일치·모호성이 드러남
- 모델 유형별로 드러나는 기획 결함
  - 데이터 모델(도메인·조합 모델): 중첩된 분할, 커버되지 않는 도메인 영역, 빈 분할, 누락·잘못된 파라미터 조합
  - 동작 모델(CRUD 매트릭스·상태 모델·시나리오 모델): 불완전한 엔티티 수명주기, 누락·잘못된 상태 전이, 데드락, 무한 루프, 누락된 예외 처리
  - 규칙 모델(결정 테이블·변성 관계): 규칙 누락, 불일치, 모호성, 중복, 복잡한 비즈니스 로직
- 기획서에 이미 모델(플로우차트, 상태도, 정책표)이 있을 때 점검 항목
  - 상태 전이 다이어그램: 누락·잘못된 상태, 부적절한 전이, 부정확한 가드 조건, 도달 불가능한 상태, 비결정적 동작
  - 활동 다이어그램(플로우차트): 연결 안 되거나 도달 불가능한 액션, 막다른 액션, 잘못된 순서, 비배타적·불완전한 분기 조건, 누락된 동기화 지점
  - 결정 테이블: 중첩·불일치 규칙, 실현 불가능한 규칙, 조합 누락·액션 누락
  - 공통: 구문 오류, 오타, 중복, 명명 불일치

### 6.3 리뷰 기법 5종

| 기법 | 방식 | 언제 적합 |
|---|---|---|
| 애드혹 리뷰 | 지침 없이 자유롭게 읽고 이상현상 기록 | 준비 시간이 없을 때. 리뷰어 역량 의존, 중복 보고 위험 |
| 체크리스트 기반 리뷰 | 사전 정의 체크리스트로 평가 | 객관 기준이 필요할 때. 문서 유형·리스크에 맞게 조정, 놓친 결함으로 정기 업데이트, 목록 밖도 볼 것 |
| 시나리오 기반 리뷰 | 예상 사용 흐름대로 문서를 "드라이 런" | 유스케이스·플로우차트 등 시나리오형 문서. 문서화된 시나리오에 갇히지 말 것 |
| 역할 기반 리뷰 | 리뷰어에게 사용자 역할(숙련자/초보/시니어/관리자 등, 퍼소나 가능) 할당 | 사용자 유형별 관점 커버가 필요할 때. 커버리지 확보+중복 방지 |
| 관점 기반 읽기 | 설계자·테스터·마케터·최종 사용자 등 관점으로 읽으며 그 관점의 2차 산출물을 직접 작성 | 심층 리뷰가 필요할 때. 예: 요구사항서로 인수 테스트 초안을 만들어보며 필요한 정보가 다 있는지 검증 |

- 기법 선택 요소: 리뷰 목적, 가용 자원, 문서 유형, 리스크, 도메인, 조직 문화
- 기획서 QA 실무 팁: "이 문서로 테스트케이스를 만들 수 있는가?"라는 관점 기반 읽기가 기획 완성도 검증에 가장 직접적 — 기대 결과 없는 기능, 예외 미정의, 모호어("적절히", "필요시")가 즉시 드러남

### 6.4 기획서 QA 실전 질문 목록 (6.2·6.3 종합)

- 입력·데이터 관점
  - 모든 입력 필드에 유효 범위·형식·필수 여부가 정의되어 있는가
  - 무효 입력 시의 동작(에러 메시지, 차단 방식)이 정의되어 있는가
  - 경계값(최소·최대)에서의 동작이 "이상/초과" 수준까지 명확한가
- 규칙 관점
  - 조건 조합별 정책이 표로 정리 가능한가 — 조합 누락·상충 규칙은 없는가
  - "그 외의 경우"의 액션이 정의되어 있는가
- 흐름·상태 관점
  - 모든 상태에서 나가는 전이가 정의되어 있는가(막다른 상태 없는가)
  - 진행 중 취소·중단·뒤로가기·중복 요청 시의 동작이 정의되어 있는가
  - 엔티티마다 생성·조회·수정·삭제가 모두 다뤄지는가, 삭제 후 참조 데이터 처리가 정의되어 있는가
- 시나리오 관점
  - 해피 패스 외에 확장(대체) 흐름과 예외 흐름이 각각 정의되어 있는가
  - 외부 연동(결제, 배송 API 등) 실패·지연·무응답 시의 동작이 정의되어 있는가
- 표현 관점
  - 기대 결과가 하나의 해석만 허용하는가 — "적절히", "필요시", "빠르게" 같은 모호어가 없는가
  - 용어가 문서 전체에서 일관되게 사용되는가

### 6.5 결과 분석과 재발 방지 (요약)

- 결함 클러스터 분석: 결함이 몰린 영역(소수 컴포넌트에 다수 결함)을 예측과 비교, 예상보다 많이 나온 곳은 테스트 강화. 심각도 높은 작은 클러스터가 사소한 큰 클러스터보다 중요
- 결함 탐지율(DDP): 해당 레벨에서 잡을 수 있었는데 유출된 결함 비율 — 낮으면 원인 분석 후 해당 레벨 강화, 심각도별 분석 권장
- 테스트 갭 분석: 최근 변경 코드 중 테스트되지 않은 부분(가장 오류 유발 가능성 높은 영역)에 집중
- 근본 원인 분석(RCA): 결함을 유형별로 분류한 뒤 자주 발생하는 유형에 RCA 수행(전수 분석은 비효율). 기법: 5 whys, 원인-결과 다이어그램, 파레토 분석
- 결함 분류: ODC(직교 분류), IEEE 1044, 심각도 기반, 결함 분류체계 모델 등 — SDLC 전반에 걸쳐 일관 분류해야 프로세스 개선으로 연결됨

## 7. 품질 특성 관점 보강 (기능 외 관점)

### 7.1 기능 적합성의 3가지 하위 특성 — 테스트 질문이 다름

| 하위 특성 | 핵심 질문 | 적합한 확인 방법 |
|---|---|---|
| 기능 성숙도(완전성) | 요구된 모든 것이 구현되었는가 | 요구사항 리뷰·인수 기준 논의(조기), 시나리오 기반 테스팅, 추적성 필수 |
| 기능 정확성 | 유효/무효 입력에 대한 결과가 정확한가 | 모든 블랙박스 기법, 효과적인 테스트 오라클 확보가 관건, 가능한 한 이른 레벨에서 |
| 기능 타당성 | 구현된 것이 사용자 요구를 실제로 충족하는가 | UI 디자인 리뷰, 탐색적·협업 기반 테스팅, 데모 세션 |

### 7.2 사용성 테스팅 기여

- 평가 측면: 상호작용 능력(효과·효율·만족), 사용자 경험(사용 전·중·후 인식), 접근성(장애·문화·언어 장벽)
- 기여 방법
  - 사용성 리뷰: 사용자 그룹 요구·비즈니스 목표·사용 상황에 맞게 리뷰 기준(체크리스트) 조정
  - 사용성 테스트 세션: 퍼소나·사용자 그룹·운영 프로파일 기반으로 세션 시나리오 설계
  - 설문(SUMI, WAMMI 등): 설계와 응답 평가 지원
- 접근성: WCAG 준수 수준(A/AA/AAA) 확인 — 사용 상황 분석으로 필요한 수준과 대상 사용자 요구 식별

### 7.3 유연성·호환성 (웹/앱 기획 관련 요점)

- 적용성 테스팅: 지원 대상 환경(OS 버전, 브라우저) 식별 → 조합 테스팅으로 환경 조합 커버, 신규 환경엔 스모크 테스트, 환경별 결함 모니터링
- 설치성 테스팅: 설치·삭제·업데이트·재구성이 다양한 환경에서 동작하는지 + 설치 후 정상 동작 + 사용 편의 + 모바일 앱 접근 권한 동작
- 상호운용성 테스팅: 시스템 간 정보 교환과 교환 정보의 상호 사용 확인(데이터 변환 포함 시 변환 검증 필수). 아키텍처·설계 문서에서 문서화 안 된 상호작용까지 파악. 검출 결함: 데이터 변환·해석 결함, 통신 프로토콜 결함, 엔드투엔드 기능 결함

## 8. 실무 적용 절차 요약 (기획서 → 테스트케이스)

1. 기획서(테스트 베이시스) 리뷰 — 6장의 리뷰 기법·모델링으로 기획 결함을 먼저 잡음
2. 리스크 분석 — 기능별 사용 빈도·중요도·피해 규모로 리스크 수준 결정 (2장)
3. 테스트 컨디션 도출 — 상위→상세로 단계적 분해, 리스크 반영, 추적성 부여 (1.2)
4. 기법 선택 — 5.3의 매핑표로 기획서 요소별 기법 결정, 리스크 수준으로 커버리지 강도 결정
5. 상위 수준 테스트케이스 설계 — 컨디션 커버 확인 (1.4)
6. 하위 수준 구체화 — 테스트 데이터·환경 요구사항 함께 정의 (1.7, 1.8)
7. 품질 기준 셀프 체크 (1.5)
8. 실행 결과 분석 — 결함 클러스터·DDP로 테스트 자체를 개선, 리그레션 스위트 조정 (2.5, 6.4)
