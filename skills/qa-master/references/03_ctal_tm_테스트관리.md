# 테스트 계획·QA 관리 지식 문서 (ISTQB CTAL-TM v3.0 증류)

- 출처: ISTQB Certified Tester Advanced Level - Test Management Syllabus v3.0 (한글 v1.0, KSTQB)
- 용도: 비개발 기획자/PM이 웹서비스·앱 프로젝트에서 테스트 계획서를 만들고 QA 활동을 관리할 때 참조하는 실무 규칙 모음
- 원문을 실무 관점으로 재구성한 요약이며, 시험 대비 내용(학습목표·문항 정보)은 제외함

## 1. 테스트 관리의 3대 활동

테스트 관리는 크게 세 덩어리로 구성됨. 프로젝트 전체·특정 테스트 단계·스프린트 등 어느 범위에서든 이 구조가 반복됨.

| 활동 | 하는 일 | 산출물 |
|---|---|---|
| 테스트 계획 | 목표·접근법·범위·자원·일정·산출물·참여자 정의 | 테스트 계획서 |
| 테스트 모니터링과 제어 | 진행 상황 추적, 계획 대비 편차 파악, 시정 조치, 상태 보고 | 진척 보고서 |
| 테스트 완료 | 산출물 정리·보관, 결과 평가, 교훈 도출, 완료 전달 | 완료 보고서 |

### 1.1 테스트 계획 활동의 원칙
- 가능한 한 프로젝트 초기에 시작. 이상적으로는 요구사항 확정 전부터 착수
- 한 번 쓰고 끝나는 문서가 아님. 변경·피드백을 반영해 프로젝트 내내 갱신하는 반복 활동
- 계획에 포함되는 과업 순서:
  1. 정황 이해: 조직의 테스트 정책·전략, 테스트 범위, 테스트 대상 파악. 이해관계자(PO, PM, 개발 리더)로부터 활동·일정 승인 획득
  2. 제품 리스크 식별·분석: 어떤 품질 문제가 생길 수 있고 얼마나 심각한지 평가
  3. 리스크 대응 방법 결정: 예방·수정·완화 조치를 계획서에 문서화
  4. 테스트 접근법 정의 + 자원 추정·할당: 인력·도구·환경·데이터 등
  5. 계획 확정: 모든 이해관계자의 수용이 필요하며, 의견 불일치는 확정 전 해소

### 1.2 테스트 모니터링과 제어의 원칙
- 지속적 활동임. 마일스톤 때만 보는 것이 아님
- 진척을 추적하려면 먼저 일정과 측정 프레임(지표·목표치)이 있어야 함
- 모니터링에 포함되는 것: 테스트 결과 수집·기록, 계획 대비 편차 식별, 새 리스크 식별·분석, 기존 리스크 변화 추적
- 제어에 포함되는 것: 편차 관리, 새/변경 리스크 처리, 테스트 시작 준비, 완료 조건 기준의 완료 승인
- 핵심 원칙: 상태를 비즈니스 이해관계자가 이해할 수 있는 방식으로 전달할 것 (예: "테스트 케이스 300개 중 250개 합격"보다 "결제 관련 핵심 리스크 5건 중 4건 해소, 1건은 환불 오류로 수정 중")

### 1.3 테스트 완료 활동
- 시점: 릴리스/스프린트 종료, 테스트 레벨 종료, 프로젝트 종료(또는 중단), 유지보수 릴리스 종료 등 마일스톤
- 해야 할 일:
  - 미해결 결함 처리: 남은 결함은 변경 요청 또는 제품 백로그 항목으로 전환
  - 완료 보고서 작성·승인: 계획서·결과서·진척 보고서·결함 보고서에서 정보를 모아 요약하고, 승인 후 이해관계자에게 전달
  - 테스트웨어 보관: 재사용 가치가 있는 테스트 케이스 등을 찾기 쉽게 정리. 결과·로그·보고서는 형상관리 시스템에 보관
  - 테스트웨어 인계: 유예·수용된 결함 정보 등을 이후 담당자(운영팀 등)에게 전달
  - 테스트 환경 정리: 테스트 데이터·스크립트·스텁 제거, 다음 주기를 위해 초기 상태로 복원
  - 교훈 수집: 회고를 통해 잘된 점·문제점을 문서화하고 프로세스 개선에 활용

## 2. 테스팅의 정황 파악

테스트 계획의 출발점은 "이 프로젝트가 어떤 조건과 제약 위에 있는가"를 아는 것. 제품 유형, 산업, 규제, 개발 방식(SDLC)에 따라 테스트 방식이 달라짐.

### 2.1 테스트 이해관계자와 각자의 관심
| 이해관계자 | 테스팅에 대한 관심·역할 |
|---|---|
| 개발자·개발 리더 | 시스템 구현, 테스트 결과에 따른 수정, 단위 테스트 수행 |
| 테스터·테스트 리더 | 테스트 계획·설계·실행, 결함 추적·보고, 자동화, 진척 보고 |
| PM·PO·비즈니스 담당 | 요구사항과 품질 수준 정의, 커버리지 권고, UAT 참여, 결과 기반 의사결정 |
| 운영팀 | 운영 인수 테스트 참여, 비기능 요구사항 정의 기여 |
| 고객·최종 사용자 | 요구사항 정의의 원천, UAT 참여 |

- 이 목록은 시작점일 뿐이며, 프로젝트마다 이해관계자 분석으로 추가 파악 필요

### 2.2 이해관계자 매트릭스 (영향력 x 관심도)
- 이해관계자 참여 우선순위를 정하고 기대치를 관리하는 도구
- 4개 사분면:
  - 프로모터(영향력 높음, 관심 높음): 전략·계획 수립의 핵심 협력자. 밀착 협업
  - 잠재층(영향력 높음, 관심 낮음): 일상엔 무관심하지만 자원·방향 결정권 보유. 핵심 사안만 간결히 보고
  - 디펜더(영향력 낮음, 관심 높음): 정성적 피드백 제공자. 정기 업데이트로 관심 유지
  - 무관심층(둘 다 낮음): 주요 마일스톤만 공지. 특정 이슈에서 의외의 통찰 제공 가능

### 2.3 개발 모델별 테스트 관리 방식 비교
| 활동 | 순차적 모델 (V-모델 등) | 반복적 모델 (스크럼 등) |
|---|---|---|
| 추정 | 초기에 테스트 레벨별 상세 추정 | 스프린트마다 스토리 계획과 함께 반복 추정 |
| 문서 | 전략·계획·케이스·보고서를 격식 있게 작성 | 인수 조건·완료 정의 중심, 문서 최소화 |
| 역할 | 테스트 관리자가 의사결정·팀 관리 | 역할 통합, 관리자 대신 퍼실리테이터/코치 |
| 도구 | 단계별 테스트 관리 도구 | CI/CD·자동화 도구가 핵심 |
| 접근법 | 단계에 맞춰 사전에 일정 수립 | 스프린트에 내장, 적응과 피드백 중시 |
| 자동화 | 전략적·선택적 구현 | 처음부터 리그레션 자동화 내장 |
| 보고 | 마일스톤 기반 보고 | 실시간 대시보드·일일 업데이트 |
| 지표 | 테스트 실행·결함율 등 전통 지표 | + 팀 속도, 번다운 차트 등 애자일 지표 |

- 하이브리드 모델: 순차적 구조와 애자일 유연성을 섞은 방식. 애자일 전환 과도기이거나(전환용), 고위험 과업은 순차적·나머지는 애자일로 하는 경우(목적 달성용)에 사용
- 하이브리드에서 관리 포인트: 팀 역량 평가, 장단점 식별, 스프린트 내 테스팅과 전통적 테스트 단계의 병행 관리, 팀 간 협업 강화

### 2.4 테스트 레벨별 관리 포인트 (웹/앱 프로젝트 기준)
- 컴포넌트(단위) 테스트: 범위·목표·완료 조건 정의, 개발팀과 협력. 테스터가 코드 리뷰 등에 참여 가능
- 통합 테스트: 개발팀과 협의해 통합 순서와 테스트 조합 결정, 상위 테스트 전략과 정합성 유지
- 시스템 테스트: 자원 할당·도구·일정을 신중히 계획. 애자일에서는 별도 단계 대신 스프린트 스토리 테스트와 통합
- 인수 테스트(UAT): 인수 조건 충족 확인을 위한 이해관계자 협업, 사용자 테스트 관리, 실사용 환경(고객 현장) 테스트 조정, 승인 절차 지원

### 2.5 테스트 유형별 관리 포인트
- 기능 테스트: 요구사항 대비 전략 수립·진척 추적, 모든 기능 커버를 위한 자원 배분
- 비기능 테스트: 성능 벤치마크 설정·평가, 보안·사용성·신뢰성 표준 충족 확인
- 블랙박스: 사용자 시나리오·비즈니스 요구 전체 커버 보장, 이해관계자 피드백 반영
- 화이트박스: 코드 커버리지 도구로 미검증 영역 식별, 기술적 통찰을 계획에 반영

## 3. 리스크 기반 테스팅 (RBT)

한정된 시간·인력으로 무엇을 먼저, 얼마나 깊게 테스트할지 정하는 핵심 방법. 리스크 레벨이 높을수록 테스트를 일찍 시작하고, 더 강하게, 더 오래 수행함.

### 3.1 기본 구조
- 제품 리스크 = 제품에 품질 문제가 존재할 잠재적 상황 (예: "결제 중복 승인 가능성", "피크 시간 응답 지연")
- 프로세스: 리스크 분석(식별 + 평가) → 리스크 통제(완화 + 모니터링). 논리적으론 순차지만 실제로는 겹쳐서 진행
- 테스팅의 역할: 결함을 발견하면 출시 전 처리 기회를 제공(리스크 완화), 결함이 안 나오면 리스크가 예상보다 낮음을 확인

### 3.2 리스크 식별
- 식별 기법: 전문가 인터뷰, 독립적 평가, 회고, 리스크 워크샵, 브레인스토밍, 체크리스트, 과거 경험 참조
- 실무 규칙:
  - 가능한 한 다양한 이해관계자를 참여시킬 것. 참여자 목록은 PM과 합의
  - 주요 이해관계자가 빠지면 중대 리스크를 놓침. 참석 불가 시 위임이라도 받을 것
  - 리스크는 균일하게 분포하지 않음 (예: 앱의 고객 대면 화면과 관리자 화면은 사용성 리스크가 전혀 다름) → 테스트 항목별 개별 리스크 식별 필요
  - 식별 과정의 부산물(요구사항 문서의 결함, 프로젝트 리스크 등)도 드러나게 하고 관련자에게 전달

### 3.3 리스크 평가 (리스크 레벨 = 발생 가능성 x 영향도)
발생 가능성을 높이는 요인:
- 기술·아키텍처의 복잡도, 조직 미성숙, 인력의 역량·가용성 문제, 팀 내 갈등
- 공급업체 계약 문제, 분산된 팀, 약한 리더십, 시간·예산 압박
- 초기 품질 활동 부족, 요구사항·제품·인력의 잦은 변경

영향도를 높이는 요인:
- 해당 기능의 사용 빈도와 중요성, 관련 비즈니스 목표의 중요성
- 평판 손상, 매출 손실, 금전적·사회적 손실, 법적 제재
- 인터페이스·통합 문제, 우회 수단 부재, 안전 요구사항

평가 방식:
- 통계적으로 유효한 데이터가 있으면 정량 평가(가능성 %, 영향도 수치의 곱)
- 대부분은 정성 평가: 가능성·영향도를 상/중/하 등 서수 척도로 매기고 리스크 매트릭스로 조합. 결과는 상대적 등급으로 해석
- 데이터가 없으면 결국 이해관계자의 주관적 판단에 기반함을 인지할 것

### 3.4 리스크 완화 — 테스트 배분 규칙
- 테스트 개발·실행에 드는 시간·노력은 리스크 레벨에 비례시킬 것
  - 높은 리스크: 일찍 시작, 엄격한 기법 사용, 가장 숙련된 인력 배치
  - 낮은 리스크: 늦게 시작, 가벼운 기법 적용
- 고려 요인: 테스트 항목별 리스크 차이, 영향받는 품질 특성(→ 테스트 유형 선택), 정적/동적 테스트 조합, SDLC상 시작 시점, 팀 역량, 규제 요구사항(안전 표준이 기법·커버리지를 규정하는 경우)
- 리스크 레벨은 리뷰 수행 여부, 테스트 독립성 수준, 리그레션 테스트 범위 같은 품질 제어 결정에도 반영
- 테스트 외 완화 수단: 우회 방안 마련, 제3자(공급업체)에게 리스크 이전, 리스크 수용

### 3.5 실행 우선순위 전략
- 깊이-우선(depth-first): 가장 높은 리스크부터 철저히 커버. 최고 리스크를 최대한 빨리 없애야 할 때 적합
- 너비-우선(breadth-first): 모든 리스크에 최소 1개 테스트를 먼저 배정하고 나머지를 레벨순으로. 전반적 품질 수준을 빨리 알고 싶을 때 적합
- 실무 패턴: 깊이-우선으로 시작 → 시간이 촉박해지면 너비-우선으로 전환해 모든 리스크를 최소 1회는 커버
- 시간이 다 되어 계획한 테스트를 못 끝냈을 때: RBT는 "테스트 연장 vs 잔존 리스크 수용"에 대한 근거 있는 권고를 경영진에게 제시할 수 있게 해줌

### 3.6 RBT 기법의 무게 선택
- 중량(heavyweight) 기법: 정형화된 절차·문서, 수학적 계산 사용. 안전 중요 시스템용
  - 예: 위험원 분석, 노출 비용 분석, FMEA(장애 모드·영향 분석), 결점 트리 분석
- 경량(lightweight) 기법: 서수 척도 기반, 적은 노력. 일반 웹/앱 서비스에 적합
  - 예: SST(요구사항 명세 필요), PRAM, PRISMA(이해관계자 입력만으로도 가능)

### 3.7 RBT 성공 판단 질문과 흔한 함정
성공 판단 질문 (회고에서 확인):
- 필요한 이해관계자가 리스크 분석에 참여했는가
- 운영 중 중대 결함 유출 사건이 있었다면 해결됐는가
- 우선순위 높은 결함 대부분이 테스트 초기에 발견됐는가
- 테스트 결과를 리스크 관점으로 이해관계자에게 설명할 수 있었는가
- 건너뛴 테스트가 실행한 테스트보다 리스크 레벨이 낮았는가

흔한 함정과 대책:
| 함정 | 대책 |
|---|---|
| 가능성·영향도 추정이 어려움 | 과거 데이터 활용, 핵심 이해관계자에게 평가 의견 요청 |
| 초반 열정 후 흐지부지 | 정기 모니터링과 리스크 보고를 루틴화 |
| 매번 같은 리스크만 나와 자만 | 적절한 인원 참여, 중요 리스크만 선별 완화 |
| 주요 리스크 누락 | 경험 있는 적임자 참여, 교육 제공 |
| 이해관계자 이탈·변경 | 리스크 분석을 1회성이 아닌 지속·반복 활동으로 운영 |

## 4. 프로젝트 테스트 전략과 테스트 접근법

### 4.1 개념 구분
- 테스트 전략: 주어진 상황에서 테스트 목표를 달성하기 위해 테스팅을 수행하는 방법의 설명. 전반적 범위·접근법·자원을 정의
- 테스트 접근법: 테스팅 과업을 구현하는 방식. 테스트 레벨·유형·기법의 선택과 조합, 테스트 프랙티스(스크립트 테스트, 수동 테스트 등) 선정
- 프로젝트 테스트 전략은 조직 테스트 전략의 영향을 받으며, 보통 테스트 계획서에 포함해 문서화
- 문서화가 항상 필수는 아니나, 순차적 모델·계약·규제 상황에서는 일반적으로 요구됨

### 4.2 접근법 선택 시 분석할 요소
- 도메인: 산업 규정·표준이 테스트 엄격도와 문서 수준을 좌우 (예: 의료는 환자 안전 중심의 철저한 UAT, 웹 보험 서비스는 사용성·A/B 테스트 중심 UAT)
- 조직 목표·중요 품질 특성: 자동화 확대, 결함 탐지 효율 향상 등이 레벨·유형 결정 기준이 됨
- 프로젝트 목표·유형: 예산·기간이 빡빡하면 RBT로 우선순위화 필수. 고객 맞춤형 개발이면 계약상 인수 기준 충족 테스트 필요
- 테스트 자원: 도구·인프라·인력·라이선스 제약 (예: 모바일 앱은 다양한 기기 확보 필요, 경험 기반 테스트는 도메인 지식 있는 테스터 필요)
- SDLC 모델: CI/CD 환경은 폭포수보다 훨씬 많은 자동 테스트 필요
- 타 시스템과의 인터페이스: 여러 시스템이 얽히면 팀 간 테스트 조정과 시스템 통합 테스트 레벨 선택이 필수
- 테스트 데이터 가용성: 운영 데이터 익명화 필요 여부, 특수 데이터 확보 난이도

- 기법-대상 궁합의 예: 코드 유지보수성은 정적 분석·코드 리뷰가 효율적, 성능은 스크립트 기반 시스템 테스트, 기능의 유용성은 사용자와 함께 하는 수동 인수 테스트가 적합

### 4.3 테스트 목표 정의 — S.M.A.R.T.
테스트 계획에는 최소한 테스트 범위·테스트 목표·완료 조건이 있어야 함. 목표는 S.M.A.R.T.로 작성:
- Specific(구체적): 명확하고 모호하지 않게
- Measurable(측정 가능): 달성 여부를 정량적으로 판단할 기준 포함
- Achievable(달성 가능): 자원·기간·역량 대비 실현 가능하게
- Relevant(관련된): 전체 프로젝트 목표와 연계
- Timely(시기 적절): 구체적 기한 설정

테스트 목표 예시 (웹/앱 맥락으로 각색):
- 릴리스일까지 정의된 완료 조건 달성
- 출시 후 고객 클레임 건수 목표치 이내 유지
- 개인정보보호 등 관련 규정 준수 확인
- 권한 있는 사용자만 데이터 접근 가능함을 확인
- 데이터 마이그레이션의 완전성·정확성·성능 검증
- 리그레션 테스트 자동화 비율 목표 달성
- 리팩토링 후 기존 기능 무결성을 리그레션 테스트로 입증
- 주요 화면 과업 완료 시간 측정으로 사용성 확인 (예: 온라인 스토어에서 주문 완료까지 걸리는 시간)

### 4.4 테스트 계획서에 담을 것 (실무 체크리스트)
- 테스트 목표와 완료 조건 (S.M.A.R.T.)
- 테스트 범위: 테스트 대상 항목, 기능·비기능 요구사항 커버 범위
- 테스트 접근법: 레벨·유형·기법·프랙티스 선택과 근거
- 리스크 분석 결과와 대응 방법
- 자원 계획: 인력(역할·필요 기술), 도구, 환경, 데이터 + 팀 구조·역할 구분·의사소통 방식 (분산 팀이면 특히 중요)
- 일정과 마일스톤, 테스트 시작 조건·완료 조건
- 산출물 목록과 보고 체계
- 계획의 층위: 프로젝트(마스터) 테스트 계획 → 필요 시 레벨별 계획(시스템 테스트 계획 등), 품질 특성별 계획(보안·성능), 애자일이면 스프린트 계획

## 5. 테스트 추정

### 5.1 추정의 3요소
- 노력: 과업 완료에 필요한 공수(시간) 또는 스토리 포인트 — "얼마나 많은 일인가"
- 기간: 달력·근무일 기준 소요 시간 — "얼마나 오래 걸리나" (노력과 기간은 다름)
- 비용: 자원·도구·인프라 예산 — "얼마가 드나"
- 순서: 테스트 레벨·활동·과업 식별 → 활동 단위로 분해 → 각각의 노력·비용 추정
- 시간-비용-품질은 서로 맞물린 삼각형. 하나를 줄이면 다른 것이 영향받음
- 추정 시 가정한 내용은 반드시 추정과 함께 문서화
- 추정은 그 시점의 가용 정보 기반임 → 범위 등 입력이 바뀌면 추정도 갱신

### 5.2 테스트 노력에 영향을 주는 요인
| 구분 | 요인 |
|---|---|
| 제품 | 요구사항 문서(테스트 베이시스) 품질, 제품 규모, 도메인 복잡도, 품질 특성 요구(보안·신뢰성 등) |
| 개발 프로세스 | 프로세스 안정성·성숙도, 개발 모델, 자동화·도구·환경 가용성 |
| 인력 | 기술·경험(특히 유사 프로젝트·도메인 경험), 휴가 등 가용성 |
| 테스트 결과 | 발견되는 결함 수·심각도, 재작업 양 (과거 통계가 있으면 정확도 상승) |
| 정황 | 분산된 팀·거점, 하위 시스템 수, 원격/현장 여부 |

### 5.3 추정 기법 선택
- 크게 메트릭 기반(과거 데이터·비율·외삽) vs 전문가 기반(델파이, 플래닝 포커 등)
- 선택 기준:
  - 추정 오차 표현 필요 여부: 3점 추정(낙관·비관·최빈값)은 기대값과 표준편차 제공
  - 과거 데이터 유무: 비율·외삽법은 신뢰할 과거 데이터 필수
  - 전문가 가용성: 델파이·플래닝 포커는 경험자 참여 필수
  - 모델링 지식: 외삽·3점 추정은 공식 사용 능력 필요
  - 시간 제약: 플래닝 포커는 빠르고 외삽법은 오래 걸림
- 실무 가이드: 복잡도 낮으면 메트릭 기반, 높으면 전문가 기반. 순차적 모델이면 와이드밴드 델파이, 애자일이면 플래닝 포커가 무난
- 애자일에서는 테스트를 따로 추정하지 않고 개발 작업에 포함해 추정하는 경우가 많음

## 6. 테스트 메트릭과 모니터링·보고

### 6.1 메트릭이 필요한 이유
- 측정하지 않는 활동은 방치되기 쉬움. 테스트 목표 달성 여부를 확인하려면 측정 방법이 정의되어야 함
- 메트릭 분류:
  - 프로젝트 메트릭: 완료 조건 대비 진행 상황 (예: 실행/합격/불합격 테스트 비율)
  - 제품 메트릭: 제품 속성 (예: 사용자 품질 기대 충족도)
  - 프로세스 메트릭: 테스트 프로세스의 성숙도·효과성
- 메트릭은 테스트 계획 단계에서 정의하고, 이후 측정·모니터링·보고

### 6.2 활동별 메트릭 매핑
| 메트릭 | 모니터링·제어용 | 완료 판정용 |
|---|---|---|
| 요구사항 커버리지 | O | O |
| 제품 리스크 커버리지 | O | O |
| 코드 커버리지 | O | |
| 실제 소요 시간 vs 계획 추정치 | O | |
| 계획 대비 상태별 테스트 케이스 실행 비율 | O | O |
| 누적 결함 수 vs 해결된 누적 결함 수 | O | |
| 자동화 계획 대비 실제 자동화 케이스 | | O |
| 실제 vs 계획 테스트 비용 | O | |

### 6.3 목적별 메트릭 예시
- 제품 리스크 관련: 연관 테스트가 전부 합격한 리스크 비율 / 일부라도 불합격한 리스크 비율 / 아직 테스트 안 끝난 리스크 비율
- 결함 관련: 누적 결함 vs 해결 결함, 그리고 다음 기준별 분류 — 컴포넌트별, 결함 출처별(요구사항/신규 기능/리그레션), 릴리스별, 유입·발견된 단계별, 우선순위/심각도별, 근본 원인별, 상태별(거부/중복/미해결/종료)
- 진행 관련: 테스트 실행 상태(계획/구현/실행/합격/불합격/미실행/건너뜀 수), 계획 공수 대비 실제 투입 공수
- 커버리지 관련: 요구사항 커버리지(케이스로 커버된 요구사항 비율), 리스크 커버리지(완화된 리스크 비율), 코드 커버리지
- 비용 관련: 미테스트 컴포넌트의 잔여 리스크, 계획 대비 실제 비용

### 6.4 보고 원칙
- 목적: 관리 의사결정에 필요한 정보를 즉시 이해 가능한 형태로 제공
- 스냅샷(특정 시점) 또는 추이(시간에 따른 변화)로 보고. 추세가 판단에 더 유용한 경우가 많음
- 상위 레벨 테스트(시스템·인수)는 요구사항·유스케이스·리스크 기준 커버리지로, 하위 레벨(단위·통합)은 구조적 커버리지(구문·인터페이스)로 보고하는 것이 적합
- 주의: 단위 테스트 커버리지 100%여도 상위 레벨에서 잡아야 할 결함·리스크는 남아 있음
- 서로 다른 메트릭 조합이 유용 (예: 미해결 결함 추이와 테스트 실행 추이의 상관, 요구사항 결함 수로 본 요구사항 문서 품질)
- 테스트 종료 결정: 신규 발견 결함이 계속 줄어드는 추세 + 합의된 완료 조건 충족을 근거로 판단

### 6.5 테스트 제어 조치의 예
- 식별된 리스크가 실제 이슈가 되면 테스트 우선순위 재조정
- 재작업 후 테스트 항목의 시작/완료 조건 재평가
- 테스트 환경 지연을 반영한 일정 조정
- 필요한 시점·위치에 자원 추가 투입

## 7. 결함 관리

### 7.1 기본 개념
- 이상 현상(anomaly): 기대 결과와 실제 결과의 불일치. 처음 관찰 시점엔 제품 결함인지, 다른 원인(자동화 실패, 테스터의 요구사항 오해)인지 알 수 없음
- 결함은 빨리 발견·제거할수록 품질 비용이 낮아짐. 유입된 단계 안에서 제거되는 것이 이상적
- 테스트 불합격이 항상 결함 보고로 이어지는 것은 아님 (예: TDD에서 개발 완료 전 실패하는 테스트는 정상)
- 결함 데이터를 시간에 걸쳐 수집·분석하면 테스트 외 개발 프로세스의 개선점도 찾을 수 있음

### 7.2 결함 워크플로우 (결함 생명주기)
기본 상태 흐름:
- 신규/미해결(OPEN/NEW): 보고서 생성 직후
- 진행 중(IN PROGRESS): 분석·수정 작업 중
- 거부(REJECTED): 처리자(주로 개발자)가 거부. 사유(잘못된 정보, 잘못된 테스트, 중복 등)를 반드시 기록
- 수정 완료(RESOLVED/FIXED): 테스터가 재현 절차에 따라 확인 테스트 수행
- 완료(CLOSED): 최종 상태. 확인 테스트 성공 또는 거부 인정

필요 시 추가하는 상태: 재오픈(RE-OPENED), 수용(ACCEPTED), 설명 필요(CLARIFICATION), 연기(DEFERRED)

워크플로우 설계 베스트 프랙티스:
- 가능하면 조직 차원에서 통일된 워크플로우 정의 (프로젝트 간 데이터 비교 가능)
- 중복·거짓양성 결함은 별도 상태를 두거나 거부 사유를 기록하며 거부 처리 (이후 프로세스 분석에 활용)
- 최종 상태는 하나만 두고, 전환 시 완료 사유를 선택하게 함
- 상태 이름은 다른 개체(스토리, 과업)의 상태 명칭과 통일 (협업 편의)
- 연속된 두 상태의 책임자는 서로 달라야 함 (같다면 명확한 근거 필요)
- 최종 상태를 제외한 모든 상태는 다음 갈래가 둘 이상이어야 함
- 상태 전환 시 입력 항목은 실제 가치를 주는 것만으로 최소화
- 애자일은 순차적 모델보다 워크플로우가 단순한 것이 일반적. 정황에 맞게 조정할 것

### 7.3 심각도와 우선순위
- 심각도(severity): 시스템·이해관계자에게 미치는 영향의 크기 (기술·비즈니스 영향 관점)
- 우선순위(priority): 수정을 얼마나 빨리 해야 하는가 (일정·비즈니스 판단 관점)
- 둘은 별개 속성. 예: 오타는 심각도 낮아도 메인 화면 브랜드명 오타면 우선순위 높음

### 7.4 결함 트리아지 (다기능 결함 관리)
- 결함 관리 프로세스·도구의 책임은 보통 테스트 조직에 있으나, 개별 결함의 처리 결정은 다기능 팀(결함 관리 위원회)이 담당
- 구성: 테스트 관리자, 개발팀, 공급업체, PM, 제품 관리/PO, 기타 이해관계자
- 트리아지 회의에서 결정: 수정할지(여러 팀이면 어느 팀이), 거부할지, 연기할지 — 수정의 이점·리스크·비용을 근거로
- 수정 결정 시 다른 작업과 비교해 우선순위 부여. 테스트팀은 객관적 정보 제공
- 대형 프로젝트는 전담 결함 관리자 임명 고려
- 원칙: 도구가 소통을 대체할 수 없고, 위원회가 좋은 도구를 대체할 수도 없음. 소통 + 도구 + 잘 정의된 워크플로우 + 적극적인 팀이 모두 필요

### 7.5 애자일 팀의 결함 관리
- 같은 공간·좋은 소통 채널이 있으면 공식 결함 보고서 없이 처리되는 경우가 많음
- 그래도 결함 보고서를 작성해야 하는 경우:
  - 스프린트 활동을 막는데 팀 내에서 즉시 수정 불가한 결함
  - 같은 스프린트 내 해결 불가한 결함 (당일 해결 불가 시 보고서 작성 규칙을 둔 팀도 있음)
  - 다른 팀이 해결해야 하거나 협력이 필요한 결함
  - 공급업체가 해결해야 하는 결함
  - 보고서가 명시적으로 요청된 경우
- 같은 스프린트에 해결 못하는 결함은 제품 백로그에 추가해 스토리와 함께 우선순위화
- 형식 수준의 결정 요인: 팀의 물리적 위치·시간대, 팀 수, 성숙도, 규모, 제품 리스크, 규제·계약 요구
- 팀이 정한 결함 관리 방식은 반드시 문서화 (위키 등 지식 관리 도구에)

### 7.6 하이브리드 환경의 결함 관리 난점
- 도구·속성 정렬: 팀마다 다른 결함 도구를 쓰는 것이 현실. 가능하면 도구 간 자동 동기화 구현
- 우선순위화: PO가 결함 회의에 참여해 영향·리스크 정보를 적극 수집. 애자일 팀 속도에 맞춰 회의는 더 자주·짧게. 최종 결정은 소수 핵심 그룹에 위임하는 것도 방법
- 계획 정렬·투명성: 개발 방식과 무관하게 모든 팀이 같은 프로젝트 계획을 따라야 함. 대시보드·공유 백로그로 팀 간 정보 공유

### 7.7 결함 보고서 품질
결함 보고서 정보의 4대 목적: 생명주기 관리, 프로젝트 상태 평가, 제품 증분 품질 평가, 프로세스 역량 평가

필수 항목 (작성자가 입력):
- 결함 제목: 이상 현상의 간단한 요약
- 상세 설명: 장애 재현 절차 포함 권장
- 심각도: 시스템·이해관계자 영향
- 우선순위: 수정 시급성

도구가 자동 생성하는 항목: 고유 ID, 생성 일시, 보고자, 발견된 프로젝트·단계, 현재 상태, 현재 담당자, 변경 이력, 참조(테스트 케이스, 연관 결함)

목적별 추가 정보:
- 해결 지원: 결함 위치(하위 시스템·컴포넌트), 발견된 릴리스 번호, 테스트 환경
- 프로젝트 상태 평가: 수정/미수정 시의 리스크·비용·기회, 대안, 영향받는 요구사항
- 제품 품질 평가: 결함 유형, 유입된 작업 산출물, 영향받는 품질 특성
- 프로세스 평가: 결함이 유입·발견·제거된 SDLC 단계, 근본 원인

작성 원칙:
- 속성이 많을수록 보고 시간이 늘고 오입력 가능성이 커짐 → 실제로 쓸 데이터만 수집
- 핵심 항목은 프로젝트 전체(이상적으론 조직 전체)에서 일관되게 유지해야 데이터 비교가 가능

### 7.8 결함 데이터로 하는 프로세스 개선
- 유입·발견·제거 단계 정보 → 단계 내 결함 봉쇄 수준 분석, 단계별 발견 효과성 개선
- 유입 단계 정보 → 결함이 가장 많이 유입되는 단계에 예방 활동 집중
- 근본 원인 정보 → 원인 제거형 개선으로 전체 결함 수 감소
- 결함 위치 정보 → 결함 클러스터 분석으로 기술 리스크 파악, 문제 컴포넌트 리팩토링 근거 확보
- 재오픈 결함 정보 → 수정(디버깅) 품질 평가
- 중복·거부 결함 정보 → 결함 보고서 작성 품질 평가
- 주의: 오버헤드를 줄이려고 결함 추적을 생략하면 프로세스 가시성이 크게 떨어지고, 데이터 부족으로 위 개선이 모두 불가능해짐

## 8. 테스트 완료 기준과 완료 보고서

- 완료 조건은 테스트 목표에서 도출하며, 계획 단계에서 미리 합의 (예: 요구사항 커버리지 100%, 심각도 상 결함 0건, 리스크 커버리지 목표 달성)
- 완료 조건 충족 여부는 합의된 메트릭으로 판정. 신규 결함 발견 추이 감소도 종료 판단의 참고 신호
- 완료 보고서에 담을 것:
  - 테스트 목표 대비 달성 여부 (완료 조건별 결과)
  - 제품 리스크: 완화된 리스크와 잔존 리스크
  - 결함 요약: 발견·해결·미해결(유예/수용) 현황
  - 테스트 진행 실적: 실행 현황, 커버리지
  - 비용·노력: 계획 대비 실제
  - 교훈과 개선 제안
- 미해결 결함은 변경 요청 또는 백로그 항목으로 넘기고, 유예·수용 결함 정보는 후속 담당자에게 인계
- 보고서는 승인 절차를 거쳐 이해관계자에게 전달

## 9. 형상관리·산출물 관리 (요약)

- 테스트 결과·로그·보고서 등 테스트웨어는 형상관리 시스템에 보관해 추적 가능하게 유지
- 재사용 가치가 있는 테스트 케이스는 접근·이해가 쉬운 형태로 식별·정리해 다음 프로젝트/주기에 활용
- 결함 보고서의 참조(테스트 케이스, 요구사항, 연관 결함) 연결로 추적성 확보
- 테스트 환경은 사용 후 데이터·스크립트·스텁을 제거하고 정해진 상태로 복원해 다음 주기에 대비

## 10. 테스팅의 비즈니스 가치 (품질 비용)

경영진·고객에게 테스트 예산을 설득할 때 쓰는 논리.

### 10.1 품질 비용 4분류
| 범주 | 내용 | 예 |
|---|---|---|
| 결함 예방 비용 | 낮은 품질을 사전에 막는 활동 | 개발자 교육, 요구사항 조기 리뷰, 팀 소통 체계 구축 |
| 감정평가 비용 | 결함을 찾는 활동 | 정적·동적 테스트 수행, 산출물 리뷰 |
| 내부 실패 비용 | 출시 전 발견한 결함의 사후 처리 | 테스트 중 발견한 결함 수정, 우회 방안 제공 |
| 외부 실패 비용 | 출시 후 결함으로 인한 손실 | 매출·평판 손실, 긴급 패치·배포·지원, 법적 비용 |

- 일반적으로 감정평가 비용 + 내부 실패 비용 < 외부 실패 비용 → 테스팅의 가치 근거
- 결함당 평균 절감액 = 평균 외부 실패 비용 - (평균 감정평가 비용 + 평균 내부 실패 비용)
- 계산 예: 외부 실패 $4,000, 감정평가 $500, 내부 실패 $200이면 결함 1건을 출시 전에 잡을 때마다 $3,300 절감

### 10.2 테스트 양의 균형
- 과도한 테스트: 리스크 감소로 얻는 이득보다 비용·지연이 큼
- 부족한 테스트: 아낀 비용보다 놓친 결함의 비용이 큼
- 최적점은 프로젝트·제품의 리스크와 조직이 감수할 수 있는 리스크 수준으로 결정 → RBT가 이 균형을 잡는 도구
- 결함 수정 비용은 SDLC 후반으로 갈수록 증가(Boehm 곡선) → 테스트는 초기부터 시작할 것
- 알려진 결함과 함께 출시하는 것이 출시 지연보다 나을 수 있음. 판단 기준은 결함의 발생 확률과 심각도

## 11. 테스트 프로세스 개선과 회고 (요약)

- 개선 사이클(IDEAL): 시작(목표·범위 합의) → 진단(현재 프로세스 평가) → 수립(개선 계획, ROI·리스크 기준 우선순위화) → 실행(교육·시범 적용·전면 도입) → 학습(효과 확인 후 다음 주기)
- 모델 기반 개선: TMMi(5단계 성숙도), TPI NEXT(16개 영역 x 4레벨) 같은 외부 모범 사례 모델과 비교. 프로젝트 수준에서는 계획·설계 등 관련 영역만 선별 적용
- 분석 기반 개선: 자기 데이터로 문제 식별 — 근본 원인 분석(피쉬본 다이어그램으로 결함 클러스터의 원인 추적), 메트릭 분석(효과성·효율성·예측 가능성), GQM(목표→질문→메트릭 순으로 측정 체계 설계)
- 정성 평가만으로 개선안을 내면 근거 없는 권고가 되기 쉬움. 데이터로 뒷받침할 것

회고 진행 5단계:
1. 소개: 목표·안건 공유, 비난 없는 분위기 조성
2. 데이터 수집: 사건 타임라인(정성) + 메트릭(정량: 진척, 결함 탐지, 효과성 등)
3. 개선안 도출: 근본 원인 분석 + 브레인스토밍
4. 조치 결정: 우선순위·책임·측정 지표 정의. 한 번에 너무 많은 개선을 시도하지 말 것
5. 마감: 회고 자체를 리뷰
- 순차적 모델에서는 테스트 완료 활동의 일부(다음 프로젝트를 위한 교훈), 애자일에서는 매 스프린트 종료 시 수행
- 결과는 반드시 문서화하고, 다음 주기에 조치의 효과를 확인

## 12. 테스트 도구 (요약)

- 도구 선정 프랙티스: 프로세스 개선 기회에서 출발, 기존 기술과의 호환성 확인, 명확한 요구사항·객관적 기준으로 평가, 공급업체/오픈소스 지원 평가, 교육 필요성 파악, 라이선스 모델 비교, 최종적으로 개념 검증(PoC) 수행
- 도입 프랙티스: 파일럿 프로젝트 → 프로세스 조정 → 사용 지침 정의 → 교육 → 점진적 배포 → 사용 데이터 수집 → 도구 소유자 지정
- 비용 구조: 일회성(선정·구매·교육·통합) + 반복(라이선스·유지보수·지속 교육) + 기회비용(도구에 쓴 시간은 테스트에 못 씀)
- ROI 리스크: 조직 미성숙으로 인한 비효율, 공급업체 정책 변경, 비용 초과, 효과 미달
- 기대 효과: 반복 수동 작업(리그레션) 감소, 테스트 주기 단축, 커버리지 증가, 인적 오류 감소, 정보 접근성 향상
- 도구 수명주기: 획득(소유자 지정, 명명 규칙 등 초기 결정) → 지원·유지보수(백업·데이터 교환) → 진화(환경 변화 대응) → 폐기(기능 대체, 데이터 보존)
- 도구별 수집 가능한 메트릭: 테스트 관리 도구(실행 상태), 요구사항 도구(커버리지 추적성), 결함 도구(상태·심각도·밀도), 정적 분석(코드 복잡도), 성능 도구(응답 시간·실패율), 커버리지 도구(실행된 코드 비율)

## 13. 팀 관리 핵심 (요약)

- 역량 4영역: 전문성(테스트 기법·도메인 지식), 방법론(분석·판단력), 사회적(소통·갈등 해결), 개인(자기 관리·책임감·회복력)
- 필요 기술은 프로젝트 정황 분석으로 도출: 도메인이 요구하는 비즈니스 지식, 기술 스택이 요구하는 전문 지식, SDLC가 요구하는 역할·기법 지식
- 스킬 매트릭스: 필요 기술 vs 보유 기술을 비교해 격차를 문서화하고 개발 계획 수립
- 역량 개발 방법: 교육·훈련, 자기 학습, 동료 학습, 멘토링·코칭, 현장 교육 — 전문성·방법론 역량은 교육·자기학습이, 사회적·개인 역량은 코칭·훈련이 효과적
- 동기 부여 요인(있으면 성장·만족): 업무 인정, 책임·자율성 확대, 의미 있는 도전 과업, 전문성 개발 기회
- 위생 요인(없으면 사기 저하): 적절한 보상, 합리적 경영 스타일, 좋은 근무 조건(명확한 명세, 안정적 테스트 환경 포함), 고용 안정, 좋은 대인 관계
- 관리자의 일: 동기 저하 요인을 지속 제거하면서 동기 부여 요인을 창출·강화
