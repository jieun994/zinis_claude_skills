# qa-master — ISTQB 기반 테스트케이스 생성·QA 스킬

ISTQB CTFL(기초)·CTAL-TA(설계 기법)·CTAL-TM(테스트 관리)·CTEL(전략)을 증류한
`references/` 문서를 근거로 작동한다. 감이 아니라 기법으로 테스트를 설계한다.

## 공통 원칙 (모든 모드에 적용)

1. **근거 없는 케이스 금지**: 모든 테스트케이스·지적사항·계획 항목에는 근거가 있어야 한다.
   - 입력 문서에 있는 내용 → 출처에 문서명·위치(시트/페이지/섹션) 기재
   - 입력에 없지만 ISTQB 기법상 필요한 케이스 → 출처에 `AI보완(기법명)` 표기 (예: `AI보완(경계값분석)`)
   - 이 구분을 생략하거나 뭉뚱그리지 않는다.
2. **기법을 명시**: 케이스마다 어떤 설계 기법으로 도출했는지 기록한다. "그냥 생각나서"는 없다.
3. **출력 스타일**: 개조식(명사형 종결), 이모지·아이콘 금지, 표 적극 사용.
4. **모르면 묻는다**: 입력 문서가 모호하면 지어내지 말고 `[확인 필요]`로 표기하거나 사용자에게 질문한다.
5. references는 필요한 것만 읽는다 (모드별 지정 참조). 전부 읽지 않는다.

## 0단계. 모드 판별

| 사용자 요청 | 모드 | 주 참조 문서 |
|---|---|---|
| 테스트케이스/시나리오 만들어줘, TC 뽑아줘 | 모드 1: 테스트케이스 생성 | 02(3장 기법 절차·5장 매핑표 중심), 01(6·7장) |
| 이 기획서 QA해줘, 검토해줘, 빠진 거 찾아줘 | 모드 2: QA 검토 | 01(5장 리뷰), 02(6장 체크포인트) |
| 테스트 계획 세워줘, 테스트 전략/일정 짜줘 | 모드 3: 테스트 계획 | 03, 04 |

- 어느 모드인지 애매하면 위 표를 보여주고 사용자에게 고르게 한다.
- 입력 문서(기획서, 요구사항정의서, IA, 플로우차트, 화면설계서, 유저스토리 등)가 없으면 먼저 요청한다.
  문서 없이 말로만 설명받은 경우에도 진행 가능하나, 모든 출처를 `구두설명` 또는 `AI보완`으로 표기한다.

### 입력이 화면설계서(PDF/이미지)인 경우 — 모드 1·2 공통

피그마 등에서 추출한 화면설계서 PDF는 다음 절차로 읽는다.

1. **이미지로 읽는 것이 기본.** 피그마 추출 PDF는 텍스트 추출 시 폰트 인코딩 문제로
   글자가 깨지는 경우가 많다. Read 도구의 PDF 읽기가 되면 그대로 쓰고, 렌더링이 안 되는
   환경이면 파이썬(PyMuPDF)으로 각 페이지를 PNG(가로 1600~2000px)로 변환한 뒤 이미지로 읽는다.
   텍스트 추출 결과는 보조로만 사용한다.
2. 페이지 수를 먼저 확인하고, 25페이지를 넘으면 대분류(메뉴/기능군) 단위로 나눠
   어디까지 진행할지 사용자에게 확인한다.
3. **채널 중복 페이지는 빠르게 스킵한다.** 동일 화면의 채널 변형(PC/모바일)이 의심되면
   헤더(Channel·화면 ID)만 먼저 확인해 중복 여부를 판단하고, 중복이면 차이점 위주로만 훑는다.
   기능 TC를 채널별로 복제하지 않으며, 채널 차이는 환경/호환 TC로 통합한다.
4. 화면(페이지) 단위로 요소를 추출해 테스트 컨디션 표로 정리한다:
   화면 ID·화면명 / 입력 필드(형식·자릿수·필수 여부·기본값) / 버튼·액션 /
   상태·분기(활성/비활성, 표시 조건) / 화면 간 이동 경로(화면 ID 연결)
5. **보이는 것만 근거로 삼는다.** 화면에 정의가 없는 값(입력 제한값, 오류 문구,
   빈 결과 처리 등)은 지어내지 말고 `[확인 필요]`로 분리한다.
6. 출처 표기 형식: `화면설계서 p.N (화면ID)` (예: `화면설계서 p.3 (PC-RE-100LAY01)`)

## 모드 1. 테스트케이스/시나리오 생성 (엑셀)

### 1-1. 입력 분석
1. 입력 문서를 전부 읽고 테스트 대상 기능 목록을 추출한다.
2. 범위 확인 기준 (둘 중 하나만 적용): 화면설계서 입력이면 25페이지 초과 시,
   그 외 문서는 기능 10개 초과 시 대분류 단위로 나눠 "이번에 어디까지 만들지" 사용자에게 확인한다.
3. 문서에서 발견한 모호함·모순은 케이스로 지어내지 말고 `[확인 필요]` 목록으로 분리한다.

### 1-2. 테스트 컨디션 도출
- `references/02_ctal_ta_설계기법.md` 1장의 절차를 따른다: 상위 컨디션("화면 X의 기능") → 상세 컨디션("입력값 Y가 조건 Z를 벗어나면 거부")으로 분해.
- 컨디션마다 근거 문서 위치를 기록해 추적성을 확보한다.

### 1-3. 기법 선택과 케이스 설계
- `references/02_ctal_ta_설계기법.md` 5장의 "기획서 요소 → 기법" 매핑표를 기준으로 선택한다. 핵심 요약:

| 기획서에 이런 요소가 있으면 | 이 기법 적용 |
|---|---|
| 입력 필드(값 범위, 자릿수, 형식) | 동등분할 + 경계값분석(2-value 기본, 리스크 높으면 3-value) |
| 조건 조합에 따라 결과가 달라지는 규칙(할인, 권한, 요금) | 결정테이블 |
| 상태·단계가 변하는 흐름(주문상태, 회원등급, 화면 전환) | 상태전이(최소 0-스위치, 핵심 흐름은 1-스위치) |
| 여러 옵션·환경의 조합(브라우저×OS×회원유형) | 페어와이즈/분류 트리 |
| 사용자 흐름·유스케이스·플로우차트 | 시나리오 테스팅(기본 흐름+대안 흐름+예외 흐름) |
| 데이터 생성·조회·수정·삭제가 있는 기능 | CRUD 테스팅 |
| 명세가 빈약하거나 시간이 부족한 영역 | 탐색적 테스트 차터 + 체크리스트 기반 |

- 각 기법의 도출 절차·커버리지 기준은 매번 references에서 확인하고 적용한다.
- 정상 케이스만 만들지 않는다. 기법상 도출되는 부정 케이스(경계 밖, 예외 흐름, 빈 결과, 취소·중단)를 반드시 포함한다.
- 검증할 동작은 있는데 기대결과가 문서에 정의돼 있지 않으면(입력 제한값·오류 문구 미정 등)
  TC는 만들되 기대결과 칸에 `[확인 필요]`를 병기한다. 기대결과를 지어내지 않는다.

### 1-4. 우선순위 부여 (리스크 기반)
- `references/01_ctfl_기초.md` 7장 기준: 리스크 수준 = 발생 가능성 × 영향.
- 상(H): 결제·개인정보·법적 요건·핵심 업무 흐름 / 중(M): 주요 기능의 대안 흐름 / 하(L): 보조 기능, 표시 문구.
- 판단 근거가 문서에 없으면 기본 규칙으로 배정하되 개요 시트에 배정 기준을 명시한다.

### 1-5. 엑셀 생성
- 스킬에 동봉된 `scripts/make_tc_excel.py`의 `build_workbook()`으로 생성한다 (openpyxl 필요).
  스타일·드롭다운 코드를 새로 짜지 않는다 — 데이터 리스트만 만들어 넘긴다.
- 파일명: `테스트케이스_<프로젝트/기능명>_v0.1_<YYYYMMDD>.xlsx`
- 시트 3개 구성:

**시트 1 `개요`** — 세로 2열(항목/내용) 형식
| 항목 | 내용 |
|---|---|
| 프로젝트명 | |
| 대상 문서 (테스트 베이시스) | 입력 문서명·버전 나열 |
| 테스트 범위 | 포함 기능 / 제외 기능 |
| 적용 기법 요약 | 사용한 기법과 적용 영역 |
| 우선순위 기준 | 리스크 기반 배정 규칙 |
| 총 시나리오 / 총 TC 수 | 기능(대분류)별 TC 수 포함 |
| 작성일 / 작성 도구 | qa-master (ISTQB 기반) |
| 확인 필요 사항 | `[확인 필요]` 목록 |

**시트 2 `테스트 시나리오`** — 컬럼:
`시나리오 ID | 구분(대분류) | 시나리오명 | 설명(사용자 관점 흐름) | 연관 TC 수 | 근거(출처)`
- 시나리오 ID 채번: `TS-<대분류약어>-<3자리>` (예: TS-LGN-001)
- 시나리오는 사용자 여정 단위(예: "비회원이 주문 후 회원 전환"), 케이스는 그 안의 검증 단위

**시트 3 `테스트케이스`** — 컬럼 19개 (설계 12 + 실행 기록 7):
`TC ID | 대분류 | 중분류 | 소분류 | 시나리오 ID | 우선순위 | 적용 기법 | 사전조건 | 테스트 절차 | 입력 데이터 | 기대결과 | 출처 | 실행결과 | 실제 결과 | 재현 절차 | 발견 환경 | 상태 | 결함 ID | 비고`

설계 시 채우는 컬럼 (TC ID~출처):
- TC ID 채번: `TC-<대분류약어>-<3자리>` (예: TC-LGN-001). 대분류 약어는 영문 3~4자로 개요 시트에 대응표 기재.
  입력 문서에 화면 ID 체계가 있으면 그 코드를 따른다 (예: 화면 ID가 PC-RE-xxx면 약어도 RE)
- 테스트 절차: 번호 매긴 단계(1. 2. 3.), 셀 내 줄바꿈
- 기대결과: 절차 마지막 단계 수행 시 확인할 것을 구체적으로 (화면 표시·데이터 변화·이동 위치)
- 출처: 근거 문서 위치 또는 `AI보완(기법명)`

실행 시 채우는 컬럼 (실행결과~비고) — 생성 시에는 실행결과만 `미실행`으로 채우고 나머지는 비워둔다:
- 실행결과: 드롭다운 `Pass,Fail,Block,미실행` (Block = 실패가 아니라 선행 조건 문제로 시도 자체가 막힘)
- 실제 결과: Fail일 때 실제로 어떻게 동작했는지 상세 기록 (상세 오류 내용 쓰는 곳)
- 재현 절차: 테스트 절차 그대로 하면 재현되는 경우 비워두고, 다른 조작으로 발견했을 때만 기록
- 발견 환경: 브라우저·기기·계정 등
- 상태: 드롭다운 `신규,수정중,수정완료,재확인완료` (결함 처리 진행 상태)
- 결함 ID: 결함 관리 툴(지라 등)을 쓰는 프로젝트만 연결, 없으면 비워둠

**공통 스타일**: 헤더 진회색 배경(#4A4A4A)·흰 글씨·굵게, 전체 테두리 실선, 머리행 고정,
자동 필터, 열 너비는 내용에 맞게 조정, 우선순위 상=연한 빨강 배경(#FCE4E4)

### 1-6. 마무리 검증
1. 모든 TC의 출처 칸이 채워졌는지 확인 (빈 출처 = 오류)
2. 시나리오 ID ↔ TC 연결 정합성 확인
3. 기능별로 정상/부정 케이스가 둘 다 있는지 확인 (부정 케이스 0건인 기능은 사유 명시)
4. 생성 후 사용자에게 보고: 기능별 TC 수, 적용 기법, `[확인 필요]` 목록
5. `[확인 필요]` 중 입력 문서 자체의 결함(모순·불일치·정의 갭)이 5건 이상이면
   "문서 결함이 많으니 모드 2(QA 검토) 리포트를 따로 만들까요?"라고 제안한다

### 1-7. 제출용 정리 (요청 시)
- "고객 제출용으로 정리해줘" 요청을 받으면 원본은 그대로 두고 별도 파일(`_제출용` 접미사)을 만든다:
  1. 적용 기법 컬럼 삭제 — `build_workbook(submission=True)`가 자동 처리
  2. 출처의 `AI보완(기법명)` → `추가 도출`로 치환 — 위 옵션이 자동 처리
  3. 개요 시트에서 내부용 표현(작성 도구 등)을 정리하고, `[확인 필요]`는 "고객 확인 요청 사항"으로 표현 변경

## 모드 2. QA 검토 (기획서/화면설계 리뷰)

### 2-1. 준비
- `references/01_ctfl_기초.md` 5장(정적 테스팅·리뷰)과 `references/02_ctal_ta_설계기법.md` 6장(리뷰 체크포인트, 기획서 QA 질문 목록)을 읽는다.

### 2-2. 검토 수행 — 3중 렌즈
1. **완전성**: 기획서 요소별 QA 질문 목록(02의 6장)을 순서대로 적용 — 입력 필드 정의 누락, 예외·오류 처리 누락, 빈 결과 화면, 취소·이탈 경로, 권한별 분기, 상태 정의 누락
2. **일관성**: 문서 내 모순(용어 불일치, 화면 간 규칙 충돌, 플로우와 화면 정의 불일치)
3. **테스트 가능성**: 검증 기준이 판정 가능한 문장인지 ("빠르게 응답" 같은 측정 불가 표현 지적)

### 2-3. 결과 리포트 (MD 파일, 요청 시 워드 변환)
- 구성: 검토 대상·기준 요약 → 지적사항 표 → `[확인 필요]` 질문 목록 → 총평
- 지적사항 표 컬럼: `번호 | 심각도(상/중/하) | 위치(페이지/화면) | 유형(누락/모순/모호/테스트불가) | 내용 | 개선 제안 | 근거(ISTQB 관점)`
- 심각도 기준: 상=구현 시 장애·재작업 유발 / 중=해석 차이 발생 가능 / 하=표현·표기 개선
- 지적 0건인 영역도 "검토했으나 특이사항 없음"으로 기록해 검토 범위를 증명한다.
